- **클로드 코드**: 앤트로픽이 소프트웨어 개발에 특화해 만든 에이전틱 코딩 도구
- **클로드 코드 CLI**: 클로드 코드를 터미널에서 실행하는 인터페이스

### 앤트로픽과 에이전틱 AI의 부상
- **에이전틱 AI**: 사용자의 질문에 답변하는 수준을 벗어나 스스로 문제를 분석하는 도구
- **MCP**: AI 모델이 외부 데이터 소스 및 도구와 상호작용 할 수 있도록 하는 표준화된 프로토콜

### 에이전틱 코딩이란
- **에이전틱 코딩**: AI가 파일 시스템 탐색, 코드 수정, 테스트 실행까지 자율로 수행하는 접근 방식. 탐색-계획-구현-커밋 워크플로로 정의

### 에이전틱 루프: 클로드 코드의 동작원리
1. 컨텍스트 수립: 프로젝트의 파일 구조를 탐색하고 관련 코드를 검색하여 필요한 정보 수집
2. 작업 수행: 실행 계획을 세우고 도구 호출
3. 결과 확인: 자신이 수행한 작업을 스스로 검증. 문제가 발견되면 클로드 코드는 컨텍스트 수립 단계로 돌아가 수정을 반복
- 에이전틱 루프의 자율성은 서브에이전트를 통해 확장. 서브에이전트는 독립되 ㄴ컨텍스트 윈도우에서 탐색하고 수행하고 결과를 요약해 메인 에이전틱에 반환

### 터미널에서 시작하는 AI 코딩
- 터미널에서 직접 실행할 수 있는 방식은 **모듈성**이라는 유닉스 철학을 따름
  ```
  $ cat security_logs_*.log | claude -p "이 로그를 분석해서 해킹 시도 패턴을 찾아서 위협 수준과 함께 보고해 줘."
  ```
  - 한번 실생되어 입력을 처리하고 종료하는 방식으로 작동하는 클로드 코드는 |(파이프)를 통해 다른 유닉스 도구와 연결되어 하나의 워크플로 구성

### 컨텍스트 윈도우와 20만 토큰 컨텍스트의 의미
- 컨텍스트 윈도우: 클로드가 한 번의 추론에서 참조할 수 있는 토큰의 총량
- 클로드 코드가 지원하는 컨텍스트 윈도우는 20만 토큰
- 토큰 수가 증가하면 정보 회상의 정확도가 떨어지고 장거리 추론 능력이 양화

### 컨텍스트 윈도우 관리 전략
1. Compaction
   - 컨텍스트 사용량의 한계 98%에 도달하면 클로드 코드는 이전 대화를 자동 요약
2. /clear
   - 대화 기록을 완전히 초기화해 깨끗한 상태에서 새 작업을 시작
3. 점진적 공개
   - 모든 정보를 처음부터 로드하는 대신, 가벼운 식별자만 유지하다가 필요할 때 도구로 데이터를 동적 로드
4. 서브에이전트 활용
   - 여러 서브에이전트가 동시에 대규모 코드베이스를 탐색하고 메인 에이전트에는 요약만 반환하는 구조
5.  구조화된 메모 작성
   - 마크다운 파일로 체크리스트를 작성하고 필요할 때 다시 불러오는 방식

### 다섯 가지 접근 영역과 핵심 기능
- 클로드 코드의 핵심 설계 원칙은 클로드가 프로그래머가 매일 사용하는 것과 동일한 도구를 필요

- 파일 시스템
  - Read, Write, Edit, Glob, Grep 도구로 프로젝트 디렉터리 내의 모든 파일을 읽고, 쓰고, 편집 가능
  - 파일 시스템 접근은 클로드 코드가 코드베이스를 이해하고 수정하는 가장 기본 수단
- 터미널
  - 클로드 코드는 사용자의 Bash 환경을 상속하므로 사용자가 설치한 모든 도구와 유틸리티에 접근 가능
- 웹
  - WebSearch와 WebFecth를 통해 인터넷에서 정보를 검색하고 가져옴
- MCP 서버
  - 클로드 코드는 외부 도구와 데이터 소스를 연결
- 에디터 컨텍스트
  - 클로드 코드는 현재 열려 있는 파일, 커서 위치, 선택된 테스트, 진단 정보(린트 오류, 타입 에러) 등 에디터의 상태 정보를 추가 컨텍스트로 활용

#### 클로드 코드의 4대 핵심 기능
1. 자연어 기반의 기능 구축
   - 개발자가 지엽적인 코드 구현보다 상위 수준의 요구 사항과 설계에 집중 가능
2. 심층 디버깅 및 문제 해결
   - 버그를 설명하거나 오류 메시지를 입력하면 클로드 코드가 코드베이스 전체를 분석하여 원인을 식별하고 수정을 제안
3. 코드베이스 탐색과 분석
4. 반복 테스트 자동화
   - 로컬 환경에서 단일 명령을 실행하거나 CI/CD 파이프라인에 통합하여 자동화된 워크 플로를 구축 가능

### 엔터프라이즈 환경에서의 클로드 코드
- 설정 파일의 버전 관리가 핵심
- .claude/commands 디렉토리에 프롬프트 템플릿을 저장하면 팀원이 공통 워크플로를 슬래시 명령으로 호출 가능
- CLAUDE.md 파이렝 프로젝트별 코드 스타일 가이드라이나 테스트 절차를 기록하면 에이전트가 해당 규칙을 자동으로 참조

### 대화형 모드와 비대화형 모드
- 대화형 모드
  - 플래그 없이 claude 명령만으로 실행하는 REPL 스타일의 대화형 세션
  - 탐색 작업과 장기 세션에 적합
  ```
  - project-root/
    - .claude/
      - commands/      # 커스텀 슬래시 명령어
      - agents/        # 커스텀 에이전트 정의
      - skills/        # 에이전트 스킬 모듈
      - hooks/         # 이벤트 훅 스크립트
      - output-styles/ # 출력 스타일 템플릿
      - settings.json  # 프로젝트 클로드 코드 설정
    - CLAUDE.md        # 프로젝트 지침서(컨텍스트로 자동 로드)
  ```
  #### 필수 명령어
  - /help
  - /clear
    - 현재 대화 컨텍스트를 초기화
    - 세션 내 대화 기록만 지우며 프로젝트 설정과 체크포인트도 유지되므로 필요하면 /rewind 명령으로 이전 상태 복구 가능
  - /exit
    - 세션 종료
    - 세션을 종료하면 현재 상태가 자동으로 저장되며 `claude -C` 명령으로 세션 제개 가능

- 비대화형 모드
  - 클로드 코드를 실행하려면 `-p` 또는 `--print` 플래그 사용
  - 프롬프트 결과만 출력한 후 종료하는데, 스크립팅과 자동화에 사용에 유리
 
### 다양한 실행 플래그
- 플래그에 따라 도구의 성격이 완전히 달라짐
- 일회성 작업 실행
  - `cladue "질문"` 처럼 별도의 플래그 없이 작업 내용을 인자로 전달해 클로드 코드를 즉시 실행 
- 비대화형 쿼리 실행
  - `claude -p "질문"` 처럼 사용하며 프롬프트에 대한 결과만 출력하고 세션 종료
- 최근 대화 세션 재개(-c, --continue)
  - 이전 진행하던 작업의 문맥을 그대로 유지하며 대화를 이어감.
- 이전 대화 세션 중 선택 재개(-r, --resume)
  - 과거 수행했던 여러 대화 세션의 목록을 확인, 특정 시점의 세션을 선택하여 다시 불러옴.

### 시스템 상태 진단과 문제 해결
- `cluade docker` 명령을 사용하면 설치가 제대로 됐는지 확인하고 실행 중에 발생하는 환경 문제를 파악

### 체크포인트와 되감기 시스템
- 파일 편집을 자동으로 추적하고 사용자가 원치 않는 변경 사항을 이전 상태로 되돌려 빠르게 실행 취소할 수 있게 하는 세션 수준 안전망
- 사용자가 새로운 프롬프트를 입력할 때마다 새 체크포인트를 자동 생성
- 변경 사항 되감기로 체크 포인트 시점으로 이동 가능
  - `Esc` 키를 두번 누르거나 /rewind 명령을 사용

 #### 체크포인트의 복구 옵션
- 대화만
  - 코드 변경 사항을 유지하면서 사용자 메시지로 되감기.
  - 클로드의 응답이 마음에 들지 않아 다른 방향으로 다시 시도하고 싶지만, 이미 작성된 코드는 유지하고 싶을 때 적합
- 코드만
  - 대화를 유지하면서 파일 변경 사항 되돌리기
  - 클로드 코드가 잘못된 방향으로 코드를 수정했을 때, 대화의 맥락은 보존하면서 파일만 복원하는데 적합
- 코드와 대화 모두
  - 세션의 이전 지점으로 둘다 복구. 작업 전체를 특정 시점으로 되돌리고 처음부터 다시 시작하고 싶을 때 적합

 #### 체크포인트가 유용한 상황
 - 시작점은 잃지 않고 대체 구현을 탐색해야 할 때
 - 잘못된 변경으로 버그가 생겼거나 기능이 깨져서 빠르게 복구해야할 때
 - 작동하는 현 상태를 유지하면서 실험해야 할 때

#### 체크포인트 한계
1. Bash를 통한 파일 변경은 추적 못함
2. 외부 변경 사항은 기록하지 않음. 다른 클로드 코드 인스턴스가 동시에 같은 파일을 편집했을 때 변경 사항이 기록되지 않음
3. 버전 관리 시스템의 대체재가 아님
