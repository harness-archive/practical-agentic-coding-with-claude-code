# 3-1 클로드 스킬의 개념과 구조
- 스킬: 클로드 기능을 확장하는 지식 모듈
- 스킬별 SKILL.md 지침 파일과 선택적 지원 파일(스크립트, 템플릿)로 구성
- 프롬프트에 원하는 내용을 입력하면 클로드가 자동으로 해당 스킬을 찾아서 사용

### 컨텍스트 윈도우는 공용 자산이다
- 유용한 지식과 패턴을 공유 가능한 형태로 캡슐화하여 팀 전체 또는 커뮤니티가 함께 활용할 수 있어야한다.
- 스킬 저장소를 깃에 커밋하면 팀원이 풀을 받는 즉시 해당 지식에 접근할 수 있으며, 도메인 전문가가 팀을 떠나더라도 그의 지식은 스킬로 보존

### 스킬의 유형
- 개인 스킬
  - `~/.claude/skills/` 디렉터리에 위치
  - 해당 개발자의 모든 프로젝트에서 사용 가능
- 프로젝트 스킬
  - 프로젝트 루트의 `.claude/skills` 디렉터리에 저장
  - 저장소에 포함되어 팀과 공유되며 자동 배포 가능
- 플러그인 스킬
  - 클로드 코드 플러그인과 함께 번들되어 배포되는 형태  
  ```text
  my-skill/
  ├── SKILL.md
  ├── refernece.md
  ├── scripts/
  │   └── helper.py
  └── templates/
      └── template.txt
  ```
  - SKILL.md는 필수, 나머지는 선택사항

 ### SKILL.md 파일 구조
 - `.claude/skills/skill-name/`에 위치
 - YAML 프론트매터와 마크다운으로 구성
 - 프론트매터 필드
   | 필드 | 필수 여부 | 설명 |
   | -- | -- | -- |
   | name | 선택 | - 소문자•영문자•하이픈만 사용<br> - 생략하면 디렉터리 이름을 스킬 이름으로 사용<br> - 최대 65자 |
   | description | 필수 | - 기능과 사용 트리거 설명<br> - 최대 1,024자 |
   | allowed-tools | 선택 | - 스킬 활성화 시 사용 가능한 도구 목록<br>예: Read, Grep, Glob |
   | model | 선택 | - 스킬 실행 시 사용할 모델 지정 |
   | context | 선택 | - 추가 컨텍스트 설정 |
   | agent | 선택 | - 서브에이전트 관련 설정 |
 - 커밋 메시지 생성 스킬의 예
   ```markdown
   ---
   name: generating-commit-message
   description: git diff에서 명확한 커밋 메시지 생성. 커밋 메시지 작성 또는 스테이징된 변경 사항 검토 시 사용
   ---

   # 커밋 메시지 생성

   ## 지침
   1. `git diff -staged` 실행
   2. 50자 미만의 요약, 상셍 설명, 영향을 받는 구성 요소로 메시지 제안

   ## 유의 사항
   - 현재형 사용
   - 무엇을 왜 설명, 어떻게가 아님
   ```
   - SKILL.md 파일은 200줄 이내로 유지
   - 이를 초과시 더 작은 단위로 분할하거나 상세 내용을 별도의 reference.md 파일로 분리
   - 하나의 스킬이 단일 책이 원칙을 따르도록 유도
  
### 세 단계의 점진적 작동 방식
- 필요하지 않은 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지하기 위해 점진적 공개
| 단계 | 레벨 | 로딩 시점 | 토큰 비용 | 콘텐츠 |
| -- | -- | -- | -- | -- |
| 1단계 | 메타데이터 | 시작 시 자동<br>(시스템 프롬프트) | 스킬 당 약 100 토큰 | 프론트매터의 name과 description를 시스템 프롬프트에 로드 |
| 2단계 | 지침 | 스킬 트리거 시<br>(Bash로 읽기) | 2,000 토큰(200줄)<br>이하 권장 | 사용자 입력이 특정 스킬의 description과 의미적으로 매칭되면, 해당 SKILL.md 파일을 Read 도구로 파일 시스템에서 읽어 컨텍스트 윈도우에 추가 | 
| 3단계 | 리소스/코드 | 필요시 개별 접근 | 사실상 무제한 | SKILL.md 실행 중 필요한 시점에, 스크립트나 참조 문서 등 개별 파일을 Read/Bash 도구로 lazy loading |
💡 시스템 프롬프트란 매 요청마다 LLM의 컨텍스트 윈도우 최상단에 포함되기 때문에, Claude는 추론 시점에 항상 스킬 목록을 참조 가능한 상태
💡 1 단계는 항상 메모리에 상주, 2,3 단계는 필요 시점에 동적으로 컨텍스트에 추가되는 on-demand 로딩 구조

### 스킬 컨텍스트 비용
- 컨텍스트 윈도우가 20만 토큰인데, 이미 시스템 프롬프트·대화 히스토리 등이 차지하고 있어서 스킬에 쓸 수 있는 실질 예산은 3만~5만 정도
- 단계별 토큰 비용
  - 1단계
    - 스킬 하나당 약 100토큰
    - 스킬 20개를 써도 총 2,000토큰 = 전채의 1%
  - 2단계
    - SKILL.md를 읽는 순간에만 비용 발생
  - 3단계
    - 문서를 읽는 경우
      : `reference.md`에 검증 규칙을 2,000줄로 작성했다면, Claude가 그 파일을 읽는 순간 2,000줄 전체가 컨텍스트 위로 올라옴.
    - 스크립트를 실행하는 경우
      : 똑같은 검증 규칙을 `validate_schema.py`로 만들면, Claude가 그 파일을 실행했을 때 결과값만 컨텍스트에 들어옴
- 토큰 최적화 전략
  - description을 압축 작성 - 100토큰 내에서 최대한 발견 가능성 확보
  - Claude가 이미 아는 건 쓰지말기 - `useState` 사용법, 파이썬 문법 같은 일반 지식은 SKILL.md에 적을 필요 없음. 이것만 해도 50~70% 절감 가능
  - -지원 파일을 도메인별로 분리 - 파일 하나에 다 몰아넣지 말고, 각 파일이 다르 파일을 연쇄 참조하지 않도록 구조
  - 입출력 예시는 1~2개만 SKILL.md에, 나머지는 EXAMPLES.md로 - 예시 5개면 SKILL.md 예산의 20~50%를 잡아먹음
  - 문서 대신 스크립트 사용 - 검증 규칙을 `.md`에 2,000줄 쓰면 2,000토큰이지만, `.py` 스크립트로 구현하면 결과값만 100~200 토큰
- 전체 컨텍스트 윈도우의 15% 이내로 유지. 15%를 초과하면 프로젝트 파일 로딩과 대화 히스토리에 할당 가능한 토큰이 부족해져 작업 품질이 떨어짐
