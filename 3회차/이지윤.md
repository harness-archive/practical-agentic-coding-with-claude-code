# 3-1 클로드 스킬의 개념과 구조
- 스킬: 클로드 기능을 확장하는 지식 모듈
- 스킬별 SKILL.md 지침 파일과 선택적 지원 파일(스크립트, 템플릿)로 구성
- 프롬프트에 원하는 내용을 입력하면 클로드가 자동으로 해당 스킬을 찾아서 사용

### 컨텍스트 윈도우는 공용 자산이다
- 유용한 지식과 패턴을 공유 가능한 형태로 캡슐화하여 팀 전체 또는 커뮤니티가 함께 활용할 수 있어야한다.
- 스킬 저장소를 깃에 커밋하면 팀원이 풀을 받는 즉시 해당 지식에 접근할 수 있으며, 도메인 전문가가 팀을 떠나더라도 그의 지식은 스킬로 보존

### 스킬의 유형
- 개인 스킬
  - `~/.claude/skills/` 디렉터리에 위치
  - 해당 개발자의 모든 프로젝트에서 사용 가능
- 프로젝트 스킬
  - 프로젝트 루트의 `.claude/skills` 디렉터리에 저장
  - 저장소에 포함되어 팀과 공유되며 자동 배포 가능
- 플러그인 스킬
  - 클로드 코드 플러그인과 함께 번들되어 배포되는 형태  
  ```text
  my-skill/
  ├── SKILL.md
  ├── refernece.md
  ├── scripts/
  │   └── helper.py
  └── templates/
      └── template.txt
  ```
  - SKILL.md는 필수, 나머지는 선택사항

 ### SKILL.md 파일 구조
 - `.claude/skills/skill-name/`에 위치
 - YAML 프론트매터와 마크다운으로 구성
 - 프론트매터 필드
   | 필드 | 필수 여부 | 설명 |
   | -- | -- | -- |
   | name | 선택 | - 소문자•영문자•하이픈만 사용<br> - 생략하면 디렉터리 이름을 스킬 이름으로 사용<br> - 최대 65자 |
   | description | 필수 | - 기능과 사용 트리거 설명<br> - 최대 1,024자 |
   | allowed-tools | 선택 | - 스킬 활성화 시 사용 가능한 도구 목록<br>예: Read, Grep, Glob |
   | model | 선택 | - 스킬 실행 시 사용할 모델 지정 |
   | context | 선택 | - 추가 컨텍스트 설정 |
   | agent | 선택 | - 서브에이전트 관련 설정 |
 - 커밋 메시지 생성 스킬의 예
   ```markdown
   ---
   name: generating-commit-message
   description: git diff에서 명확한 커밋 메시지 생성. 커밋 메시지 작성 또는 스테이징된 변경 사항 검토 시 사용
   ---

   # 커밋 메시지 생성

   ## 지침
   1. `git diff -staged` 실행
   2. 50자 미만의 요약, 상셍 설명, 영향을 받는 구성 요소로 메시지 제안

   ## 유의 사항
   - 현재형 사용
   - 무엇을 왜 설명, 어떻게가 아님
   ```
   - SKILL.md 파일은 200줄 이내로 유지
   - 이를 초과시 더 작은 단위로 분할하거나 상세 내용을 별도의 reference.md 파일로 분리
   - 하나의 스킬이 단일 책이 원칙을 따르도록 유도
  
### 세 단계의 점진적 작동 방식
- 필요하지 않은 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지하기 위해 점진적 공개  
  | 단계 | 레벨 | 로딩 시점 | 토큰 비용 | 콘텐츠 |
  | -- | -- | -- | -- | -- |
  | 1단계 | 메타데이터 | 시작 시 자동<br>(시스템 프롬프트) | 스킬 당 약 100 토큰 | 프론트매터의 name과 description를 시스템 프롬프트에 로드 |
  | 2단계 | 지침 | 스킬 트리거 시<br>(Bash로 읽기) | 2,000 토큰(200줄)<br>이하 권장 | 사용자 입력이 특정 스킬의 description과 의미적으로 매칭되면, 해당 SKILL.md 파일을 Read 도구로 파일 시스템에서 읽어 컨텍스트 윈도우에 추가 | 
  | 3단계 | 리소스/코드 | 필요시 개별 접근 | 사실상 무제한 | SKILL.md 실행 중 필요한 시점에, 스크립트나 참조 문서 등 개별 파일을 Read/Bash 도구로 lazy loading |

💡 시스템 프롬프트란 매 요청마다 LLM의 컨텍스트 윈도우 최상단에 포함되기 때문에, Claude는 추론 시점에 항상 스킬 목록을 참조 가능한 상태
💡 1 단계는 항상 메모리에 상주, 2,3 단계는 필요 시점에 동적으로 컨텍스트에 추가되는 on-demand 로딩 구조

### 스킬 컨텍스트 비용
- 컨텍스트 윈도우가 20만 토큰인데, 이미 시스템 프롬프트·대화 히스토리 등이 차지하고 있어서 스킬에 쓸 수 있는 실질 예산은 3만~5만 정도
- 단계별 토큰 비용
  - 1단계
    - 스킬 하나당 약 100토큰
    - 스킬 20개를 써도 총 2,000토큰 = 전채의 1%
  - 2단계
    - SKILL.md를 읽는 순간에만 비용 발생
  - 3단계
    - 문서를 읽는 경우
      : `reference.md`에 검증 규칙을 2,000줄로 작성했다면, Claude가 그 파일을 읽는 순간 2,000줄 전체가 컨텍스트 위로 올라옴.
    - 스크립트를 실행하는 경우
      : 똑같은 검증 규칙을 `validate_schema.py`로 만들면, Claude가 그 파일을 실행했을 때 결과값만 컨텍스트에 들어옴
- 토큰 최적화 전략
  - description을 압축 작성 - 100토큰 내에서 최대한 발견 가능성 확보
  - Claude가 이미 아는 건 쓰지말기 - `useState` 사용법, 파이썬 문법 같은 일반 지식은 SKILL.md에 적을 필요 없음. 이것만 해도 50~70% 절감 가능
  - -지원 파일을 도메인별로 분리 - 파일 하나에 다 몰아넣지 말고, 각 파일이 다르 파일을 연쇄 참조하지 않도록 구조
  - 입출력 예시는 1 ~ 2개만 SKILL.md에, 나머지는 EXAMPLES.md로 - 예시 5개면 SKILL.md 예산의 20 ~ 50%를 잡아먹음
  - 문서 대신 스크립트 사용 - 검증 규칙을 `.md`에 2,000줄 쓰면 2,000토큰이지만, `.py` 스크립트로 구현하면 결과값만 100~200 토큰
- 전체 컨텍스트 윈도우의 15% 이내로 유지. 15%를 초과하면 프로젝트 파일 로딩과 대화 히스토리에 할당 가능한 토큰이 부족해져 작업 품질이 떨어짐

### 도구 접근 권한
- `allowed-tools`를 통해 스킬이 사용할 수 있는 도구 제한
```
name: safe-file-reader
allowed-tools: Read, Grep, Glob # 이 3개만 사용 가능
```
- 파일을 읽기만 하고 수정은 못하게 막음
- 보안이나 코드 리뷰용 스킬에 유용

# 3-2 스킬 사용과 개발
### 스킬 목록과 스킬 조합
- 스킬 확인 방법
  ```bash
  ls ~/.claude/skills/ # 스킬 목록 보기
  cat ~/.claude/skills/my-skill/SKILL.md # 스킬 내용 보기
  ```
- 스킬 조합
  - `analyzing-api-requirements`: 요구 사항 분석
  - `implementing-rest-endpoitns`: 코드 작성
  - `testing-api-endpoints`: 테스트
  - 각 스킬은 API 개발 과정의 특정 단계에 집중하며 필요한 시점에 순차적으로 또는 병렬로 활성화

### 커스텀 스킬 개발
개발 프로세스 3단계
> 1단계: 목적 정의 -> 2단계: SKILL.md 작성 -> 3단계: 지원 파일 추가
- 1단계: 목적 정의
  - 세 가지를 먼저 결정해야 함:
    - 작업 범위 - 어떤 작업을 지원할 것인지
    - 활성화트리거 - 언제 활성화되어야 하는지
    - 필요 도구 - 어떤 도구가 필요한지
- 2단계: SKILL.md 작성
  - 디렉터리 먼저 생성:
    ```bash
    mkdir ~p ~/.claude/skills/<스킬명>
    ```
    그 다음 SKILL.md 파일 작성. 구성은 두 부분:
    1. 프론트매터(YAML)
       ```yaml
       name: creating-react-component
       description: React 함수형 컴포넌트 생성, props 타입 정의, 훅 구현
       allowed-tools: Read, Write, Edit, Grep
       ```
    2. 본문 지침 - Claude가 따를 규칙
       예를 들어 React 스킬이라면:
       | 항목 | 규칙 |
       | -- | -- |
       | 컴포넌트 | 함수형 + TypeScript props 타입 명시 |
       | 상태관리 | `useState`, `useReducer` |
       | 부수효과 | `useEffect` |
       | 파일 구조 | import -> 타입 -> 컴포넌트 -> 스타일 -> export |
       | 네이밍 | 컴포넌트 `PascalCase`, 훅`useXxx`, 이벤트`handleXxx` |
       | 주의 | 컴포넌트당 하나의 책임, props 구조분해, early return |
       - 본문에 입출력 예시를 2개 이상 넣어야 함.
       - Claude가 패턴을 정확히 학습하려면 단순한 케이스 + 복잡한 케이스를 모두 포함, 출력은 항상 같은 형식을 유지
         예시: API 문서화 스킬
         > 입력: GET /users/{id} 코드 -> 출력: 조회 API 문서
         > 입력: POST /users 코드 -> 출력: 생성 API 문서 (요청 본문 + 응답코드)
         > 입력: Delete /users/{id} -> 출력: 삭제 API 문서 (인증 필요 + 권한 명시)
- 3단계: 지원 파일 추가(선택)
  | 파일/폴더 | 용도 |
  | -- | -- |
  | reference.md | 상세 참고 문서 |
  | scripts/ | 헬퍼 스크립트 |
  | templates/ | 코드 템플릿 |
  SKILL.md에서 파일들을 링크로 참조하면 Claude가 필요할 때 점진적으로 로드
- 예시
  ```markdown
  ---
  name: creating-react-components
  description: React 함수형 컴포넌트 생성, props 타입 정의, 훅 구현. React 컴포넌트 작성 또는 리팩토링 시 사용
  allowed-tools: Read, Write, Edit, Grep
  ---

  # React 컴포넌트 생성

  ## 핵심 원칙

  컴포넌트는 함수형으로 작성하며, 타입스크립트를 사용하여 props 타입을 명시적으로 정의한다.
  상태 관리에는 useState와 useReducer를, 부수 효과에는 useEffect를 활용한다.

  ## 파일 구조

  컴포넌트 파일은 다음 구조를 따른다.
  1. import 문
  2. 타입 정의(Props, State)
  3. 컴포넌트 함수
  4. 스타일(CSS-in-JS 또는 외부 파일)
  5. export 문

  ## 네이밍 규칙

  - 컴포넌트: PascalCase(예: UseProfile)
  - 파일: 컴포넌트와 동일(예: UseProfile.txt)
  - 훅: camelCase, use 접두사(예: useUserData)
  - 이벤트 핸들러: handle 접두사(예: handleSubmit)

  ## 유의 사항

  - 컴포넌트당 하나의 책임
  - props느 구조 분해 할당 사용
  - 조건부 렌더링은 early return 패턴 활용
  - 메모이제이션은 실제 성능 문제 발생 시에만 적용

  상세 패턴은 [reference.md](./refrence.md) 참조
  템플릿은 [templates/component.tsx](./templates/components.tsx) 참조
  ```

### 스킬에서 환각 방지 지침 작성
스킬을 쓸 때 Claude가 없는 API를 호출하거나 잘못된 정보를 확실한 척 말하는 걸 막으려면 SKILL.md 본문에 명시:
> "모르면 모른다고 말할 것. 불확실한 정보는 추측하지 말것.
이 한줄이 스킬의 신뢰도를 크게 높여줌.

1. 불확실성 표현 허용 지침: 클로드가 불확실성을 인정하고 '모르겠다'라고 대답하도록 명시적으로 허용
   ```markdown
   ---
   name: api-documentation-generator
   description: API 엔드포인트 문서 자동 생성. OpenAPI 스펙 작성, 요청/응답 예시 생성시 사용
   ---

   # API 문서 생성

   ## 정확성 원칙

   확실하지 않은 정보에 대해서는 다음과 같이 표현한다.
   - "이 설정값은 공식 문서에서 확인이 필요합니다."
   - "버전별 차이가 있을 수 있으므로 릴리즈 노트를 참조하세요."
   - 이 동작은 환경에 따라 다를 수 있습니다."

   ## 금지 사항

   - 확인되지 않은 API 버전 정보를 단정적으로 서술하지 않는다.
   - 테스트하지 않은 설정값을 기본값으로 제시하지 않는다.
   - 추측에 기반한 에러 코드나 응답 형식을 작성하지 않는다.
   ```
2. 지식 범위 제한 지침: 클로드가 참조할 수 있는 정보의 범위를 명확히 제한.
   다음은 내 코드만을 기반으로 문서를 생성하도록 제한하는 예시
   ```markdown
   ---
   name: codebase-analyzer
   description: 프로젝트 코드베이스 분석 및 구조 문서화. 아키텍처 파악, 의존성 분석 시 사용
   ---

   ```
3. 출처 명시 지침: 정보의 출처를 명시하도록 지침을 작성하면 사용자가 정보의 신뢰성을 직접 검증할 수 있음.

   ```markdown
   ---
   name: library-usage-guide

   # 라이브러리 사용 가이드

   ## 출처 명시 규칙

   모든 기술 정보에는 출처를 명시한다.
   - API 사용법: 공식 문서 URL 또는 "프로젝트 내 사용 예시 참조"
   - 설정 옵션: 해당 파일 경로(예: src/config/database.ts:15)
   - 버전 정보: package.json 또는 공식 릴리스 페이지

   ## 응답 형식

   라이브러리 이름(버전: package.json 기준 또는 "확인 필요")

   설치 방법(출처: 공식 문서 또는 프로젝트 설정 파일)
   설정 예시(출처: 프로젝트 내 파일 경로)
   ```

### 코드 실행 통합
- 스킬의 `scripts/` 폴더에 실행 스크립트를 넣으면 Claude가 직접 실행할 수 있음.
- 핵심 원칙 2가지:
  - Claude가 토큰 소비 없이 빠르게 처리해야 할 결정론적 작업에 적합
  - 오류 상황은 스크립트 안에서 직접 처리(Claude에게 떠넘기지 말 것)
    ```text
    data-processing-skill/
      ├── SKILL.md
      ├── scripts/
      │   ├── format_data.py      ← 데이터 포맷 변환
      │   └── validate_schema.py  ← 스키마 검증
      ├── reference.md
      ├── requirements.txt
      └── templates/
          └── data_template.json
    ```
    - 스크립트 실행에 문제가 발생하면 다음 세가지 확인
      - 실행 권한
      - 의존성 설치 여부(claude.ai는 PyPI/npm 설치 가능, API환경은 런타임 설치 불가)
      - 경로가 유닉스 스타일 슬래시인지
    - SKILL.md에서 스크립트를 참조할 때 명확한 사용 지침 제공
      ```markdown
      ## 스크립트 사용

      ### format_data.py
      데이터 변환 스크립트입니다.
      사용법:
      python scripts/format_data.py --input data.json --output formatted.json

      입력: JSON 형식의 원본 데이터
      출력: 정규화된 JSON 데이터
      오류 처리: 잘못된 형식의 경우 종료 code 1과 함께 오류 메시지 출력

      ### validate_schema.py
      데이터 스키마 검증 스크립트입니다.
      사용법:
      python scripts/validate_schema.py --data data.json --schema schema.json
      
      입력: 검증할 데이터와 스키마 파일
      출력: 검증 결과 (PASS/FAIL)
      오류 처리: 스키마 불일치 시 상세 오류 메시지와 함께 종료 code 2 반환
      ```

### 테스트와 검증 방법
- 설명과 일치하는 질문을 해서 스킬이 활성화되는지 확인, 활성화되지 않는다면 다음을 확인
1. `descriptoin`이 구체적인지 확인
2. 스킬 디렉터리 경로를 확인(개인 스킬은 `~/.claude/skills/스킬명/SKILL.md`, 프로젝트 스킬은 `.claude/skills/스킬명/SKILL.md`)
3. YAML 유효성을 검사(시작과 종료`---`, 탭 사용금지, 올바른 들여쓰기, 유효한 구문)
4. 디버그 모드를 활용(`claude --debug`로 스킬 로딩 문제 확인)

# 3-3 스킬 고급 사용과 최적화
### 동적 스킬 로딩 패턴
- 동적 로딩 패턴: 스킬은 3단계 점진적 작동 방식을 따름
  ```text
  database-operations/
    ├── SKILL.md          # 1단계: 핵심 원칙 (100줄 이내)
    ├── reference/
    │   ├── queries.md    # 2단계: 쿼리 작성 상세 가이드
    │   ├── migrations.md # 2단계: 마이그레이션 패턴
    │   └── optimization.md # 2단계: 성능 최적화 기법
    ├── templates/
    │   ├── model.py.template     # 3단계: 모델 템플릿
    │   └── migration.py.template # 3단계: 마이그레이션 템플릿
    └── scripts/
        └── generate_model.py     # 3단계: 모델 생성 스크립트
  ```
  - SKILL.md에 각 레퍼런스 파일의 링크와 함께 언제 해당 파일을 참조해야하는지 명시
  ```markdown
  # 데이터베이스 작업

  ## 핵심 원칙(항상 적용)
  - ORM 우선 사용, 필요시 raw SQL
  - 인덱스 설계 먼저, 쿼리 작성 나중
  - 트랜잭션 범위 최소화
  
  ## 빠른 참조(자주 사용)
  - 기본 CRUD: Model.create(), Model.get(), Model.update(), Model.delete()
  - 필터링: Model.filter(field=value)
  - 정렬: Model.order_by('field')
  
  ## 상세 가이드(필요시 로드)
  - 복잡한 조인 쿼리: [queries.md](./reference/queries.md)
  - 마이그레이션 전략: [migrations.md](./reference/migrations.md)
  - 쿼리 최적화: [optimization.md](./reference/optimization.md)
  ```
  
### 스킬 성능 최적화
1. 정보 밀도 향상: 불필요한 설명 제거, 핵심 정보면 간결하게 작성
2. 조건부 로딩: 작업 유형에 따라 필요한 정보만 로드
3. 캐싱 고려: 자주 사용되는 정보는 SKILL.md에 포함, 드물게 사용되는 정보는 별드 파일로 분리

- SKILL.md 내에 Version History 섹션을 포함

  ```markdown
  ## Version History

  - 초기 버전 릴리스
  ```
- 버전을 관리할 때는 호환성을 고려

### 팀 스킬 개발
- 프로젝트 저장소의 ./claude/skills/ 디렉터리에 저장하고 깃으로 관리
- 팀 스킬을 개발할 때 다음 원칙을 따른다
  1. 표준화된 구조: 팀 내 모든 스킬이 동일한 디렉터리 구조와 파일 명명 규칙을 따름
  2. 리뷰 프로세스 도입: 코드 리뷰처럼 스킬 변경도 리뷰를 거침. description의 명확성, 예시의 품질, 도구 권한 설정의 적절성 등을 검토
  3. 피드백 수집/반영: 스킬이 예상대로 활성화되지 안헉나 지침이 불명확한 경우 등 실사용에서 발견되는 문제 수집 후 반영

### 디버깅 기법
- `claude --debug` 명령을 사용하면 스킬 로딩 과정에서 발생하는 문제 확인 가능
- 디버그 로그는 `~/.claude/debug/` 디렉터리에 세션별로 저장
  ```
  [DEBUG] Loading skills from directories:
    managed=/Library/Application Support/ClaudeCode/.claude/skills,
    user=/Users/username/.claude/skills,
    project=/Users/username/project/.claude/skills
  
  [DEBUG] Loaded 28 unique skills
  (managed: 0, user: 0, project: 28, duplicates removed: 0)
  
  [DEBUG] getSkills returning: 28 skill dir commands, 0 plugin skills
  ```
  - 확인 포인트:
    1. 스킬 검색 경로는 올바른가?
       - managed, user, project 세 가지 경로 모두 표시
    3. 동일한 이름의 스킬이 여러 위치에 있는가?
       - `dupolicates removed: 0`이 아니라면 중복이 있다는 의미
    5. 플러그인으로 제공되는 스킬이 정상적으로 로드되었는가?
       - plugin skills 개수로 확인
   - 일반적인 문제와 해결방법
     1. 스킬이 활성화되지 않는다: description이 모호하거나 트리거 용어가 없는 경우
     2. 여러 스킬이 충돌: 유사한 dexcription을 가진 스킬이 동시에 활성화
     3. 스크립트 실행 오류: 의존성 미설치, 실행 권한 부족, 경로 형식 오류 등이 원인
     4. YAML 파싱 오류: 프론트매터의 구문 오류는 스킬 전체를 로드하지 못하게 함.

### 재사용 가능한 스킬 패턴
- 재사용성이 높은 스킬의 세가지 특성
  1. 특정 프로젝트나 기술 스택에 종속되지 않는다
  2. 다양한 상황에 적용할 수 있도록 description에 충분한 트리거 조건 패턴을 담기
  3. 명확한 체크리스트와 단계별 지침을 가져 다른 스킬과 조합하기 쉬움
- 세가지 스킬 유형
  - 프로젝트 특화 패턴:
    - 특정 프로젝트의 규칙과 관행을 담기
    - 프로젝트 고유의 아키텍처, 명명 규칙, 코딩 스타일 등을 정의
  - 도메인 특화 패턴:
    - 특정 기술 스택이나 프레임워크에 특화된 형태
    - React 컴포넌트, 장고 모델, FastAPI 엔드포인트 등 특정 기술의 유의 사항을 담기
  - 범용 유틸리티 패턴:
    - 가장 재사용성이 높은 형태
    - 코드 리부, 문서 생성, 테스트 작성, 리팩터링 등 모든 프로젝트에서 필요한 작업
- 예시
  ```markdown
    ---
  name: reviewing-code-changes
  description: 코드 변경 사항 검토, 잠재적 버그 식별, 개선 제안. 풀 리퀘스트 또는 코드 리뷰 시 사용
  allowed-tools: Read, Grep, Glob
  ---
  
  # 코드 리뷰
  
  ## 검토 체크리스트
  
  ### 정확성
  [ ] 비즈니스 로직이 요구 사항을 충족하는가?
  [ ] 엣지 케이스가 처리되는가?
  [ ] 오류 처리가 적절한가?
  
  ### 유지보수성
  [ ] 코드가 읽기 쉬운가?
  [ ] 함수/클래스가 단일 책임을 가지는가?
  [ ] 중복이 최소화되었는가?
  
  ### 성능
  [ ] 불필요한 연산이 있는가?
  [ ] N+1 쿼리 문제가 있는가?
  [ ] 메모리 누수 가능성이 있는가?
  
  ### 보안
  [ ] 입력 검증이 적절한가?
  [ ] 민감 정보가 노출되지 않는가?
  [ ] SQL 인젝션, XSS 등 취약점이 없는가?
  
  ## 리뷰 결과 형식
  
  1. 요약: 전체적인 코드 품질 평가 (1-2문장)
  2. 필수 수정: 병합 전 반드시 수정해야 할 항목
  3. 권장 수정: 개선하면 좋지만 선택적인 항목
  4. 긍정적 피드백: 잘 작성된 부분에 대한 인정
  ```

### 스킬 문서화 유의 사항
- 스킬 자체도 문서지만 스킬에 대한 메타 문서화도 중요
- 팀 전체 스킬을 한눈에 볼 수 있는 인덱스 문서가 있으면 스킬 찾기/활용이 쉬워짐
- 스킬 문서화에 포함해야 할 핵심 요소:
  - 스킬의 목적과 적용 범위(언제 쓰는 스킬인지)
  - 실제 사용 예시(어떻게 활성화하는지)
  - 다른 스킬과의 관계(조합 패턴)
  - 알려진 제한 사항이나 주의 사항

### 스킬 격리 실행: context: fork 메커니즘
- 왜 필요한가?
  - 대규모 코드 분석 같은 작업은 탐색 결과(grep, 파일 내용 등)로 컨텍스트 창이 꽉참
  - 컨텍스트가 꽉 차면 이후 대화 품질이 급격히 떨어짐
  - 이를 해결하기 위해 부모 대화와 완전히 분리되 ㄴ서브에이전트에서 스킬을 실행
- 동작방식:
  - 스킬이 별도의 서브에이전트 컨텍스트에서 실행됨
  - 서브에이전트는 부모 대화 히스토리에 접근 불가
  - 작업 완료 후 결과요약만 부모 대화에 반환
- 설정 방법:
  ```markdown
  ---
  name: code-analysis
  description: 코드 품질을 분석하고 상세한 코드 리뷰 생성
  context: fork
  agent: Explore
  allowed-tools:
    - Read
    - Grep
    - Glob
  ---
  ```
- `agent`
  - `context: fork`가 설정된 경우에만 의미를 가짐
  - 서브 에이전트 동작 모드 결정
  - 세가지 옵션
    - `Explore` - 코드베이스 탐색 전용, 읽기 전용 모드
    - `Plan` - 컨텍스트 수집 후 실행 계획 수립에 특화
    - `general-purpose` - 기본값, 제약 없는 범용 모드
- 쓰면 좋은 상황:
  - 수백 개 파일을 탐색하는 대규모 코드베이스 분석
  - SPEC 문서 생성, TDD/DDD 구현, 테스트 전략 수립 등 구조화된 워크플로
  - `ast-grep` 같은 구조적 검색 도구를 활용한 코드 분석 스킬(구조적 패턴 매칭 결과가 수십 건에 달해 중간 데이터가 부모 대화를 오염시킬 수 있는 경우)
- 토큰 절감 효과:
  - fork 없이 실행 시 -> 탐색 과정에서 30,000 ~ 50,000 토큰 소비
  - fork 사용 시 -> 결과 요약 1,000 ~ 2,000 토큰만 부모에 전달
  - 약 95% 토큰 절감
- 쓰면 안 되는 상황:
  - 이전 대화 내용이나 사용자 선호도 등 부모 컨텍스트를 참조해야 하는 스킬
  - 짧은 코드 리뷰처럼 수백 토큰 수준의 소규모 작업(서브에이전트 생성 오버헤드가 더 비효율적)
