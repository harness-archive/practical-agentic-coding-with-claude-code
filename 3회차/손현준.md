## 1. 클로드 스킬의 개념과 3가지 유형
스킬은 "컨텍스트 윈도우는 공용 자산이다"라는 철학 아래, 개인의 유용한 지식과 패턴을 캡슐화하여 팀 전체가 공유할 수 있도록 돕는 기능입니다. 사용자가 프롬프트에 관련 내용을 입력하면 자동으로 해당 스킬을 찾아 활성화합니다.

적용 범위에 따라 다음 세 가지로 나뉩니다:
- **개인 스킬**: `~/.claude/skills/`에 저장되며, 개발자의 모든 로컬 프로젝트에서 동작합니다.
- **프로젝트 스킬**: 프로젝트 루트의 `.claude/skills/`에 저장되어 Git을 통해 팀원들과 표준 워크플로를 공유합니다.
- **플러그인 스킬**: 플러그인 마켓플레이스를 통해 설치하고 사용하는 커뮤니티 범용 스킬입니다.

## 2. 스킬 구조 및 3단계 점진적 작동 방식
스킬 디렉터리는 필수 지침 파일인 `SKILL.md`와 선택적 지원 파일(`scripts/`, `templates/` 등)로 구성됩니다. 불필요한 토큰 낭비를 막기 위해 세 단계로 점진적 공개(Progressive Disclosure)를 수행합니다.

1. **1단계 (메타데이터)**: 클로드 시작 시 프론트매터의 `name`과 `description` 필드만 로딩합니다 (약 100 토큰 소비).
2. **2단계 (지침)**: 사용자의 요청이 `description` 내용과 일치해 스킬이 트리거되면, Bash를 통해 `SKILL.md` 본문의 상세 지침을 읽어옵니다. 단일 책임 원칙을 위해 한글 기준 150줄 이내 작성을 권장합니다.
3. **3단계 (리소스/코드)**: 작업 수행 중 필요할 때만 외부 참조 문서나 파이썬 스크립트 등에 개별 접근하므로 사전에 컨텍스트를 낭비하지 않습니다.

## 3. 컨텍스트 토큰 최적화 및 권한 제어
전체 20만 토큰 중 스킬 시스템에 효율적으로 할당 가능한 예산은 3~5만 토큰 수준이므로, 스킬의 전체 토큰 비용을 컨텍스트 윈도우의 15% 이내로 유지하는 것이 좋습니다.

- **토큰 최적화 전략**: `description`의 정보 밀도를 높이고, 클로드가 이미 아는 범용 지식은 제거합니다. 대표적인 입출력 예시 1~2개만 `SKILL.md`에 남기고 나머지는 분리하며, 복잡한 검증 로직은 스크립트 파일로 작성해 실행 결과만 반환받도록 처리합니다.
- **보안/권한 제어**: 프론트매터의 `allowed-tools` 필드를 통해 스킬 활성화 시 사용할 수 있는 도구를 제한할 수 있습니다. 예컨대 코드 분석 스킬에 `Read, Grep, Glob`만 허용하여 코드 수정을 원천 차단합니다.

## 4. 커스텀 스킬 개발 및 디버깅 전략
자신만의 스킬을 만들 때는 `목적 정의` ➔ `SKILL.md 작성` ➔ `지원 파일 추가` 순으로 진행합니다.

- **환각(Hallucination) 방지**: 스킬 내에 "확인되지 않은 버전 정보를 단정적으로 서술하지 않는다" 같은 명확한 금지 사항을 작성하고, 참조할 지식의 범위를 로컬 코드로 엄격히 제한하는 지침을 포함해야 안전합니다.
- **동적 로딩 패턴 적용**: 모든 세부 가이드를 한 번에 로드하지 말고, `SKILL.md` 안에 "복잡한 쿼리 작성 시에는 queries.md 참조"와 같이 조건부로 로딩되게 연결해둡니다.
- **디버깅 (`claude --debug`)**: 스킬이 예상대로 작동하지 않는다면 디버그 모드를 실행합니다. 스킬 검색 경로, 중복된 스킬명 여부, 로드 실패 원인 등을 `~/.claude/debug/` 로그에서 확인할 수 있습니다.

## 5. 서브에이전트 격리 실행 (context: fork)
대규모 코드베이스 분석이나 구조화된 TDD 워크플로처럼 방대한 토큰(수만 토큰)을 소비하는 작업을 수행할 때, 메인 대화의 품질 저하를 막기 위해 **서브에이전트 컨텍스트에서 스킬을 격리 실행**하는 고급 기법입니다.

- **사용법**: `SKILL.md`의 프론트매터에 `context: fork`를 선언하고, `agent: Explore` 등 서브에이전트의 작동 모드를 지정합니다.
- **격리 효과**: 탐색 중 발생하는 중간 데이터를 부모 대화에서 분리합니다. 분석에 30,000~50,000 토큰이 쓰이더라도 부모 대화로는 1,000~2,000 토큰 분량의 핵심 요약 결과만 전달되므로, 약 95%의 토큰을 절감할 수 있습니다.
- **주의사항**: 이전 대화 히스토리나 문맥에 의존해야 하는 작업, 또는 수백 토큰 수준의 아주 단순한 수정 작업에는 오버헤드가 발생하므로 `context: fork`를 사용하지 않는 것이 좋습니다.
