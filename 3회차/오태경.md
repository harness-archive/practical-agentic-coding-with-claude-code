## 스킬
### 스킬이란? 
- Claude의 기능을 확장하는 재사용 가능한 지시사항
- CLAUDE.md가 "프로젝트 전체에 적용되는 규칙"이라면, 스킬은 "특정 작업을 위한 지침서"
- SKILL.md 파일로 작성하며, Claude가 관련 작업 시 자동으로 로드하거나 /skill-name으로 직접 호출 가능


### 스킬의 유형
Q. 개인 폴더, 프로젝트 폴더 등 여러 곳에 같은 이름의 스킬이 있다면 어느 쪽이 우선일까? [공식문서](https://code.claude.com/docs/ko/skills#where-skills-live)

- 아래 순서에 맞게 우선순위 적용
1. 엔터프라이즈
    - 조직의 모든 사용자
2. 개인
    - `~/.claude/skills/<skill-name>/SKILL.md`
    - 모든 프로젝트
3. 프로젝트
    - `.claude/skills/<skill-name>/SKILL.md`
    - 특정 프로젝트
4. 플러그인
    - `<plugin>/skills/<skill-name>/SKILL.md`
    - 플러그인이 활성화된 위치

## SKILL.md 파일 구조
- SKILL.md는 모든 내용을 포함할 순 있지만 본문 자체는 간결하게 유지하는것이 좋다.
  - [공식 문서](https://code.claude.com/docs/ko/skills#add-supporting-files)
  - **SKILL.md 파일은 200줄(영문, 한글은 150줄) 이하로 유지**
- SKILL.md는 간결하게 유지하기 위해 상세한 내용이나 참조 자료를 별도 파일들로 분리하여 필요할때만 로드하도록 하는것이 좋다.
```
my-skill/
├── SKILL.md           # 주요 지침 (필수)
├── reference.md       # SKILL.md에서 참조할 수 있는 지원 파일, 상세 설정 가이드, 예제 코드, 참조 자료 등 작성
├── template.md        # Claude가 출력을 생성할 때 참조하는 템플릿
├── examples/
│   └── sample.md      # 예상/기대 출력 예시
└── scripts/
    └── validate.sh    # Claude가 스킬 실행 중 실행할 수 있는 보조 스크립트
```
- 디렉토리에 묶어 배포하면, 스킬이 실행될 때마다 모두 컨텍스트에 올라오는 것이 아니라 Claude가 필요하다고 판단할 때만 로드
- scripts 디렉토리의 스크립트 작성시 오류 상황은 클로드에게 넘기지말고 스크립트 내부에서 처리해야한다.
-

```markdown
## 스크립트 사용
데이터 변환 스크립트의 사용법은 아래와 같다.
$ python3 scripts/transform.py

입력 : JSON 형식의 원본 데이터
출력 : 정제된 JSON 형식의 데이터
오류 처리 : 입력 데이터가 JSON 형식이 아닐 경우, code 1과 오류 메시지를 출력하고 프로그램을 종료한다.

<br>

## 참조 
- For complete API details, see [reference.md](reference.md)
- For usage examples, see [examples.md](examples.md)
```
- <u>지원 파일을 상대 참조시켜두어 클로드가 각 파일의 내용과 로드 시기를 알 수 있도록 하면 좋다.</u>

#### `프론트매터 옵션 (Frontmatter Options)`
- 상단의 YAML 프론트매터로 스킬의 이름, 호출 조건, 권한 등을 설정
- **모든 필드는 선택적**이며 **Claude가 skill을 언제 사용할지 알 수 있도록 description만 권장**함. 

  | 필드 | 필수 | 설명                                                                                         |
  |------|------|--------------------------------------------------------------------------------------------|
  | `name` | N | 스킬 목록에 표시되는 이름. <br> 디렉토리 이름으로 기본값이 설정. <br> 대소문자 하이픈만 사용, 최대 65자 <br> 행위를 드러내는 동명사 형태를 권장 |
  | `description` | `Y` | 스킬이 무엇을 하는지, 언제 사용할지 스킬의 용도와 사용 시점을 설명 <br> 이 필드를 통해 클로드가 스킬을 자동으로 로드할지 판단                 |
  | `allowed-tools` | N | 스킬이 활성화되었을 때 사용할 수 있는 도구 목록 (Read, Grep, Bash(npm test) 등).                                |
  | `disallowed-tools` | N | 스킬이 활성화되었을 때 Claude가 사용하지 못하도록 차단해야하는 도구 목록.                                               |
  | `model` | N | 스킬 실행시 사용할 모델.                                                                             |
  | `context` | N | 추가 컨텍스트 설정 (context: fork) <br> 스킬이 독립적인 subagent 환경에서 실행                                  |
  | `agent` | N | `context: fork`가 설정되었을 때 사용할 subagent 타입.  (Explore, Plan 등)                               |

Q. 스킬 프론트매터 옵션 중 allowed-tools에 도구를 명시하면 권한 체크는 어떻게 되는가?
- allowed-tools에 명시된 도구는 스킬이 활성화되었을 때 Claude가 권한을 요청하지 않고 사용할 수 있다.
<br>

#### `마크다운`
- 해당 스킬을 사용할 때 클로드가 따라야할 지침, 유의사항, 예시 등을 포함한다.

<br>

- 클로드는 스킬을 세 단계로 로드하여 필요하지 않는 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지한다.
1. 메타데이터 단계 : 프론트매터이 name, description을 자등으로 로드
2. 지침 단계 : 특정 스킬이 매칭되면 Bash를 통해 SKILL.md를 컨텍스트 윈도우에 추가
3. 리소스/코드 단계 : 그 외 추가적인 정보(reference.md, scripts 등)는 클로드가 필요할 때만 로드

## 스킬 컨텍스트 비용
사용 가능한 스킬을 파악하려면 스킬 설명이 컨텍스트에 로드되어야 한다. 

만약 스킬이 수십 개라면? 설명민으로도 컨텍스트를 꽤 잡아먹을 수도 있다.

활성화된 스킬이 공유하는 토큰은 유한하다. 그렇기에 정해진 예산 내에서 스킬의 가치를 극대화하는것이 중요하다!!
- 1단계 메타데이터는 스킬당 약 100 토큰이 고정 소비된다.
- 2단계 지침 단계는 활성시에만 발생하는 변동 비용
- 3단계 지원 단계는 예측하기 어렵다.

그렇기에 아래와 같은 최적화 전략을 수립하자.
1. description을 통해 최대한 발견 가능성을 높인다.
2. 클로드가 이미 알고 있는 지식은 SKILL.md 본문에서 제거한다.
   - **프로젝트의 컨벤션, 도메인, 팀 특화 패턴, 외부에 공개되지 않는 내부 API 명세 등의 정보들이 필요하다.**
3. 지원 파일의 구조를 도메인 별로 분리
   - 다만 SKILL.md로부터 1단계 깊이만 유지하도록 하고 그 이상 깊이는 추천 X
4. 스킬의 입출력 예시를 최적화한다.
   - 출력은 생각보다 토큰을 잡아먹는다. 대표적인 예시 1~2개만 SKILL.md에 명시하고 그 외 예시는 지원 파일인 EXAMPLES.md로 분리한다.
   - 입출력 예시를 2개 이상 작성해 클로드가 패턴을 학습하도록 하는것이 좋다. 다만 예시가 많아질수록 토큰 비용이 증가한다.
   - 좋은 입력 예시는 다양한 입력을 제공하여 간단한 입력, 복잡한 입력, 엣지 케이스를 포함하는것이 좋다.
   - 좋은 출력 예시는 일관성있게 동일한 형식과 품질 기준을 따르는것이 좋다. (예: JSON, Markdown 등)
5. 스크립트 우선 전략을 채택하는것도 좋다.
   - 실행 가능한 스크립트는 결과만 컨텍스트에 반영하기때문에 비교적 토큰 비용이 낮다.

Q. 위 단계마다 컨텍스트 비용 예산을 정해놓는 방법은 없을까?

A. 찾아보니 settings.json 옵션에 skillListingBudgetFraction 라는 필드로 예산을 설정할 수 있는 것 같지만 각 단계마다는 설정을 못하는 것 같음.

## 스킬 사용과 개발
- 클로드에게 물어보거나 / 명령어를 통해 스킬을 호출할 수 있다.
- 여러 스킬을 조합해서 사용할 수 있다.
  - 각 스킬들은 필요한 시점에 순차적으로 or 병렬로 활성화된다.
- 커스텀한 스킬을 개발하려면 아래 순서를 따른다.
1. 스킬 정의
   - 스킬의 목적과 범위를 정의하고 언제 활성화되어야 하는지, 어떤 도구가 필요한지 정의
2. 디렉토리 생성
3. SKILL.md 작성
   - 프론트매터에 name, description,을 정의하고 본문에 구체적인 내용 작성
4. 지원 파일 추가
   - 복잡한 스킬이라면 상세 레퍼런스를 작성하거나 스크립트를 작성한다.
   - SKILL.md에 지원 파일들을 상대 링크로 참조하도록 하면 필요시 로드한다.
- 스킬 사용이 잘 되지 않을 때는 claude --debug 를 사용하여 스킬 로딩 문제를 확인한다.

### 스킬에서 환각 방지 지침
- 존재하지 않는 정의나 API를 참조하지 않도록 SKILL.md에 환각 방지 지침을 포함시킨다.
1. 불확실성 표현 허용 지침
   - 클로드가 불확실성을 인정하고 모르겠다라고 대답하도록 명시적으로 허용한다.
   ```markdown
   ...
   
   ## 정확성 원칙
   확실하지 않은 정보에 대해서는 다음과 같이 표현한다.
   ~~~
   ## 금지 사항
   - 확인되지 않은 API 버전 정보를 단정하여 서술하지 않는다.
   - 추측에 기반한 에러 코드나 응답 형식을 작성하지 않는다.
   ~~~ 
   ```
2. 지식 범위 제한 지침
   - 클로드가 참조할 수 있는 정보의 범위를 명확히 제한한다.
3. 출처 명시 지침
    - 클로드가 참조한 정보의 출처를 명시하도록 지침을 포함한다.

## 스킬 고급 사용과 최적화
