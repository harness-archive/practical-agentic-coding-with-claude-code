## 1. 스킬
### 스킬이란? 
Claude의 기능을 확장하는 재사용 가능한 지시사항

CLAUDE.md가 "프로젝트 전체에 적용되는 규칙"이라면, 스킬은 "특정 작업을 위한 지침서"

SKILL.md 파일로 작성하며, Claude가 관련 작업 시 자동으로 로드하거나 /skill-name으로 직접 호출 가능


### 스킬의 유형
Q. 개인 폴더, 프로젝트 폴더 등 여러 곳에 같은 이름의 스킬이 있다면 어느 쪽이 우선일까? [공식문서](https://code.claude.com/docs/ko/skills#where-skills-live) <br>
A. 아래 순서에 맞게 우선순위 적용

1. 엔터프라이즈
    - 조직의 모든 사용자
2. 개인
    - `~/.claude/skills/<skill-name>/SKILL.md`
    - 모든 프로젝트
3. 프로젝트
    - `.claude/skills/<skill-name>/SKILL.md`
    - 특정 프로젝트
4. 플러그인
    - `<plugin>/skills/<skill-name>/SKILL.md`
    - 플러그인이 활성화된 위치

## 2. SKILL.md 파일 구조
- SKILL.md는 모든 내용을 포함할 순 있지만 본문 자체는 간결하게 유지하는것이 좋다.
  - [공식 문서](https://code.claude.com/docs/ko/skills#add-supporting-files)
  - **SKILL.md 파일은 200줄(영문, 한글은 150줄) 이하로 유지**
- SKILL.md는 간결하게 유지하기 위해 상세한 내용이나 참조 자료를 별도 파일들로 분리하여 필요할때만 로드하도록 하는것이 좋다.
```
my-skill/
├── SKILL.md           # 주요 지침 (필수)
├── reference.md       # SKILL.md에서 참조할 수 있는 지원 파일, 상세 설정 가이드, 예제 코드, 참조 자료 등 작성
│   └── queries.md     # 쿼리 작성 상세 가이드
├── template.md        # Claude가 출력을 생성할 때 참조하는 템플릿
├── examples/
│   └── sample.md      # 예상/기대 출력 예시
└── scripts/
    └── validate.sh    # Claude가 스킬 실행 중 실행할 수 있는 보조 스크립트
```
- 디렉토리에 묶어 배포하면, 스킬이 실행될 때마다 모두 컨텍스트에 올라오는 것이 아니라 Claude가 필요하다고 판단할 때만 로드
- scripts 디렉토리의 스크립트 작성시 오류 상황은 클로드에게 넘기지말고 스크립트 내부에서 처리해야한다.

<br>

- 스킬은 `YAML 프론트매터` 와 `마크다운 본문` 으로 구성되어있다.
- argument-hint는 스킬이 호출될 때 Claude가 사용자에게 어떤 입력을 요청할지 안내하는 메시지

- <u>지원 파일을 상대 참조시켜두어 클로드가 각 파일의 내용과 로드 시기를 알 수 있도록 하면 좋다.</u>

#### `프론트매터 옵션 (Frontmatter Options)`
- 상단의 YAML 프론트매터로 스킬의 이름, 호출 조건, 권한 등을 설정
- **모든 필드는 선택적**이며 **Claude가 skill을 언제 사용할지 알 수 있도록 description만 권장**함. 

  | 필드 | 필수 | 설명                                                                                         |
  |------|------|--------------------------------------------------------------------------------------------|
  | `name` | N | 스킬 목록에 표시되는 이름. <br> 디렉토리 이름으로 기본값이 설정. <br> 대소문자 하이픈만 사용, 최대 65자 <br> 행위를 드러내는 동명사 형태를 권장 |
  | `description` | `Y` | 스킬이 무엇을 하는지, 언제 사용할지 스킬의 용도와 사용 시점을 설명 <br> 이 필드를 통해 클로드가 스킬을 자동으로 로드할지 판단                 |
  | `allowed-tools` | N | 스킬이 활성화되었을 때 사용할 수 있는 도구 목록 (Read, Grep, Bash(npm test) 등).                                |
  | `disallowed-tools` | N | 스킬이 활성화되었을 때 Claude가 사용하지 못하도록 차단해야하는 도구 목록.                                               |
  | `model` | N | 스킬 실행시 사용할 모델.                                                                             |
  | `context` | N | 추가 컨텍스트 설정 (context: fork) <br> 스킬이 독립적인 subagent 환경에서 실행                                  |
  | `agent` | N | `context: fork`가 설정되었을 때 사용할 subagent 타입.  (Explore, Plan, general-purpose)                             |

Q. 스킬 프론트매터 옵션 중 allowed-tools에 도구를 명시하면 권한 체크는 어떻게 되는가? <br>
A. allowed-tools에 명시된 도구는 스킬이 활성화되었을 때 Claude가 권한을 요청하지 않고 사용할 수 있다.

<br>

#### `마크다운`
- 해당 스킬을 사용할 때 클로드가 따라야할 지침, 유의사항, 예시 등을 포함한다.

<br>

#### 클로드의 스킬 로딩 단계 (동적 스킬 로딩)
클로드는 스킬을 세 단계로 로드하여 필요하지 않는 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지한다.
1. 메타데이터 단계 : 프론트매터이 name, description을 자등으로 로드
2. 지침 단계 : 특정 스킬이 매칭되면 Bash를 통해 SKILL.md를 컨텍스트 윈도우에 추가
3. 리소스/코드 단계 : 그 외 추가적인 정보(reference.md, scripts 등)는 클로드가 필요할 때만 로드

## 3. 스킬 컨텍스트 비용
사용 가능한 스킬을 파악하려면 스킬 설명이 컨텍스트에 로드되어야 한다. 

만약 스킬이 수십 개라면? 설명민으로도 컨텍스트를 꽤 잡아먹을 수도 있다.

활성화된 스킬이 공유하는 토큰은 유한하다. 그렇기에 정해진 예산 내에서 스킬의 가치를 극대화하는것이 중요하다!!
- 1단계 메타데이터는 스킬당 약 100 토큰이 고정 소비된다.
- 2단계 지침 단계는 활성시에만 발생하는 변동 비용
- 3단계 지원 단계는 예측하기 어렵다.

그렇기에 아래와 같은 최적화 전략을 수립하자.
1. description을 통해 최대한 발견 가능성을 높인다.
2. 클로드가 이미 알고 있는 지식은 SKILL.md 본문에서 제거한다.
   - **프로젝트의 컨벤션, 도메인, 팀 특화 패턴, 외부에 공개되지 않는 내부 API 명세 등의 정보들이 필요하다.**
3. 지원 파일의 구조를 도메인 별로 분리
   - 다만 SKILL.md로부터 1단계 깊이만 유지하도록 하고 그 이상 깊이는 추천 X
4. 스킬의 입출력 예시를 최적화한다.
   - 출력은 생각보다 토큰을 잡아먹는다. 대표적인 예시 1~2개만 SKILL.md에 명시하고 그 외 예시는 지원 파일인 EXAMPLES.md로 분리한다.
   - 입출력 예시를 2개 이상 작성해 클로드가 패턴을 학습하도록 하는것이 좋다. 다만 예시가 많아질수록 토큰 비용이 증가한다.
   - 좋은 입력 예시는 다양한 입력을 제공하여 간단한 입력, 복잡한 입력, 엣지 케이스를 포함하는것이 좋다.
   - 좋은 출력 예시는 일관성있게 동일한 형식과 품질 기준을 따르는것이 좋다. (예: JSON, Markdown 등)
5. 스크립트 우선 전략을 채택하는것도 좋다.
   - 실행 가능한 스크립트는 결과만 컨텍스트에 반영하기때문에 비교적 토큰 비용이 낮다.

Q. 위 단계마다 컨텍스트 비용 예산을 정해놓는 방법은 없을까? <br>
A. 찾아보니 settings.json 옵션에 skillListingBudgetFraction 라는 필드로 예산을 설정할 수 있는 것 같지만 각 단계마다는 설정을 못하는 것 같음.

## 4. 스킬 사용과 개발
- 클로드에게 물어보거나 / 명령어를 통해 스킬을 호출할 수 있다.
- 여러 스킬을 조합해서 사용할 수 있다.
  - 각 스킬들은 필요한 시점에 순차적으로 or 병렬로 활성화된다.
- 커스텀한 스킬을 개발하려면 아래 순서를 따른다.
1. 스킬 정의
   - 스킬의 목적과 범위를 정의하고 언제 활성화되어야 하는지, 어떤 도구가 필요한지 정의
2. 디렉토리 생성
3. SKILL.md 작성
   - 프론트매터에 name, description,을 정의하고 본문에 구체적인 내용 작성
4. 지원 파일 추가
   - 복잡한 스킬이라면 상세 레퍼런스를 작성하거나 스크립트를 작성한다.
   - SKILL.md에 지원 파일들을 상대 링크로 참조하도록 하면 필요시 로드한다.
- 스킬 사용이 잘 되지 않을 때는 claude --debug 를 사용하여 스킬 로딩 문제를 확인한다.

### 스킬에서 환각 방지 지침
존재하지 않는 정의나 API를 참조하지 않도록 SKILL.md에 환각 방지 지침을 포함시킨다.
1. 불확실성 표현 허용 지침
   - 클로드가 불확실성을 인정하고 모르겠다라고 대답하도록 명시적으로 허용한다.
   ```markdown
   ...
   
   ## 정확성 원칙
   확실하지 않은 정보에 대해서는 다음과 같이 표현한다.
   ~~~
   ## 금지 사항
   - 확인되지 않은 API 버전 정보를 단정하여 서술하지 않는다.
   - 추측에 기반한 에러 코드나 응답 형식을 작성하지 않는다.
   ~~~ 
   ```
2. 지식 범위 제한 지침
   - 클로드가 참조할 수 있는 정보의 범위를 명확히 제한한다.
3. 출처 명시 지침
    - 클로드가 참조한 정보의 출처를 명시하도록 지침을 포함한다.

<br>

### 스킬에서 안전 방지 지침
배포처럼 위험한 작업은 Claude가 자동으로 실행하면 안된다. 프론트매터로 호출 권한을 제어할 수 있다.
```markdown
---
name: deploy
description: 테스트 확인 후 프로덕션에 배포합니다.
disable-model-invocation: true
---

배포 전 체크리스트:
1. `pytest`로 전체 테스트 실행
2. 실패하는 테스트가 있으면 중단하고 보고
3. 전부 통과하면 `git push origin main` 실행
4. 배포 결과 확인
```
- disable-model-invocation 
    - true면 사용자가 직접 /로 호출할 때만 실행됨. Claude가 스스로 판단하여 자동 로드하는 것을 차단.

## 5. 스킬 고급 사용과 최적화
복잡한 시스템 개발에서는 팀 단위로 협업하여 개발하는 고급 패턴이 필요하다.

불필요한 설명을 제거하고 핵심 정보만 간결하게

SKILL.md에 작업 유형에 따라 필요한 정보만 로드하도록 상세 파일 분리

드물게 사용되는 정보는 별도 파일로 분리

```markdown
# 디비 작업
## 핵심원칙 (우선 적용)
- ORM 우선 사용

## 빠른 참조 (자주 사용)
- 기본 CRUD 사용

## 상세 가이드 (필요시 로드)
- 복잡한 조인 쿼리 [quries.md](./reference/queries.md)
```

### 스킬 버전 관리
스킬이 계속 변경이 됨에 따라 변경 이력을 추적해야할 수도 있고 필요시 이전 버전으로 롤백할 수 있어서 버전 관리를 하면 좋다.

SKILL.md 내에 Version History 섹션을 포함하여 변경 사항도 문서화하는것이 좋다.

```markdown
## Version History
### 1.0.0
- Initial release

### 1.0.1
- Fixed a bug
```

### 디버깅 기법
claude --debug 명령을 사용하면 스킬 로딩 과정에서 발생하는 문제 확인 가능

디버그 로그는 ~/.claude/debug/ 에 기록되며 디렉토리 하위 세션별로 저장된다.

디버그 로그를 통해 핵심정보 3가지는 다음과 같다.
- 스킬 검색 경로
- 동일한 이름의 스킬이 여러 위치에 있는지? (duplicates removed : 0)
- 플러그인으로 제공된 스킬이 정상적으로 로드되었는지? (plugin skill: 0)

### 재사용 가능한 스킬 패턴
다양한 상황에 적용할 수 있도록 description에 충분한 트리거 조건을 포함한다.

프로젝트 특화 패턴은 특정 프로젝트의 규칙을 명시한다. (아키텍쳐, 명명 규칙, 코딩 스타일)

도메인 특화 패턴은 특정 기술 스택이나 프레임워크에 특화된 규칙을 명시한다.

범용 유틸리티 패턴은 코드 리뷰, 문서 생성, 테스트 작성, 리팩토리등 규칙을 담는다.


### 스킬 문서화 유의 사항
스킬 자체도 문서이기에 모든 스킬을 나열하는 인덱스 문서도 관리하면 스킬 발견과 활용에 용이.

## 6. 스킬 격리 실행 (context:fork) 메커니즘
많은 토큰을 소비하는 작업이나 중간 결과물이 대화의 품질을 저하시키는 경우 격리 실행이 필요하다.

프론트매터 옵션에 `context : fork`를 설정하면 스킬이 **메인(부모) 대화가 아닌 완전히 격리된 서브 에이전트 컨텍스트에서 실행**된다.

`context : fork` 가 설정된 스킬은 **본문이 서브에이전트를 구동하는 프롬프트가 된다.**

```markdown
---
name: deep-research
description: Research a topic thoroughly
context: fork
agent: Explore
---

Research thoroughly:

1. Find relevant files using Glob and Grep
2. Read and analyze the code
3. Summarize findings with specific file references
```


서브 에이전트는 메인(부모) 대화에 접근할 수 없다. 작업이 완료되면 결과가 메인(부모) 대화에 반환된다.

agent 필드는 context:fork가 설정된 경우에만 유의미하다. <br>
agent 필드 옵션:
- general-purpose : 모든 도구를 사용하는 범용 에이전트 (기본값)
- Explore : 읽기 전용 도구만 사용하는 빠른 탐색 에이전트 (Haiku 모델)
- Plan : 컨텍스트 수집 및 계획 수립용 읽기 전용 에이전트

**Explore 및 Plan 에이전트는 컨텍스트를 작게 유지하기 위해 CLAUDE.md 및 git status를 건너뛴다.**
[공식 문서](https://code.claude.com/docs/ko/skills#run-skills-in-a-subagent)

<br>

### 언제 context:fork 를 적용하기에 좋은가?
- 대규모 코드 베이스 분석
  - 수백개의 파일 탐색 및 아키텍쳐 파악하는 작업은 수만 토큰을 소비할 수 있다.
  - 이 과정에서 중간 결과물들로 인해 대회의 품질이 저하될 수 있다.
  - context:fork 컨텍스트에서 탐색을 한다면 결과만 대화에 전달되기에 효율성이 극대화된다.
- 세션 격리가 필요한 워크플로우
  - 스팩 문서 생성, TDD/DDD 구현 사이클, 테스트 전략 수립과 같은 구조회된 워크플로에서는 각 단계가 독립적인 컨텍스트여야 단계간 간섭 방지
- 도구 전문화 기반 분석

- MoAI-ADK 시스템의 워크 플로 스킬은 실제로 context:fork 설정을 사용한다.
- context:fork 와 allowed-tools를 조합하면 격리와 보안을 동시에 처리 가능
