# 클로드 코드 실전 가이드: 6장 메모리와 대화 세션 관리

6장에서는 클로드 코드가 작업 맥락(컨텍스트)을 지속해서 유지하도록 돕는 정교한 4계층 메모리 시스템과, 토큰 낭비를 막고 안전하게 코드를 실험할 수 있는 세션 및 체크포인팅 관리 전략을 다룹니다. 

## 1. 4계층 메모리 시스템과 자동 메모리
클로드 코드는 매번 같은 지침을 반복하지 않도록 세션 간에 기본 설정을 기억하는 메모리 시스템을 제공합니다. 구체적인 설정일수록 더 높은 우선순위(4 > 3 > 2 > 1 순)를 가집니다.

*   **1계층 (관리형 메모리)**: 조직 전체에 적용되는 보안 및 코드 표준 정책입니다. `/Library/Application Support/ClaudeCode/CLAUDE.md` 등에 위치하며, `settings.json`의 예외 처리 설정으로도 제외할 수 없는 강력한 전사적 규칙입니다.
*   **2계층 (프로젝트 메모리)**: 프로젝트 루트의 `CLAUDE.md`에 저장되며 깃(Git)을 통해 팀원들과 공유됩니다. 코딩 스타일이나 아키텍처 결정을 담으며, 하위 디렉터리에 있는 파일은 해당 경로 접근 시 온디맨드(on-demand)로 로딩됩니다.
*   **3계층 (사용자 메모리)**: `~/.claude/CLAUDE.md`에 저장되는 개인 전역 설정입니다. 본인만 사용하는 단축키나 선호하는 에디터 스타일을 정의합니다.
*   **4계층 (프로젝트 로컬 메모리)**: `CLAUDE.local.md`에 저장되며 깃에 커밋되지 않습니다. 개인의 로컬 테스트 DB URL이나 샌드박스 설정 등에 활용합니다.
*   **자동 메모리 (Auto Memory)**: `settings.json`에서 활성화하면 클로드가 세션 중 학습한 선호 코딩 패턴이나 빌드 방식 등을 `MEMORY.md` 파일에 자동으로 기록합니다. 파일의 처음 200줄이 다음 세션 시작 시 로드되어 맥락을 자연스럽게 이어줍니다.

## 2. 가져오기(@)와 모듈형 룰 시스템
컨텍스트 토큰을 절약하기 위해 핵심 지침만 유지하고 상세 내용은 동적으로 로드하는 모듈화 전략이 필수적입니다.

### @ 가져오기 (동적 로딩)
`CLAUDE.md` 내부에 `@docs/architecture.md`와 같이 작성하면 해당 파일 내용을 자동으로 읽어옵니다. 
*   **5단계 재귀 제한**: 가져온 파일 안에서 또 다른 파일을 참조하는 재귀적 가져오기는 최대 5단계까지만 허용되어 무한 순환 참조를 방지합니다.
*   **코드 블록 내 예외 처리**: 마크다운 코드 블록(백틱) 내부에 작성된 `@패키지명`이나 데코레이터 패턴은 일반 텍스트로 처리되어 예기치 않은 로딩이나 충돌을 막아줍니다.

### 모듈형 룰 시스템 (`.claude/rules/`)
`CLAUDE.md`가 "무엇을 기억할 것인가"라면, 모듈형 룰은 **"언제 어떤 지침을 적용할 것인가"**를 세밀하게 제어합니다.
*   특정 언어나 작업에만 필요한 룰 파일을 만들어 두고, 파일 상단 프론트매터에 `paths: "**/*.py"`와 같이 지정합니다. 
*   이렇게 설정하면 **클로드가 파이썬 파일을 다룰 때만 해당 코딩 룰을 로드**하므로, 불필요한 토큰 낭비를 획기적으로 줄일 수 있습니다. 반면 보안 정책처럼 항상 적용되어야 하는 규칙은 `paths` 없이 작성합니다.

## 3. 체크포인팅과 깃(Git) 하이브리드 워크플로
클로드 코드는 **모든 프롬프트 입력마다 30일간 보관되는 체크포인트를 자동 생성**합니다. `Esc` 키를 두 번 누르거나 `/rewind` 명령어로 언제든 과거로 되돌아갈 수 있습니다.

### 선택적 복구 옵션
되감기 메뉴에서는 세 가지 복구 방식을 선택할 수 있습니다.
1.  **대화만 복구**: 작성된 코드 변경 사항은 그대로 유지하되, 클로드의 응답 방향이 마음에 들지 않아 프롬프트만 다시 작성하고 싶을 때 사용합니다.
2.  **코드만 복구**: 클로드가 코드를 잘못 수정했을 때, 대화의 맥락은 그대로 보존하면서 코드 파일만 이전 상태로 되돌립니다.
3.  **대화와 코드 모두 복구**: 특정 시점으로 세션 전체를 완전히 되돌리고 새로 시작할 때 사용합니다. 잘못 소모된 토큰 비용과 오염된 컨텍스트가 깔끔하게 세탁됩니다.

### 하이브리드 워크플로 전략
체크포인트는 `rm`, `mv` 같은 터미널 Bash 명령어를 통한 수정은 추적하지 못합니다. 이를 보완하기 위해 클로드 코드의 `post-bash hook`을 설정하여 Bash 명령 실행 후 자동으로 깃 커밋(Git Commit)을 남기도록 구성할 수 있습니다.
결론적으로 **실험적이고 빠른 변경 시도는 체크포인팅으로 반복하고, 만족스러운 최종 결과가 나오면 깃으로 커밋하여 영구 보존하는 하이브리드 방식**이 가장 이상적입니다.

## 4. 다중 세션 관리와 컨텍스트 최적화
프로젝트가 커질수록 컨텍스트 윈도우가 소모되어 응답 속도가 느려지고 정보 유실이 발생할 수 있습니다.

*   **세션 상태와 격리**: 세션은 활성, 유휴, 보관(30일 후 자동 정리) 상태를 거칩니다. 기능 개발, 버그 수정, 코드 리뷰는 서로 다른 독립 세션으로 분리해야 컨텍스트 오염을 막을 수 있습니다. 무거운 분석 작업은 20만 토큰의 독립 컨텍스트를 가지는 서브에이전트에게 위임하고 결과만 돌려받는 것이 좋습니다.
*   **강제 초기화와 내보내기**: 내장된 자동 압축(Auto Compact) 기능은 유용하지만 50% 이상 압축 시 중요 정보가 유실될 위험이 있습니다. 따라서 토큰 한계에 도달하면 `/export` 명령으로 중요 대화나 결정 사항을 외부 파일(`decisions.md`)로 내보낸 뒤, `/clear` 명령으로 세션을 완전히 초기화하는 것이 훨씬 안전한 최적화 방법입니다.
*   **ROI 기반 메모리 저장**: 컨텍스트 윈도우 낭비를 막기 위해 세션당 5회 이상 참조되는 100토큰 분량의 핵심 규칙만 메모리에 저장하고, 0.2회 수준으로 드물게 참조되는 방대한 정보는 필요할 때 프롬프트나 조건부 로딩으로 넘겨야 합니다.

## 5. 100만 토큰 환경과 원격 제어
*   **1M 컨텍스트 윈도우 활용**: Opus 4.6 및 Sonnet 4.6 모델은 100만(1M) 토큰의 거대한 컨텍스트를 지원합니다. 이를 통해 파일 분석 후 매번 압축(`/compact`)해야 했던 번거로움을 크게 줄일 수 있습니다. 단, 20만 토큰을 초과하면 추가 비용이 발생하므로 제어가 필요하다면 `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` 환경 변수를 선언하여 기존 20만 토큰 환경으로 강제 제한할 수 있습니다.
*   **웹 원격 제어 (Remote Control)**: 로컬 터미널에 클로드 코드 세션을 켜둔 상태라면, 동일한 계정으로 `claude.ai` 웹 인터페이스에 로그인하여 외부 브라우저에서도 로컬 세션에 접속할 수 있습니다. 이를 통해 로컬의 파일 시스템에 접근하여 외출 중에도 개발 작업을 원격으로 모니터링하고 가벼운 코드 수정을 지시할 수 있습니다.
