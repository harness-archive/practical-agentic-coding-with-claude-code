# 6장 메모리와 대화 세션 관리

## 1. 메모리 시스템 이해하기
- CLAUDE.md 파일과 자동 메모리 시스템을 통해 프로젝트 컨벤션, 가이드라인, 자주 쓰는 명령어 등를 지속적으로 유지한다.
- CLAUDE.md 파일에 작성된 지침을 세션 시작시 자동으로 로드하여 컨텍스트에 포함시킨다.

### 1.1 메모리 계층 구조
- 네 가지 계층 구조로 구성되어있으며 메모리 파일을 계층적으로 읽는다.

| 메모리 타입      | 위치                                                          | 용도                  | 공유 범위       | 우선순위 |
|-------------|-------------------------------------------------------------|---------------------|-------------|-----|
| 관리형 정책      | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS) | 보안 정책, 전사 규칙 등      | 조직 내 모든 사용자 | 4   | 
| 프로젝트 메모리    | 프로젝트 루트의 `./CLAUDE.md` 또는 `./.claude/CLAUDE.md` | 프로젝트별 컨벤션, 빌드 명령어 등 | 프로젝트 내      | 3 |
| 사용자 메모리     | `~/.claude/CLAUDE.md` | 개인 스타일, 개인 워크플로 등   | 개인 모든 프로젝트  | 2 |
| 프로젝트 로컬 메모리 | `./CLAUDE.local.md` | 개인 프로젝트 설정  | 개인 현재 프로젝트  | 1 |


### 1.2 자동 메모리
- settings.json에 autoMemoryEnabled를 true로 설정하면 Claude가 세션 중 학습한 내용을 스스로 기록. 프로젝트 패턴, 디버깅 인사이트, 선호도 등을 자동으로 저장
- `~/.claude/projects/<project-hash>/MEMORY.md 파일에 저장되며, 세션 시작시 자동으로 로드하여 이전 세션 내용과 자연스럽게 연결된다.
- 현재 작업 디렉토리에서 시작해서 루트 디렉토리까지 재귀적으로 올라가며 CLAUDE.md  || CLAUDE.local.md 파일을 읽는다.

### 1.3 CLAUDE.md 가져오기와 조건부 로딩
- `@경로` 구문을 통해 클로드 코드가 세션 시작시 참조된 파일의 내용을 컨텍스트에 동적 로딩 방식으로 포함시킬 수 있다.
- 프롬프트에도 적용이 가능하며 작업이 끝나면 컨텍스트에서 제거되어 토큰을 절약한다.

ex)
```markdown
프로젝트 개요는 @README.md를 사용
git 워크플로는 @docs/git-workflow.md를 사용
홈 디렉토리의 개인 설정은 @~/.claude/my-style.md를 사용
```
- `@경로` 구문을 통해 가져온 파일 내에 또 `@경로` 구문이 있을 경우, 재귀적으로 파일을 로드하되 최대 다섯 단계까지만 허용한다. (무한 루프 방지)
- `@경로` 구문을 백틱이나 코드 블록내에 감싸면 일반 텍스트로 처리된다.
- 이 구문을 통해 필요한 정보를 필요한 시점에 로드하여 모듈화 전략이나 도메인별 독립된 파일로 관리할 수 있다.

### 1.4 메모리 시스템을 자유자재로
- `/memory` 명령어를 통해 현재 세션에 로드된 파일 목록을 확인할 수 있다.
- `/init` 명령어를 통해 클로드가 코드베이스를 분석하여 빌드 명령어, 테스트 지시사항, 프로젝트 컨벤션을 자동으로 담은 CLAUDE.md를 생성
  - Q. CLAUDE.md 파일이 이미 존재하면 덮어쓰나요?
  - A. 아니요. 이미 존재하는 경우에는 덮어쓰지 않고 제안한다.

### 1.5 메모리 최적화와 장기 작업 관리
- 대규모 프로젝트나 장기작업에서는 메모리를 관리하지 않으면 컨텍스트 윈도우 낭비하거나 일관성있는 답변을 하지않거나 메모리 로드하고 파싱하는데 시간이 걸리므로, 응답 지연이 발생하게된다.
- ROI를 계산하여 특정 정보를 매번 프롬프트에 전달하면 좋을지 메모리에 저장하면 좋을지 판단하도록 한다. 
- 프로젝트 루트의 CLAUDE.md는 전체 프로젝트의 규칙을 담고 각 모듈 디렉토리의 CLAUDE.md는 모듈별 규칙을 담도록 한다.
- 메모리 파일은 하위처럼 단계적으로 발전시켜야한다.
1. 부트스트랩 메모리 : /init 명령으로 자동 생성되는 메모리는 기본 프로젝트 정보만 담는다
2. 경험 기반 메모리 : 개발을 진행하면서 자주 사용하는 패턴, 반복되는 지침 등을 추가한다.
3. 최적화된 메모리 : 메모리를 정기적으로 검토하고 더이상 사용하지 않는 정보는 정리한다.

### 1.6 모듈형 룰 시스템
- `.claude/rule/` 디렉터리 기반으로 모듈식으로 지침을 여러 파일로 구성할 수 있다.
- 아래처럼 하위 디렉토리로 규칙을 구성할 수 있다.
    ```
    your-project/
    ├── .claude/
    │   ├── CLAUDE.md           # 주 프로젝트 지침
    │   └── rules/
    │       ├── code-style.md   # 코드 스타일 가이드라인
    │       ├── testing.md      # 테스트 규칙
    │       └── security.md     # 보안 요구사항
    ```
- CLAUDE.md가 `무엇을 기억할 것인가` 를 정의한다면 모듈형 룰은 `언제 어떤 지침을 적용할 것인가`
- frontmatter 에 `path` 필드를 통해 조건부 로딩이 가능하다.
```markdown
---
paths:
  - "src/**/*.{ts,tsx}"
  - "lib/**/*.ts"
  - "tests/**/*.test.ts"
---
```
- paths 필드가 없는 규칙은 무조건 로드되며 모든 파일에 적용된다.
- paths 필드에서 glob 패턴을 사용하여 확장명, 디렉토리 또는 조합으로 파일을 일치시킨다.

| 패턴          | 일치                           |
|-------------|------------------------------|
| **/*.ts      | 모든 디렉토리의 모든 TypeScript 파일    | 
| src/**/*    | src/ 디렉토리 아래의 모든 파일          |
| *.md     | 프로젝트 루트의 마크다운 파일             |
| src/**/*.{ts,tsx} | src/ 디렉토리 아래의 ts, tsx 확장자 파일 |

## 2. 체크포인팅과 복구 시스템
클로드 코드 세션의 코드 상태와 대화 컨텍스트를 이전 시점으로 되돌리는 방법은 2가지가 있다.
1. ESC 연속 두번 누르기
2. `/rewind` 명령어를 통해 인터페이스

### 2.1 rewind
Rewind 인터페이스가 열리면 되돌아가고 싶은 시점을 선택하면 5가지 옵션이 나온다.
- `Restore code and conversation` : 코드와 대화를 모두 되돌리기
- `Restore conversation` : 대화만 되돌리고 현재 코드는 그대로 유지
- `Restore code`: 파일 변경만 되돌리고 대화는 그대로 유지
- `Never mind` :  아무것도 하지 않는다.

### 2.2 체크포인팅의 제한 사항과 대응 전략
- 체크포인트로 rm, mv, cp 같은 Bash 명령어로 변경한 파일은 추적되지 않는다. 외부 시스템에 대한 변경도 마찬가지

### 2.3 체크포인팅과 깃
- 체크포인트는 세션 단위의 빠른 복구용, 영구적인 이력 관리는 Git
- 체크포인트를 빠르게 활용하여 시도하고 되돌리고 만족스러운 결과는 git 커밋으로 영구 보존하여 하이브리드로 관리하자.
- 계획 모드와 Rewind를 조합하여 계획 모드로 Claude가 무엇을 할지 미리 확인하고 계획이 괜찮으면 실행시키고 <br>
결과가 예상과 다르면 Rewind로 되돌린 뒤 계획 모드에서 다른 방향으로 시도

## 3. 세션 관리 전략
세션은 3가지 상태를 가진다.
1. 활성 (Active) : 현재 대화중인 세션, 컨텍스트가 메모리에 로드되어있음
2. 유후 (Idle) : 일시 중단된 세션, 
3. 보관 (Archived) : 세션 종료

세션은 30일동안 보관되며 이 기간동안 체크포인트가 유지된다. 

### 3.1 컨텍스트 윈도우 최적화와 세션 복구
- /clear: 빈 컨텍스트로 새로 시작, 체크포인트는 유지되지 않는다.
- /export: 현재 대화를 클립보드에 복사하거나 일반 텍스트 파일로 저장
- @ 참조로 파일을 로드해 사용
- 독립적인 작업은 서브 에이전트에 위임
- 반복 참조 정보는 메모리 파일에 저장

### 3.2 다중 세션 관리
독립적인 작업은 별도 세션에서 진행하고, 관련있는 작업은 컨텍스트를 공유해야한다. <br>

서브 에이전트는 독립적인 20만 토큰 컨텍스트를 가지므로 메인 세션의 제약을 받지 않는다. <br>

서브 에이전트가 중단되었거나 추가 지시가 필요할 때 agentId로 제어할 수 있다. [서브 에이전트 재개](../4회차/오태경.md#서브-에이전트-재개)

### 3.3 Remote Control
claude.ai/code 웹 인터페이스나 모바일 앱에서 로컬 Claude Code 세션을 원격으로 제어할 수 있다.

명령어로 시작 claude remote-control 하거나 활성 세션에서 /remote-control로 시작할 수 있다.
