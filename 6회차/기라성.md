# 6. 메모리와 대화 세션 관리

## 6-1 메모리 시스템 이해하기
- 클로드코드는 세션 간에 기본 설정을 기억하는 계층적 메모리 시스템을 제공한다. 
- 이 시스템은 코드 스타일 가이드라인, 워크플로의 일반 명령어, 프로젝트별 지침등을 저장하고 관리한다. 
- CLAUDE.md 파일에 작성된 지침을 세션 시작 시 자동으로 로드하여 컨텍스트에 포함시킨다. 

### 1) 네 가지 계층적 메모리 (계층 순서로 우선순위 적용)
- 첫 번째 계층 : 관리형 정책
  - 이 파일은 조직 전체에 적용되는 지침을 담으며 IT나 데브옵스 팀에서 관리한다. 
  - macOS에서는 /Libraray/Application Support/ClaudeCode/CLAUDE.md 경로
  - 조직의 모든 사용자가 이 설정을 공유하며 보안 정책이나 코드 표준 같은 전사적 규칙을 정의하는데 활용
- 두 번째 계층 : 프로젝트 메모리
  - 프로젝트 루트의 ./CLAUDE.MD 또는 ./.claude/CLAUDE.md 파일에 저장되며 소스 제어를 통해 팀에 공유된다. 
  - 프로젝트별 코딩 스타일, 아키텍처 결정, 빌드 명령어 등을 문서화 하는데 적합하다. 
- 세 번째 계층 : 사용자 메모리
  - ~/.claude/CLAUDE.md 파일에 저장되며 개인의 모든 프로젝트에 적용되는 기본 설정을 담는다.
  - 본인만 사용하며 선호하는 에디터 스타일, 자주 사용하는 단축키 패턴, 개인적은 워크플로 등을 정의한다. 
- 네 번째 계층 : 프로젝트 로컬 메모리 
  - ./CLAUDE.local.md 파일에 저장되며 현재 프로젝트에 대한 개인적인 기본 설정을 담는다. 
  - 이 파일은 깃에 커밋되지 않으며 개인의 샌드박스 URL이나 선호하는 테스트 데이터, 로컬 개발 환경 설정 등 프로젝트에서 개인적으로 적용할 설정을 담는데 활용한다. 

### 2) 자동 메모리
- settings.json에서 autoMemoryEnabled: true일 때 활성화 되며 클로드가 세션 중 학습한 내용, 예를 들어 프로젝트의 빌드 방식, 선호하는 코딩 패턴, 자주 사용하는 테스트 명령어등을 자동으로 저장한다. 
- ~/.claude/projects/<product-hash>/MEMORY.md 파일에 저장

### 3) CLAUDE.md 가져오기와 조건부 로딩
- @path/to/import 구문은 외부 파일을 CLAUDE.md의 컨텍스트로 가져오는 핵심 기능
- 이 구문은 클로드 코드가 세션 시작시 메모리 파일을 파싱하는 과정에서 참조된 파일의 내용을 컨텍스트에 포함시키는 동적 로딩 방식으로 동작한다. 
- 이를 통해 CLAUDE.md 자체는 간결하면서도 체계적으로 지침을 관리할 수 있다. 

### 4) 메모리 시스템을 자유자재로
#### /memory 명령으로 직접 편집
- 세션 중에 /memory 명령을 사용하면 시스템 편집기에서 메모리 파일을 열 수 있다.
- 광범위한 추가나 구성이 필요할 때 유용하다. 

#### /init 명령으로 프로젝트 메모리 부트스트랩
- 새 프로젝트에서 메모리를 설정하려면 /init 명령을 사용한다. 
- 이 명령은 코드베이스를 분석하여 적절한 CLAUDE.md 파일을 생성한다. 

### 5) 메모리 최적화와 장기 작업 관리
- 장기 작업에서는 메모리 관리가 특히 중요하며 대규모 프로젝트에서 컨텍스트 윈도우가 제한되므로 불필요한 정보가 축적되면 성능이 저하될 수 있다. 
- 메모리가 비효율적으로 관리되면 다음과 같은 문제가 발생한다. 
  - 컨텍스트 윈도우 낭비
  - 응답 지연
  - 일관성 손실
#### 대규모 프로젝트를 위한 모듈화 전략
- 단일 CLAUDE.md 파일에 모든 정보를 담으면 유지보수가 어렵고 컨텍스트 관리도 비효율 적이다. 
- 프로젝트 루트의 CLAUDE.md는 전체 프로젝트의 일반 규칙을 담고 각 모듈 디렉토리에 CLAUDE.md는 모듈별 특수 규칙을 담는다.

#### 동적 메모리 로딩 패턴
- 클로드에 특정 작업 시작시 관련 메모리를 활성화하도록 요청할 수 있다.
- 이 방식은 작업별로 필요한 정보만 컨텍스트에 로드하여 토큰을 절약한다. 
```text
### FE 작업시
> 프론트엔드 작업을 시작할게. @docs/frontend-guidelines.md를 참고해서 리팩토링 해줘
### DB 작업시
> 데이터 베이스 관련작업할때는 @docs/database-best-practice.md를 참조해줘
```

#### 메모리 파일 성숙도 모델
- 메모리 파일은 프로젝트와 함께 단계적으로 발전시켜야한다. 
  - 1단계-부트스트랩 메모리 : /init 명령으로 자동 생성된 메모리는 기본 프로젝트 정보만 담는다
  - 2단계-경험 기만 메모리 : 개발을 진행하면서 자주사용하는 패턴, 반복되는 지침, 팀 규칙 등을 메모리에 추가한다. 
  - 3단계-최적화된 메모리 : 메모리를 정기적으로 검토하고 정리한다. 더이상 사용하지 않는 정보는 제거하고 자주 참조되는 정보는 상위로 이동하며 관련 정보는 그룹화한다. 

### 6) 모듈형 룰 시스템
- 단일 파일에 모든 지침을 담았을때 일어날 수 있는 문제를 해결하기 위해 .claude/rules/ 디렉토리 기반의 모듈형 룰 시스템을 제공한다. 
- 프로젝트 지침을 주제별, 도메인별로 독립된 마크다운 파일로 분리하고 각 파일의 적용 조건을 프로트매터로 명시한다. 

#### 디렉토리 구조와 구성 원칙
- 룰 파일은 마크다운 형식으로 작성하고 필요에 따라 프론트 매터를 포함한다. 
```text
.claude/rules
|
|----core/
|     |------ constitution.md     # 핵심 원칙 (항상 로드)
|     |------ security.md     # 보안 원칙 (항상 로드)
|
|----development/
|     |------ coding-standards.md     # 코딩 표준 (항상 로드)
|     |------ skill-authoring.md      # 스킬 작성 가이드 (항상 로드)
|
|----languages/
|     |------ python.md     # pyhon 규칙 (*.py 작업시에만 로드)
|     |------ go.md         # go 규칙 (*.go 작업시에만 로드)
```

#### CLAUDE.md 와의 관계와 차이점
- 둘은 상호 보완적인 관계
- CLAUDE.md는 프로젝트의 핵심 아이덴티티와 고수준 지침을 담는 진입점이고 .claude/rules/는 세부적인 실행 규칙을 모듈화하여 관리하는 확장 메커니즘
- CLAUDE.md에는 프로젝트의 핵심 원칙, 전체 아키텍처 방향, 팀 공유가 필요한 기본 설정 등 항상 필요한 최소한의 정보만 담는다
- .claude/rules/에는 언어별 코딩 규칙, 도메인별 아키텍처 가이드, 워크플로 정의, 도구별 설정 등 상세하고 조건부로 적용되는 규칙을 배치한다. 
- CLAUDE.md에서 @.claude/rules/ 경로의 파일을 참조하여 연결할 수도 있다.

### 7) 사용자 수준 룰과 고급 기능
- 사용자 홈 디렉토리의 ~/.claude/rules/ 에 사용자 수준 룰 파일을 둘 수 있다. 
- 모든 프로젝트에 적용된느 개인 룰을 정의하며 프로젝트 룰과 마찬가지로 자동 로딩된다. 


## 6-2 체크포인팅과 복구 시스템
### 1) /rewind 화면과 되감기 사용법
- 체크포인트로 되돌아가는 방법은 2가지가 있다. /rewind 명령어를 사용하거나 esc 키를 2번 연속으로 누르면 된다. 
  - Restore conversation(대화만) : 코드 변경 사항ㅇ르 유지하면서 사용자 메시지로 되감기. 클로드의 응답이 마음에 들지 않아 다른 방향으로 다시 시도하고 싶지만 이미 작성된 코드는 유지하고 싶을때 유용
  - Restore code(코드만) : 대화를 유지하면서 파일 변경 사항 되돌리기
  - Restore code and conversation(코드와 대화 모두) : 세션의 이전 지점으로 둘 다 복구

### 2) 체크포인팅의 제한 사항과 대응 전략
- rm, mv, cp 같은 bash 명령을 통한 파일 수정은 추적되지 않는다.
- 하지만 hooks의 post-bash hook을 설정하면 bash 명령으로 인한 변경도 기록할 수 있다. 
- 클로드가 bash 명령을 실행한 후 자동으로 깃 커밋을 생성하도록 설정하면 깃 히스토리를 통해 복구할 수 있다.

## 6-3 세션 관리 전략
- 세션 생애 주기를 이해하면 컨텍스트를 효과적으로 관리하고 장기 작업에서 최적의 성능을 유지할 수 있다. 
- 세션은 크게 3가지 상태를 거친다.
  - 활성 상태(Active) : 현재 대화중인 세션으로 모든 컨텓스트가 메모리에 로드된다.
  - 유휴 상태(IDE) : 대화가 일시 중단된 세션으로 컨텍스트가 저장되어있지만 활성화 되지 않은 상태
  - 보관 상태(Archived) : 세션이 종료되어 체크포인트만 유지되는 상태

### 1) 컽넥스트 윈도우 최적화와 세션 복구
- 컨텍스트 윈도우는 다음 네가지 방법으로 최적화가 가능
  - /clear 명령
  - @참조로 파일을 로드해 사용
  - 독립적인 작업은 서브에이전트에 위임
  - 반복적으로 참조해야 하는 정보는 메모리 파일(CALUDE.md)에 저장

### 2) 세션 데이터 보관
- 중요한 세션의 데이터를 장기 보관해야 할 때는 다음처럼 대화 기록을 마크다운 파일로 내보내는것이 가장 간단하다.
