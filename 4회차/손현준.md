## 1. 오케스트레이터-워커 패턴과 컨텍스트 격리
서브에이전트는 클로드 코드가 모든 것을 단독으로 처리하는 대신, 도메인별 특화된 AI 어시스턴트에게 작업을 위임하는 시스템입니다.

- **오케스트레이터-워커 (Orchestrator-Worker)**: 메인 에이전트가 작업 분배와 조율(오케스트레이터)을 맡고, 서브에이전트가 실제 분석과 구현(워커)을 담당합니다. 복잡성을 줄이기 위해 한 에이전트가 코드 리뷰, 디버깅 등 하나의 역할만 전담하는 **단일 책임 원칙**이 권장됩니다.
- **컨텍스트 격리**: 서브에이전트 사용의 가장 큰 장점입니다. 각 서브에이전트는 독립적인 20만 토큰의 컨텍스트를 가지므로, 방대한 로그나 코드를 분석한 뒤 **핵심 요약 결과만 메인 대화로 반환하여 부모 컨텍스트의 오염과 낭비를 막습니다**.
- **실행 패턴**: 이전 결과를 다음 에이전트의 입력으로 넘기는 순차 파이프라인 구조나, 프론트엔드/백엔드 분석을 동시에 진행하는 병렬 실행 구조로 설계할 수 있습니다.

## 2. 서브에이전트 호출 및 작업 재개
- **자동 위임과 명시적 호출**: 프롬프트에 `@에이전트명(agent)` 형태로 직접 지정하거나, 자연어로 요청하면 클로드가 에이전트에 정의된 `description`을 바탕으로 적절한 에이전트를 자동 선택해 실행합니다.
- **작업 재개 (Resume)**: 서브에이전트가 생성될 때 부여받는 고유한 `agentId`를 활용해 대화 상태를 복원할 수 있습니다. "에이전트 a9adef5을 재개하고 권한 로직도 추가 분석해 줘"라고 요청하면, 별도 저장된 `.jsonl` 파일에서 이전 문맥 전체를 로드해 중단된 지점부터 이어 작업합니다.

## 3. 커스텀 에이전트 개발과 고급 설정
프로젝트 환경에 맞는 에이전트를 만들기 위해 `.claude/agents/` 디렉터리에 마크다운(Markdown) 파일을 생성하여 관리합니다. 상단의 YAML 프론트매터(Frontmatter)를 통해 에이전트의 동작을 정밀하게 제어합니다.

- **기본 설정**: 에이전트 식별을 위한 `name`, 자동 위임의 기준이 되는 `description`, 접근 가능한 도구를 제한하는 `tools`, 사용할 AI 모델을 지정하는 `model` 필드가 있습니다.
- **고급 제어 권한 (`permissionMode`)**: `plan` (파일 읽기 전용), `acceptEdits` (파일 수정 자동 승인), `bypassPermissions` (모든 권한 확인 생략) 등의 모드를 지정하여 에이전트의 자율성과 보안을 통제합니다.
- **자동 훅과 기억 (`hooks` & `memory`)**: `PreToolUse`, `PostToolUse`, `Stop` 등의 라이프사이클 이벤트 발생 시 터미널 명령을 자동 실행할 수 있으며, `memory` 필드를 통해 세션이 종료되어도 에이전트가 학습한 내용을 영구적으로 저장(user, project, local 범위)할 수 있습니다.
- **스킬 주입 (`skills`)**: 3장에서 다룬 스킬들을 불러올 수 있으며, 범용 규칙(기반 스킬)을 먼저 선언하고 프로젝트 특화 스킬을 나중에 선언하는 방식으로 로딩 순서(우선순위)를 조절합니다.

## 4. 에이전트 팀(Agent Teams)을 통한 협업
단발성 요청/응답 구조인 서브에이전트의 한계를 넘어, 여러 에이전트가 동시에 협력하는 진보된 구조입니다 (실험적 기능).

- **영속적(Persistent) 협업**: 에이전트 팀의 팀원들은 작업이 끝나도 즉시 소멸하지 않고 팀이 해산될 때까지 자신의 컨텍스트를 유지하며 활동합니다.
- **자율 태스크 조율**: 메인 세션이 리더 역할을 맡아 공유 태스크 목록을 생성하면, 팀원들이 메일박스를 통해 소통하며 미해당 작업을 자율적으로 선택(`self-claim`)해 처리합니다.
- **파일 소유권 (File Ownership)**: 동시 수정 충돌을 막기 위해 프론트엔드 에이전트는 `src/ui/`를, 백엔드는 `src/api/`를 담당하는 식으로 배타적인 파일 소유권을 할당하여 안전성을 확보합니다.
- **적용 기준**: 3개 이상의 도메인이 엮여 있거나 수정 대상 파일이 10개 이상일 때 에이전트 팀의 효율이 극대화됩니다.

## 5. CLI 기반 동적 에이전트 파이프라인
정적인 파일 설정 없이, 자동화 스크립트나 CI/CD 파이프라인 내에서 즉석으로 에이전트를 활용하는 방식입니다.

- **`--agent` 플래그**: 기존에 정의해 둔 에이전트를 직접 지정하여 비대화형(`-p`)으로 실행합니다. (예: 배포 전 CI 환경에서 보안 감사 전용 에이전트 자동 실행).
- **`--agents` 플래그**: 파일 시스템에 마크다운을 만들 필요 없이, 명령어 란에 JSON 형식으로 에이전트의 이름, 설명, 프롬프트를 인라인(Inline)으로 정의해 즉석에서 호출합니다.
