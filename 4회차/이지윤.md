# 4-1 서브에이전트의 개념과 아키텍처
- 메인 클로드 코드가 "뭘 해야 하는지(스킬)"을 안다면, 서브에이전트는 그 지식을 실제로 "누가, 어떻게 실행하는지"를 담당하는 존재

### 핵심 정리
- 서브에이전트 = Claude Code의 Agent 도구로 호출되는 특화 AI 어시스턴트
- 각자 독립적으로 가진 것: 컨텍스트 윈도우, 커스텀 시스템 프롬프트, 할당된 도구
- 메잌 클로드 코드 인스턴스 하나가 모든 일을 처리하는 방식과는 근본적으로 다름

### 왜 중요한가(컨텍스트 격리)
- 여러 서브에이전트에게 작업을 위임 → 각자 따로 작업 수행
- 메인 대화창에는 핵심 결과만 반환됨
- 즉, 메인 컨텍스트가 지저분해지지 않고 "정제된 정보"만 쌓임

### 정의 및 저장 방식
- 마크다운 파일로 정의
- 파일 구성 = YAML 프론트메터 + 마크다운 본문(=시스템 프롬프트)
 
## 에이전트 위임 패턴: 오케스트레이터 - 워커
메인 에이전트(오케스트레이터)는 "팀장, 서브에이전트(워커)는 "각 분야 담당자"

### 구조 요약
- 패턴명: 오케스트레이터-워커
- 비유: 마이크로서비스 구조 - 각 서비스(서브에이전트)가 자기 영역에서 최적 성능 발휘
- 서브에이전트의 전문 영역 예: 코드 리부, 디버깅, 테스트 작성 등

### 오케스트레이터의 역할
1. 작업 분배
2. 결과 통합

## 단일 책임 원칙 서브에이전트
단일 책임의 효과
1. 에이전트 동작이 예측 가능해짐
2. 지침 작성이 명확해짐
3. 문제 발생 시 원인 파악이 쉬워짐

## 컨텍스트 엔지니어링 원칙
- 서브에이전트에게 "무엇을 보여주고, 무엇을 숨길지"를 설계하는 기술
- 적정 고도(right altitude) - 방향은 주되 판단할 여지는 남기는 수준

### 두가지 작동 차원
1. 초기 컨텍스트 설계: 코드 리뷰 에이전트 호출 시 전체 코드베이스가 아니라 변경된 파일 + 관련 의존성만 전달
2. 작업 중 컨테그스트 관리: 에어전트가 파일을 읽고 명령을 실행하며 컨텍스트가 누적되므로, 필요한 정보만 선택적으로 수집하도록 지침 제공

### 에이전트 체인과 병렬 실행
순차 실행은 `.then().then()` 프로미스 체인처럼 앞 결과가 다음 입력이 되는 파이프라인, 병렬 실행은 `Promise.all()`처럼 서로 무관한 작얿을 동시에 처리하는 방식

### 설계 시 주의 사항
1. 각 단계의 입출력 형식을 명확시 정의(앞 에이전트 출력 = 뒤 에이전트 기대 입력과 일치해야 함)
2. 실패 처리 방법 고려
   - 일부 실패: 무시하고 진행 가능
   - 치명적 실패: 전체 파이프라인 중단 + 오류 보고
  
## 다중 클로드 워크플로
- 서브에이전트: 같은 클로드 코드 세션 하나 안에서 역할을 나눠 부름
- 다중 클로드 워크플로: 터미널 창을 여러 개 열어서 클로드 코드 자체를 여러 개 동시에 돌림  
Tip: 지금은"에이전트 팀"이라는 공식 기능이 있어서 여러 클로드를 조율하는 게 더 쉬워졌지만, 그 기능이 나오기 전에는 개발자들이 직접 `git worktree`로 폴더를 쪼개고 각 폴더에서 클로드 코드를 하나씩 띄우는 식으로 "수동으로" 병렬 워크플로를 만들어 씀.

### 핵심 정리
- 여러 클로드 코드 인스턴스를 병렬로, 서로 다른 터미널에서 실행
- 독립 작업을 위해 별도의 깃 작업 트리 사용
- 대규모 프로젝트에서 개발 속도 향상, 품질 향상

## 언제 서브에이전트를 사용하는가
1. 작업의 복잡도
   - 단순 파일 수정, 짧은 코드 생성 → 메인 대화에서 직접 처리
   - 여러 파일에 걸친 분석, 장시간 작업 → 서브에이전트에 위임
2. 컨텍스트 소모량
   - 대규모 파일 읽기, 명령 실행이 필요하면 서브에이전트로 메인 컨텍스트 보존
3. 전문성 요구 수준
   - 특정 도메인 전문 지침이 필요한 작업은 해당 지침으로 구성된 서브에이전트 활용
  
# 4-2. 서브에이전트 사용하기
클로드 코드는 기본 제공 서브에이전트(내장)와 직접 만든 커스텀 서브에이전트를 모두 지원하고 `Agent` 도구로 이 둘을 호출
"목록 확인 → 호출 → 결과 통합 → 작업 재개 → 에러 처리 → 성능 축정"까지 실무 흐름 순서로 이어짐

### 상요 가능한 서브에이전트 목록
`/agent` 명령은 npm의 `npm list`처럼. 지금 이 프로젝트에서 쓸 수 있는 에이전트 전체 목록을 보여줌.

| 구분 | 정의 위치 | 사용 범위 | 우선 순위 |
| 프로젝트 에이전트 | `.claude/agents/` | 해당 프로젝트에서만 | 높음 |
| 사용자 에이전트 | `~/claude/agents/` | 모든 프로젝트 | 중간 |
| Built-in Agents | 기본 내장 | 항상 사용 가능 | 낮음 |
- 같은 이름의 에이전트가 양쪽에 있으면 프로젝트 에이전트가 우선(마치 로컬 설정이 전역 설정보다 우선하는 것과 같은 원리)

### Built-in Agents(기본 내장)
- `general-purpose`(sonnet): 범용 멀티스텝 작업 처리
- `statusline-setup`(sonnet): 상태 표시줄 설정
- `Explore`(haiku): 코드베이스 탐색 및 검색
- `Plan`(inherit): 구현 계획 및 설계 - Read/Glob/Grep/Bash 도구 접근, 플랜 모드에서 코드구조 분석 시 자동호출
- `Claude-code-guide`(haiku): 클로드 코드 사용 가이드

### 에이전트 호출 문법
- 자동 호출
  - 요청 내용과 에이전트의 `description`필드를 비교해서 클로드 코드가 자동으로 적절한 에이전트 선택
  - `description`에 `use PROACTIVELY`, `MUST BE USED`같은 키워드를 넣으면 더 적극적으로 자동 위임됨
- 명시적 호출
  - 에이전트 이름을 직접 언급
  > code-reviewer 서브에이전트를 사용해서 내 최근 변경 사항을 검토해 줘.
- 컨텍스트와 함께 호출
  - `@<에이전트이름>(agent)` 형태로 파일 경로∙에러 메시지 등 추가 정보 전달
  > @code-reviewer(agent) 서브에이전트로 @src/auth/login.ts 파일을 리뷰해 줘.
  - `@` 기호 + 탭 완성으로 파일/디렉터리 쉽게 참조
 
### 에이전트 결과 통합
여러 에이전트 리뷰 결과를 하나로 합칠 때, 코드 리뷰(정적 분석) 관점과 디버거(런타임 에러) 관점이 서로 다른 우선운위를 제안할 수 있으니 프로젝트 상황에 맞춰 최종 판단이 필요

**흐름**  
1. 서브에이전트 작업 완료 → 결과가 메인 대화로 반환
2. 단일 에이전트 결과는 바로 활용 or 다음 작업 입력으로 사용
3. 여러 에이전트 결과는 통합하여 종합 결론 도출(상충하는 의견은 우선 순위 판단 필요)

### 작업 재개(resume)기능 활용
서브에이전트는 호출할 때마다 매번 새 인스턴스로 시작. 이전 작업을 이어가려면 `agentId`라는 세션 티켓을 명시적으로 사용해서 "재개"해야 함.

핵심 메커니즘
- 각 실행마다 고유한 `agentId`할당
- 대화 내용은 `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.json`에 저장
- 재개 시 이 파일에서 이전 컨텍스트 전체를 복원 + 새 지시사항 추가 실행

워크플로 예시
1. 초기 호출(agentId 요청 포함) → `agentId: a9adef5` 반환
2. 중간 확인: 진행 상황만 물어봄
3. 작업 재개: `에이전트 a9adef5를 재개하고 ~~ 계속 실행` → 이전 컨텍스트 그대로 유지한 채 이어서 작업

활용 예: 복잡한 분석을 단계별로 진행 — 1세션(구조 파악) → 2세션(문제 분석) → 3세션(개선안 도출)처럼 각 단계 사이 결과를 검토하고 방향을 조정하며 정교한 분석 가능
