## 4-1 서브에이전트의 개념과 아키텍처

### 서브에이전트란 무엇인가?

한 줄 비유: **"메인 클로드 코드"가 프로젝트 매니저라면, "서브에이전트"는 각자 전문 분야를 가진 팀원**입니다.

- 스킬(Skill)이 "무엇을 알아야 하는지"에 대한 지식이었다면, 서브에이전트는 그 지식을 실제로 활용해서 **작업을 대신 수행하는 존재**입니다.
- 서브에이전트는 클로드 코드 안에 있는 `Agent`라는 도구를 통해 호출되는, 특정 역할에 특화된 AI 어시스턴트입니다.
- 각 서브에이전트는 다음 3가지를 **자기 것으로 따로** 가지고 있습니다.
  1. 자기만의 컨텍스트 윈도우 (자기만의 "작업 기억 공간")
  2. 자기만의 시스템 프롬프트 (자기만의 "업무 지침서")
  3. 자기에게 할당된 도구 (자기가 쓸 수 있는 "연장통")
- 이게 왜 중요하냐면, 원래는 클로드 코드 인스턴스 하나가 모든 일을 처리했는데, 서브에이전트를 쓰면 일을 나눠서 각자 전문적으로 처리할 수 있기 때문입니다.

### 서브에이전트의 가장 큰 장점: 컨텍스트 격리

**컨텍스트 격리**란 쉽게 말해 "각자 자기 방에서 일하고, 결과만 보고한다"는 뜻입니다.

- 여러 서브에이전트에게 각각 다른 작업을 맡기면, 각자 자기 방(컨텍스트)에서 일을 처리하고 **핵심 결과만** 메인 대화방으로 가져옵니다.
- 그 덕분에 메인 대화(=내가 클로드와 나누는 대화창)는 지저분해지지 않고 계속 깔끔하게 정리된 상태를 유지할 수 있습니다.
- 비유하자면, 팀장에게 부하 직원 10명이 "일하는 과정에서 겪은 시행착오"까지 전부 보고하면 팀장이 정신없겠죠? 대신 "결론만" 보고하면 팀장은 핵심 판단에만 집중할 수 있습니다. 서브에이전트가 딱 이 역할을 합니다.

### 서브에이전트는 어떻게 저장되나?

- 서브에이전트는 **마크다운(.md) 파일**로 정의합니다.
- 저장 위치는 "누가 쓸 수 있는지"에 따라 두 곳으로 나뉩니다.

| 범위 | 저장 경로 | 누가 쓸 수 있나 |
|---|---|---|
| 사용자 범위 | `~/.claude/agents/` | 내 컴퓨터의 모든 프로젝트에서 사용 가능 |
| 프로젝트 범위 | `.claude/agents/` | 이 프로젝트에서만, 팀원과 공유 가능 |

- 파일 하나는 두 부분으로 구성됩니다.
  1. **YAML 프론트매터** (파일 맨 위, `---`로 감싸진 부분) → 설정값 정의
  2. **마크다운 본문** → 이 서브에이전트의 "시스템 프롬프트"(= 역할과 행동 지침)

## 에이전트 위임 패턴: 오케스트레이터-워커

**쉬운 설명**: "오케스트레이터(Orchestrator)"는 지휘자, "워커(Worker)"는 연주자라고 생각하면 됩니다. 지휘자가 직접 악기를 연주하지 않고 각 연주자에게 지시하며 전체를 조율하듯이, 메인 에이전트는 직접 코드를 짜지 않고 서브에이전트(워커)들에게 일을 나눠주고 결과를 모읍니다.

- 서브에이전트 시스템은 **오케스트레이터-워커(Orchestrator-Workers) 패턴**을 따릅니다.
- 메인 에이전트 = 오케스트레이터 역할, 서브에이전트 = 워커 역할
- 마치 각 기능을 담당하는 서비스가 자기 분야에서 최고 성능을 내는 **마이크로서비스 구조**와 비슷합니다. 서브에이전트도 코드 리뷰, 디버깅, 테스트 작성처럼 각자 자기 영역에서 전문성을 발휘합니다.

**오케스트레이터의 역할은 딱 2가지**
1. 작업을 잘게 나눠서 나눠주기 (분배)
2. 각자 가져온 결과를 하나로 합치기 (통합)

**예시 — 복잡한 리팩터링 작업 흐름**
1. 오케스트레이터가 `코드 분석 에이전트`에게 "현재 구조를 파악해줘"라고 지시
2. 그 결과를 받아서 오케스트레이터가 리팩터링 계획을 세움
3. `테스트 작성 에이전트`에게 "안전망(테스트)을 먼저 만들어줘"라고 지시
4. 오케스트레이터가 실제 리팩터링을 수행
5. `문서화 에이전트`에게 "변경 사항을 기록해줘"라고 지시

```
오케스트레이터(메인 에이전트)
        ↓ 작업 분석 · 위임 · 통합
  ┌──────────────┬────────────┬──────────────┐
  code-reviewer → debugger → test-writer
  (코드 품질 검토) (오류 진단)  (테스트 작성)
```

## 단일 책임 원칙과 서브에이전트

**쉬운 설명**: 리액트 컴포넌트 설계 원칙과 똑같습니다. 하나의 컴포넌트가 UI도 그리고, 데이터도 fetch하고, 상태 관리까지 다 하면 유지보수가 힘들어지죠? 서브에이전트도 마찬가지로, 3장의 스킬에 적용했던 "단일 책임 원칙"이 여기서도 똑같이 적용됩니다. **각 서브에이전트는 하나의 명확한 책임만 가져야 합니다.** 한 에이전트가 여러 역할을 담당하면 프롬프트가 복잡해지고, 어떻게 동작할지 예측하기가 어려워집니다.

**권장: 단일 책임**
- `code-reviewer`: 코드 품질, 보안, 유지보수성 검토
- `debugger`: 오류 진단 및 근본 원인 분석
- `test-writer`: 테스트 케이스 작성

**지양: 책임 혼재**
- `code-helper`: 코드 리뷰 + 버그 수정 + 테스트 작성 + 문서화를 한 에이전트가 다 담당 → 이렇게 짜지 말 것

**단일 책임을 지켰을 때의 효과 3가지**
1. 에이전트 동작이 예측 가능해집니다.
2. 지침(프롬프트) 작성이 명확해집니다.
3. 문제가 생겼을 때 원인을 파악하기 쉬워집니다.

## 컨텍스트 엔지니어링 원칙

**쉬운 설명**: 서브에이전트에게 "무엇을 보여주고, 무엇을 숨길지"를 설계하는 기술입니다. 함수에 전역 상태를 통째로 넘기지 않고 필요한 props만 넘기는 것과 비슷합니다. 서브에이전트를 효과적으로 쓰려면 **컨텍스트 엔지니어링(Context Engineering)**을 이해해야 합니다. 이는 AI 모델에게 제공하는 컨텍스트(=정보)를 체계적으로 설계하는 방법론입니다.

- 핵심 개념은 **적정 고도(right altitude)**입니다. 너무 상세하게 알려주면 에이전트가 핵심을 놓치고, 너무 대충 알려주면 잘못된 판단을 내립니다. 에이전트에게 방향을 줄 만큼 구체적이면서도, 스스로 판단할 여지는 남겨두는 정도의 지침이 딱 "적정 고도"에 해당합니다.

**입력 → 실행 → 출력, 3단계 관리**

| 단계 | 내용 |
|---|---|
| 입력 컨텍스트 설계 | 변경된 파일만 전달, 관련 의존성 포함, 불필요한 정보 제외 |
| 서브에이전트 실행 | 독립적인 20만 토큰 컨텍스트 안에서 작업 |
| 출력 컨텍스트 관리 | 핵심 결과만 반환, 상세 로그 제외, 요약된 분석 결과만 반환 |

**컨텍스트 엔지니어링은 두 가지 차원에서 작동합니다.**
1. **초기 컨텍스트 설계**: 코드 리뷰 에이전트를 호출할 때, 전체 코드베이스를 통째로 주는 게 아니라 **변경된 파일 + 관련 의존성만** 전달하는 게 효과적입니다.
2. **작업 중 컨텍스트 관리**: 에이전트가 파일을 읽고 명령을 실행하면서 컨텍스트가 계속 쌓이므로(누적되므로), 필요한 정보만 선택적으로 수집하도록 지침을 줘야 합니다.

**컨텍스트 격리의 이점 (다시 강조)**
- 서브에이전트의 가장 큰 장점은 격리된 컨텍스트에서 작업한 뒤 **관련 정보만** 오케스트레이터(메인 에이전트)에게 돌려준다는 점입니다.
- 예: 수천 개의 로그 파일을 분석할 때, 서브에이전트가 각 파일을 읽고 분석한 뒤 핵심 발견 사항만 요약해서 반환하면, 메인 대화창은 계속 정리된 정보 상태로 유지됩니다.
- 각 서브에이전트가 독립적으로 **20만 토큰**의 컨텍스트를 가진다는 점도 실질적인 이점입니다. 메인 대화에서는 높은 수준의 계획과 조율만 담당하고, 실제 분석이나 코드 생성 같은 무거운 작업은 서브에이전트에 위임할 수 있기 때문입니다.
- 각 서브에이전트는 자신의 작업에 필요한 파일만 읽고, 작업이 완료되면 핵심 결과만 반환합니다. 이렇게 하면 메인 대화의 컨텍스트는 계속 정제된 정보로 채워지고, 전체 작업 흐름을 효과적으로 관리할 수 있습니다.

```
입력 컨텍스트 설계
  변경된 파일만 전달 · 관련 의존성 포함 · 불필요한 정보 제외
        ↓
서브에이전트 실행 (독립 20만 토큰 컨텍스트)
        ↓
출력 컨텍스트 관리
  핵심 결과만 반환 · 상세 로그 제외 · 요약된 분석 결과
```

## 에이전트 체인과 병렬 실행

**쉬운 설명**: 순차 실행은 `.then().then()`처럼 프로미스를 체인으로 잇는 것과 비슷하게, 앞 단계 결과가 다음 단계 입력이 되는 방식입니다. 병렬 실행은 `Promise.all()`처럼 서로 무관한 작업을 동시에 처리하는 방식이라고 보면 됩니다.

서브에이전트는 순차적으로 체인을 구성하거나, 병렬로 실행할 수 있습니다.

- **순차 실행**: 이전 에이전트의 결과가 다음 에이전트의 입력이 되는 파이프라인 구조입니다. 예를 들어 코드 분석 에이전트가 문제를 식별하면, 그 결과를 바탕으로 수정 에이전트가 코드를 변경하고, 테스트 에이전트가 변경 사항을 검증하는 식입니다.
- **병렬 실행**: 서로 독립적인 작업을 동시에 처리할 때 유용합니다. 예를 들어 여러 모듈의 코드 품질을 동시에 검토하거나, 프런트엔드와 백엔드 코드를 각각 다른 에이전트가 분석하는 경우입니다. 병렬 실행은 전체 작업 시간을 단축시키고, 각 에이전트가 서로의 컨텍스트를 오염시키지 않도록 보장해줍니다.

**에이전트 체인을 설계할 때 주의할 점**
1. 각 단계의 입출력 형식을 명확히 정의해야 합니다. 이전 에이전트의 출력 형식이 다음 에이전트가 기대하는 입력과 일치해야 파이프라인이 원활하게 작동합니다.
2. 중간 단계에서 실패가 발생했을 때의 처리 방법도 미리 고려해야 합니다. 일부 실패는 무시하고 진행해도 되지만, 치명적인 실패라면 전체 파이프라인을 중단하고 오류를 보고해야 합니다.

```
작업 요청 → [병렬 실행: 프런트엔드 분석 / 백엔드 분석 / DB 스키마 분석] → 통합 결과

작업 요청 → [순차 실행(파이프라인): 코드 분석 → 수정 → 테스트]
```

## 다중 클로드 워크플로

**쉬운 설명**: 서브에이전트가 "한 프로그램 안에서 함수를 나눠 쓰는 것"이라면, 다중 클로드 워크플로는 "여러 터미널(프로세스) 자체를 병렬로 띄우는 것"입니다. 실제 개발팀의 협업 방식과 비슷합니다 — 한 명은 기능 구현, 한 명은 코드 리뷰, 한 명은 QA 테스트를 동시에 진행하는 것처럼요.

- **다중 클로드 워크플로(multi-claude workflow)**는 여러 클로드 코드 인스턴스를 병렬로, 서로 다른 터미널에서 실행하여 복잡한 작업을 처리하는 고급 패턴입니다.
- 이 패턴에서는 하나의 인스턴스가 코드를 작성하는 동안 다른 인스턴스가 리뷰를 수행하거나, 독립적인 작업을 위해 별도의 **깃 작업 트리(git worktree)**를 사용합니다.
- 왜 굳이 폴더를 나누냐면, 클로드 코드 여러 개가 같은 프로젝트 폴더를 동시에 수정하면 파일이 충돌하기 때문입니다. `git worktree`는 같은 저장소를 물리적으로 다른 폴더 여러 개에 동시에 펼쳐놓는 기능이라, 폴더별로 독립적인 작업 공간을 만들 수 있습니다.
- 이러한 접근 방식은 대규모 프로젝트에서 개발 속도를 높이고 품질을 개선하는 데 효과적입니다.

다중 클로드 워크플로는 실제 개발팀의 협업 방식과 유사합니다. 한 개발자가 기능을 구현하고, 다른 개발자가 코드 리뷰를 준비하고, QA 엔지니어가 테스트 케이스를 작성하는 것처럼 여러 클로드 코드 인스턴스가 각자의 역할을 수행합니다. 이러한 병렬 작업은 전체 개발 사이클을 단축시키고, 각 인스턴스가 자신의 전문 영역에 집중할 수 있게 해줍니다.

> **Tip**: 이 방식은 서브에이전트를 깃 작업 트리로 물리적으로 분리된 디렉터리에 배치하고, 동일한 코드베이스를 각 에이전트에 복제하여 병렬로 처리하는 구조입니다. 공식 "에이전트 팀" 기능(4-4절 참고)이 도입되기 이전에는, 개발자들이 이 방식을 활용해 직접 다중 에이전트 워크플로를 구현했습니다.

## 언제 서브에이전트를 사용하는가

서브에이전트 사용 여부를 결정할 때는 다음 **3가지 기준**으로 고려합니다.

1. **작업의 복잡도**: 단순한 파일 수정이나 짧은 코드 생성은 메인 대화에서 직접 처리하는 것이 효율적입니다. 반면 여러 파일에 걸친 분석이나 장시간 실행되는 작업은 서브에이전트에 위임하는 것이 좋습니다.
2. **컨텍스트 소모량**: 작업 수행에 대규모 파일 읽기나 명령 실행이 필요하다면 서브에이전트 사용을 고려합니다. 메인 대화의 컨텍스트를 보존하면서 심층적인 분석을 수행할 수 있기 때문입니다.
3. **전문성 요구 수준**: 특정 도메인에 대한 상세한 지침이 필요한 작업은 해당 지침으로 구성된 서브에이전트를 활용하면 더 나은 결과를 얻을 수 있습니다.

**표 4-1. 서브에이전트에 적합한 작업**

| 구분 | 서브에이전트 사용 적합 | 직접 처리 적합 |
|---|---|---|
| 파일 작업 | 대규모 파일 읽기/분석 | 간단한 파일 수정 |
| 전문성 | 특정 도메인 전문 지식 필요 | 일반적인 작업 |
| 실행 시간 | 장시간 실행되는 독립적 작업 | 빠른 질문과 답변 |
| 병렬화 | 병렬 처리 가능한 독립 작업들 | 순차적 단일 작업 |
| 컨텍스트 | 메인 대화 컨텍스트 보존 필요 | 컨텍스트 적게 필요 |
| 코드 생성 | 복잡한 다중 파일 생성 | 짧은 코드 생성 |
| 각 방식의 특징 | 컨텍스트 분리, 전문성 활용, 병렬 실행 | 빠른 응답, 낮은 오버헤드, 단순 처리 |

## 4-2 서브에이전트 사용하기

전체 흐름: 클로드 코드는 기본 제공되는 **내장 서브에이전트**와 직접 만드는 **커스텀 서브에이전트**를 모두 지원하며, `Agent` 도구를 통해 이들을 호출합니다. 이 절은 다음 순서로 진행됩니다: **목록 확인 → 호출 방법 → 결과 합치기 → 작업 이어서 하기 → 에러 대처 → 성능 확인**.

### 사용 가능한 서브에이전트 목록

- `/agents` 명령어를 치면 지금 프로젝트에서 쓸 수 있는 서브에이전트 목록이 전부 나옵니다. (npm 프로젝트에서 `npm list`로 설치된 패키지를 확인하는 것과 비슷한 느낌이에요.)
- 이 목록은 **프로젝트 에이전트**와 **사용자 에이전트**를 구분해서 보여줍니다.

**우선순위 규칙**: 같은 이름의 에이전트가 양쪽에 다 있으면, **프로젝트 에이전트가 우선**입니다. (프로젝트 로컬 설정이 전역 설정보다 우선하는 것과 같은 원리라고 생각하면 쉽습니다.) 이렇게 하면 프로젝트마다 특화된 에이전트를 만들면서도, 평소 쓰던 범용 에이전트도 그대로 유지할 수 있습니다.

| 구분 | 저장 위치 | 사용 범위 | 우선순위 |
|---|---|---|---|
| 프로젝트 에이전트 | `.claude/agents/` | 이 프로젝트에서만 | 높음 |
| 사용자 에이전트 | `~/.claude/agents/` | 모든 프로젝트 | 중간 |
| Built-in Agents (내장) | 기본 제공 | 항상 사용 가능 | 낮음 |

**내장 에이전트(Built-in Agents) 목록** — 별도로 만들지 않아도 기본으로 딸려오는 에이전트들입니다.
- `general-purpose` (sonnet 모델): 이것저것 여러 단계가 필요한 범용 작업 처리
- `statusline-setup` (sonnet 모델): 화면 하단 상태 표시줄 설정
- `Explore` (haiku 모델): 코드베이스를 탐색하고 검색
- `Plan` (inherit): 구현 계획 및 설계 담당. Read/Glob/Grep/Bash 도구만 쓸 수 있고, "플랜 모드"에서 코드 구조를 분석할 때 자동으로 호출됨
- `claude-code-guide` (haiku 모델): 클로드 코드 사용법을 알려주는 가이드 역할

### 에이전트 호출 문법

**비유**: 자동 완성으로 알아서 함수를 골라 쓰는 것(자동 호출)과, 함수 이름을 콕 집어서 부르는 것(명시적 호출)의 차이라고 보면 됩니다.

**① 자동 호출**: 내가 그냥 말로 요청하면, 클로드 코드가 각 에이전트의 `description`(설명) 필드와 내 요청을 비교해서 알아서 딱 맞는 에이전트를 골라줍니다.
- `description`에 `use PROACTIVELY`(적극적으로 사용하라), `MUST BE USED`(반드시 사용해야 함) 같은 문구를 넣어두면, 관련 작업이 감지될 때 더 적극적으로 자동 호출됩니다.

**② 명시적 호출**: 에이전트 이름을 직접 부릅니다.
```
> code-reviewer 서브에이전트를 사용해서 내 최근 변경 사항을 검토해 줘.
> debugger 서브에이전트로 이 에러를 조사해 줘.
> test-writer 서브에이전트에 auth 모듈의 유닛 테스트를 작성하도록 요청해 줘.
```

**③ 컨텍스트와 함께 호출**: `@에이전트이름(agent)` 형태로 파일 경로나 에러 메시지 같은 구체적인 정보를 같이 넘길 수 있습니다. (`@` 기호를 치고 탭 키를 누르면 파일/디렉터리를 쉽게 지정할 수 있어요.)
```
> @code-reviewer(agent) 서브에이전트로 @src/auth/login.ts 파일을 리뷰해 줘.
> @agent-debugger(agent) 서브에이전트로 이 에러를 분석해 줘: TypeError: Cannot read property 'id' of undefined
> @test-writer(agent) 서브에이전트에 @src/utils/parser.ts 함수를 테스트하도록 요청해 줘.
```

### 에이전트 결과 통합

- 서브에이전트가 작업을 끝내면 그 결과는 메인 대화로 돌아옵니다.
- 하나짜리 결과는 그대로 쓰거나 다음 작업 재료로 쓰면 되지만, 여러 에이전트의 결과가 서로 다른 의견을 내는 경우도 있습니다. (예: 코드 리뷰어는 "리팩터링부터 하라"고 하고, 성능 분석 에이전트는 "일단 인라인 처리하라"고 할 수도 있음)
- 이럴 땐 사람이 프로젝트 상황과 우선순위를 고려해 **최종 판단**을 내려야 합니다.

**예시**
```
코드 리뷰어 결과:
- 반복 코드 3곳 발견
- 에러 처리 부재 2곳
- 보안: 입력 검증 누락 1곳

디버거 결과:
- Null 참조 오류 가능성 2곳
- 비동기 처리 경쟁 조건 1곳

통합 우선순위:
1. 보안 입력 검증 (높음)
2. 널 참조 오류 수정 (높음)
3. 에러 처리 추가 (중간)
4. 비동기 경쟁 조건 (중간)
5. 반복 코드 리팩터링 (낮음)
```

### 작업 재개 기능 활용

**왜 필요한가**: 서브에이전트는 호출할 때마다 항상 "새 인스턴스"로 시작합니다. 마치 브라우저 탭을 새로고침하면 이전에 하던 작업이 다 날아가는 것과 비슷해요. 그래서 이전 작업을 이어서 하려면 "재개(resume)" 기능을 명시적으로 써야 합니다. 특히 오래 걸리는 연구나 분석 작업에서 유용합니다.

**작동 원리**
- 서브에이전트가 실행될 때마다 고유한 `agentId`가 부여됩니다.
- 대화 내용은 `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.json` 파일에 저장됩니다.
- 재개를 요청하면 이 파일에서 **이전 대화 내용을 통째로 복원**하고, 새로운 지시사항을 추가해서 이어서 실행합니다.

**작업 재개 워크플로 예시**
1. **초기 호출** (agentId 요청 포함)
   ```
   > @agent-code-analyzer 에이전트로 인증 모듈 리뷰를 시작해 줘. 완료되면 agentId를 알려줘.
   Task Output(non-blocking) a9adef5(agentId)
   ```
2. **중간 확인**
   ```
   > 인증 모듈 분석 결과가 어떻게 되었나요?
   ```
3. **작업 재개** (agentId 사용)
   ```
   > 에이전트 a9adef5를 재개하고 권한 부여 로직도 함께 분석해 줘.
   ```

이 기능을 쓰면 복잡한 분석을 단계별로 나눠서 진행할 수 있습니다. 예를 들어 1단계에서 코드 구조를 파악하고, 2단계에서 문제점을 분석하고, 3단계에서 개선 방향을 도출하는 식으로요. 각 단계 사이에 결과를 확인하고 방향을 조정할 수 있어서 더 정교한 분석이 가능합니다.

### 에러 처리 및 복구

서브에이전트를 실행하다 보면 파일을 못 찾거나, 명령이 실패하거나, 컨텍스트 한도를 초과하는 등의 에러가 생길 수 있습니다.

| 에러 유형 | 원인 | 대처 방법 |
|---|---|---|
| 파일 접근 오류 | 경로가 틀렸거나 권한 부족 | 경로 다시 확인, 존재하는 파일로 재시도 |
| 명령 실행 실패 | 명령어 문법 문제, 환경 설정 문제 | 권한 설정 확인, 필요한 도구 접근 권한 부여 |
| 컨텍스트 초과 | 한 번에 너무 많은 작업을 시킴 | 작업 범위를 줄이거나 `/clear` 후 재시도, 파일을 나눠서 처리 |

**에러 처리 예시**
```
1. 파일 접근 오류
> @agent-code-reviewer로 src/missing.ts 파일을 검토해 줘.
Error: File not found
-> 대응: 파일 경로 확인, 존재하는 파일로 재시도

2. 명령 실행 실패(권한 오류)
> @agent-debugger로 sudo 명령을 실행해 줘.
Error: Permission denied
-> 대응: 권한 설정 확인, 필요한 도구 접근 권한 부여

3. 컨텍스트 초과
Error: Context limit exceeded
-> 대응: 작업 범위 축소, 파일을 나눠서 처리
```

**복구 원칙**
- 그렇게 중요하지 않은 작업의 에러는 그냥 기록만 하고 넘어갈 수 있습니다.
- 핵심 작업에서 에러가 났다면 작업을 멈추고 사용자에게 알려야 합니다.
- 재시도가 가능한 에러는 정해진 횟수까지는 자동으로 다시 시도합니다.

### 서브에이전트 성능 측정

**비유**: 웹페이지 성능을 Lighthouse 점수(로딩 속도, 번들 크기 등)로 재듯이, 서브에이전트도 "시간, 품질, 얼마나 많이 먹었나(토큰량)"로 성능을 측정합니다.

**3가지 성능 지표**
1. **작업 완료 시간**: 작업을 시작해서 결과를 돌려주기까지 걸린 시간
2. **결과 품질**: 결과가 얼마나 정확하고 쓸모 있는지 (발견한 문제가 실제로 맞는지, 제안이 적절한지)
3. **컨텍스트 사용량**: 작업하는 데 소모한 토큰(=AI가 읽고 쓰는 글자 단위) 양. 같은 결과를 더 적은 토큰으로 낸 에이전트가 더 효율적입니다.

**측정 방법**: 그냥 이렇게 요청하면 됩니다 — "완료되면 소요 시간과 사용한 컨텍스트 양을 알려줘"

```
> 전체 코드를 @agent-code-reviewer 에이전트로 분석하고 버그가 있으면
  @agent-debugger가 개선한 뒤, 완료되면 소요 시간과 사용한 컨텍스트 양을 알려줘.

Subagent Performance Dashboard
| Agent         | Avg Time | Quality | Context |
| code-reviewer | 45s      | 4.5/5   | 35K     |
| debugger      | 60s      | 4.2/5   | 42K     |
```

성능을 개선하려면 한 번에 끝내려 하지 말고, 프롬프트를 다듬고 도구 접근 권한을 조정하고 작업 범위를 명확히 하는 걸 **반복적으로** 해야 합니다.

## 4-3 커스텀 서브에이전트 개발과 스킬 주입

**이 절의 목적**: 3장에서 배운 "스킬"(지식 모듈)을 서브에이전트에 넣어주는 구체적인 방법을 다룹니다. 한마디로 **3장의 지식 + 4장의 실행 체계 = 시너지**라는 게 이 절의 핵심입니다. 실무에서는 프로젝트마다 요구 사항이 다르기 때문에, 미리 만들어진 에이전트만으로는 부족하고 나만의 맞춤형 에이전트를 만드는 게 필수적입니다.

### 서브에이전트 정의 파일 구조

- 커스텀 서브에이전트는 **마크다운 파일 하나**로 정의합니다.
- 프로젝트 에이전트 → `.claude/agents/`, 사용자 에이전트 → `~/.claude/agents/`에 저장
- 파일 이름 자체가 그 에이전트의 고유 식별자입니다. 그래서 `code-reviewer.md`, `debugger.md`, `test-writer.md`처럼 이름만 봐도 무슨 역할인지 알 수 있게 짓는 게 좋습니다.
- 파일 구성은 스킬 파일과 비슷합니다. 위쪽 프론트매터(YAML)는 "설정", 아래쪽 본문은 "이 에이전트의 시스템 프롬프트(역할·행동 방식·제약 조건 등을 자세히 적은 지침서)"입니다.

```
.claude/agents/
├── code-reviewer.md    # 코드 리뷰 전문 에이전트
├── debugger.md         # 디버깅 전문 에이전트
├── test-writer.md      # 테스트 작성 에이전트
└── security-auditor.md # 보안 감사 에이전트
```

- 프로젝트 에이전트는 Git 같은 버전 관리 시스템에 포함시켜야 팀원과 동일한 에이전트 설정을 공유하고, 나중에 누가 언제 뭘 바꿨는지 이력도 추적할 수 있습니다.

### YAML 프론트매터 완전 가이드

YAML 프론트매터는 서브에이전트의 "설정값"을 정의하는 부분입니다. 필수로 넣어야 하는 필드와 선택으로 넣는 필드가 있습니다.

```yaml
---
name: code-reviewer
description: 코드 품질 전문가, 보안, 유지보수성 전문가. 코드 변경 직후 사용
tools: Read, Grep, Glob, Bash
model: inherit
skills: moai-foundation-core, moai-lang-unified
---
```

**표 4-2. 서브에이전트의 필드**

| 필드 | 필수 여부 | 설명 | 예시 |
|---|---|---|---|
| name | 필수 | 이 에이전트의 고유한 이름(소문자, 하이픈만 사용) | code-reviewer |
| description | 필수 | 에이전트가 뭘 하는지에 대한 설명 | 코드 품질 전문가 |
| tools | 선택 | 쉼표로 구분한, 사용할 수 있는 도구 목록 | Read, Grep, Glob, Bash |
| model | 선택 | 어떤 AI 모델을 쓸지 (sonnet, opus, haiku, inherit) | inherit |
| skills | 선택 | 이 에이전트에 넣어줄 스킬 목록 | moai-core, moai-lang |
| disallowedTools | 선택 | 못 쓰게 막을 도구 목록 (tools와 같이 못 씀) | Edit, Write |
| permissionMode | 선택 | 권한을 어떻게 처리할지 모드 | acceptEdits |
| hooks | 선택 | 특정 시점에 자동 실행할 명령 | PreToolUse, PostToolUse |
| memory | 선택 | 이전 작업 내용을 기억해둘 범위 | user, project, local |

**각 필드 쉽게 이해하기**
- `name`, `description`은 **반드시** 있어야 합니다. `name`은 소문자와 하이픈만 써서 고유하게 짓고, `description`은 클로드 코드가 "이 에이전트를 언제 자동으로 부를지" 판단하는 기준이 되므로 명확하게 써야 합니다.
- `tools`는 **허용 목록(화이트리스트)** 방식입니다. 즉 여기 적힌 도구만 쓸 수 있습니다. 생략하면 모든 도구를 상속받아 쓸 수 있는데, 필요한 도구만 콕 집어 명시하는 게 보안상 더 안전합니다. 예를 들어 코드 리뷰만 하는 에이전트는 파일을 수정할 필요가 없으니 `Read, Grep, Glob, Bash`만 주고 `Write`는 빼는 식입니다.
- `model`은 사용할 AI 모델을 정합니다. `sonnet`, `opus`, `haiku` 중 하나를 고르거나 `inherit`(부모 세션이 쓰는 모델을 그대로 물려받음)을 씁니다. 복잡한 분석이 필요하면 `opus`, 빠른 응답이 중요하면 `haiku`, 보통은 `sonnet`이나 `inherit`이 적당합니다.

### 에이전트 고급 설정 필드

표 4-2의 아래쪽 4개 필드(`disallowedTools`, `permissionMode`, `hooks`, `memory`)는 에이전트의 동작을 더 세밀하게 조절하는 고급 기능입니다.

#### disallowedTools: 도구 거부 목록

- `tools`가 "이것만 허용"이라는 화이트리스트 방식이라면, `disallowedTools`는 반대로 "이것만 금지"라는 **블랙리스트** 방식입니다.
- 여기 적힌 도구는 못 쓰고, 적히지 않은 나머지 도구는 부모로부터 그대로 상속받아 씁니다.
- `tools`와 `disallowedTools`는 **같이 쓰면 안 됩니다.** 한 에이전트에는 둘 중 하나만 정의합니다.
- 언제 쓰나: "거의 모든 도구를 쓰되, 위험한 도구 한두 개만 막고 싶을 때" 적합합니다.

```yaml
---
name: safe-coder
description: 안전한 코드 작성 전문가. 위험한 도구를 제외하고 모든 도구 사용 가능
disallowedTools: Bash, WebFetch
model: sonnet
---
```

#### permissionMode: 권한 동작 모드

`permissionMode`는 에이전트가 도구를 실행할 때 "허락을 받을지 말지"를 결정하는 필드입니다. 기본값은 `default`이며 5가지 모드가 있습니다.

**표 4-3. permissionMode의 다섯 가지 모드**

| 모드 | 동작 방식 | 어울리는 상황 |
|---|---|---|
| default | 표준적인 권한 확인 (사용자에게 매번 승인 요청) | 범용 에이전트 |
| acceptEdits | 파일 편집 작업은 자동으로 승인 | 신뢰할 수 있는 구현 에이전트 |
| dontAsk | 모든 권한 요청을 자동으로 거부 | 엄격한 샌드박스 에이전트 |
| bypassPermissions | 모든 권한 확인을 아예 건너뜀 | 완전 자동화 파이프라인 |
| plan | 읽기 전용 모드 (쓰기 작업은 무조건 차단) | 연구 및 분석 에이전트 |

**중요 포인트**: `plan` 모드로 설정하면, 설령 `tools` 필드에 `Write`나 `Edit`가 들어 있어도 실제로는 파일을 절대 수정할 수 없습니다. `permissionMode`가 `tools`보다 더 상위에서 작동하기 때문입니다.

**선택 기준 (그림 4-7 흐름)**
1. 파일 수정이 필요한 작업인가? → **아니오**면 `plan` 모드 사용 (읽기 전용 탐색/분석 환경)
2. **예**라면 → 사용자 확인이 매번 필요한가? → **예**면 `default`
3. 확인이 필요 없다면 → 완전 자동화가 목적인가? → 부분 자동화면 `acceptEdits`, 완전 자동화면 `bypassPermissions`

```yaml
---
name: code-analyzer
description: 코드베이스 분석 전문가. 코드를 수정하지 않고 분석만 수행
tools: Read, Grep, Glob
permissionMode: plan
model: haiku
---
```

#### hooks: 생애 주기 훅

- `hooks`는 특정 시점에 셸(터미널) 명령을 자동으로 실행하는 기능입니다.
- 이 훅은 해당 에이전트 안에서만 작동하고, 전역 설정과는 독립적입니다.
- 3가지 시점을 지정할 수 있습니다.
  - `PreToolUse`: 도구를 실행하기 **직전**에 호출 (도구 실행을 검증하거나 사전 작업 수행)
  - `PostToolUse`: 도구 실행이 **완료된 직후**에 호출 (실행 결과를 기록하거나 후처리 작업)
  - `Stop`: 에이전트 실행이 **종료될 때** 호출 (정리 작업이나 결과 보고)
- 실질적 가치: **자동화된 품질 게이트**를 만들 수 있다는 점입니다. 예를 들어 `Write` 도구 호출 전에 린트 검사를 자동으로 돌리거나, 코드 변경 후 자동으로 테스트를 돌리는 파이프라인을 만들 수 있습니다.

```yaml
---
name: audited-coder
description: 모든 작업이 감사 로그에 기록되는 개발 에이전트
tools: Read, Write, Edit, Grep, Glob, Bash
hooks:
  PostToolUse:
    - matcher: Write
      command: "echo '[AUDIT] File written: $CLAUDE_FILE_PATH' >> .audit-log.txt"
  Stop:
    - command: "echo '[AUDIT] Agent session ended' >> .audit-log.txt"
---
```

#### memory: 영속 메모리

- `memory`는 에이전트가 세션이 끝나도 **배운 내용을 계속 기억**하게 해주는 기능입니다.
- 이 필드를 설정하면 "메모리를 읽고 쓰라"는 지침이 시스템 프롬프트에 자동으로 추가되고, `MEMORY.md` 파일의 처음 200줄이 컨텍스트에 자동으로 로드됩니다. 이걸 읽고 쓰기 위한 Read, Write, Edit 도구도 자동으로 활성화됩니다.

**표 4-4. memory 범위**

| 범위 | 저장 경로 | 팀과 공유(VCS)? | 용도 |
|---|---|---|---|
| user | `~/.claude/agent-memory/에이전트 이름/` | 불가 | 프로젝트를 넘나드는 개인 학습 내용 |
| project | `.claude/agent-memory/에이전트 이름/` | 가능 | 팀과 공유하는 프로젝트 지식 |
| local | `.claude/agent-memory-local/에이전트 이름/` | 불가 | 이 프로젝트 한정 개인 메모 |

**선택 가이드**: 배운 내용이 프로젝트와 무관하면 `user` 범위, 프로젝트 특화 내용이면 팀과 공유할지 여부에 따라 `project`(공유) 또는 `local`(비공유)을 고릅니다.

```yaml
---
name: learning-reviewer
description: 리뷰 이력을 기억하는 코드 리뷰 에이전트
tools: Read, Grep, Glob
permissionMode: plan
memory: user
---
코드 리뷰어로서 다음 프로세스를 따릅니다.

작업 시작 전:
- 메모리 파일을 확인하여 이전 리뷰에서 학습한 패턴을 참조합니다.

작업 완료 후:
- 이번 리뷰에서 발견한 새로운 패턴이나 주의 사항을 메모리에 기록합니다.
```

`user`가 가장 권장되는 기본 범위입니다. 프로젝트를 이동해도 에이전트의 학습 내용이 유지되기 때문입니다. `project`는 팀 전체가 공유해야 하는 프로젝트 지식(예: 코드베이스 특유의 패턴, 팀 고유 규칙)을 저장할 때 적합하고, `local`은 개인적인 실험이나 임시 메모용입니다.

**고급 필드 조합 예시** — 4가지 고급 필드를 모두 적용한 보안 감사 에이전트:

```yaml
---
name: security-auditor
description: 보안 취약점 분석 전문가. 코드 보안 검토 필요시 사용
disallowedTools: Write, Edit
permissionMode: plan
memory: project
hooks:
  Stop:
    - command: "echo 'Security audit completed at $(date)' >> .security-audit-log.txt"
---
보안 감사 전문가로서 다음 역할을 수행합니다.

작업 시작 전: 메모리에서 이전 감사에서 발견된 취약점 패턴을 확인합니다.

감사 프로세스:
1. 대상 코드의 보안 취약점 분석
2. OWASP Top 10 기준 검토
3. 민감 정보 노출 확인

작업 완료 후: 새로 발견된 취약점 패턴을 메모리에 기록합니다.
```

이 에이전트는 `disallowedTools`로 파일 수정을 차단하고, `permissionMode: plan`으로 읽기 전용을 이중으로 보장하며, `memory: project`로 감사 이력을 팀과 공유하고, `hooks`의 `Stop` 이벤트로 감사 완료를 자동으로 기록합니다.

### skills 필드 사용법

**쉬운 설명**: 자바스크립트에서 여러 라이브러리를 import 하듯이, 스킬도 여러 개를 나열해서 에이전트에 조합해줄 수 있습니다. 단, **적는 순서가 곧 우선순위**라는 점이 중요합니다.

- `skills` 필드에 지정된 스킬은 에이전트 실행 시 자동으로 로드되어 컨텍스트에 포함됩니다.
- 스킬은 쉼표로 구분해서 나열하며, **지정된 순서대로** 로드됩니다.

**나열 순서의 원칙 2가지**
1. 일반적인(범용) 규칙을 먼저 배치하고, 특수한(도메인) 규칙을 나중에 배치합니다. (기반 스킬의 일반 규칙이 먼저 적용되고, 도메인 스킬의 특수 규칙이 필요한 경우에만 오버라이드되도록)
2. 프로젝트 특화 규칙은 가장 마지막에 배치합니다. (팀이나 프로젝트만의 특수한 규칙이 범용 규칙을 오버라이드하도록)

```yaml
skills: moai-foundation-core, moai-lang-unified, moai-project-specific
# 로딩 순서:
# 1. moai-foundation-core (최고 우선순위) - 기본 규칙
# 2. moai-lang-unified (중간 우선순위) - 언어별 규칙
# 3. moai-project-specific (낮은 우선순위) - 프로젝트 규칙
```

**주의할 점**: 스킬을 너무 많이 조합하면 컨텍스트가 급격히 늘어나므로 토큰 효율을 생각해서 필요한 스킬만 골라야 합니다. 또한 서로 다른 방법론을 지시하는 스킬들을 같이 지정하면 에이전트의 동작이 오히려 일관되지 않게 될 수 있습니다.

```yaml
---
name: tdd-developer
description: TDD 방법론에 따른 개발 전문가. 테스트 주도 개발 필요시 사용
tools: Read, Write, Edit, Bash, Grep, Glob
model: sonnet
skills: moai-workflow-tdd, moai-testing-patterns
---
```

### 스킬 조합 방식

기본적인 스킬 조합 방식은 **기반 스킬(공통 규칙) + 도메인 스킬(특정 분야 지식)**을 함께 쓰는 것입니다. 기반 스킬은 프로젝트의 기본 규칙을 제공하고, 도메인 스킬은 그 위에 특정 분야의 세부 지식을 얹어줍니다.

```yaml
---
name: security-auditor
description: 보안 취약점 분석 전문가. 보안 검토 필요시 사용
tools: Read, Grep, Glob, Bash
model: opus
skills: moai-foundation-core, moai-security-patterns, moai-owasp-top10
---
보안 감사 전문가로서 다음 역할을 수행합니다.

호출 시:
1. 대상 코드의 보안 취약점 분석
2. OWASP Top 10 기준 검토
3. 민감 정보 노출 확인
4. 인증/인가 로직 검증

출력 형식:
- 발견된 취약점 목록(심각도별 분류)
- 각 취약점에 대한 수정 권고
- 참조 가이드라인 명시
```

상위 스킬이 하위 스킬을 참조하도록 구성하면 에이전트는 **계층적으로 구성된 지식**을 활용할 수 있습니다. 즉 "역할 특화 조합" 방식은 에이전트의 특정 역할에 딱 필요한 지식만 골라서 조합하는 것입니다. 코드 리뷰 에이전트라면 코딩 표준, 코드 품질 지표, 리뷰 체크리스트 관련 스킬만 조합하는 식이죠.

### 프롬프트 작성 가이드

프론트매터 다음에 오는 본문이 곧 에이전트의 시스템 프롬프트입니다. 좋은 프롬프트는 다음을 구체적이고 명확하게 정의해야 합니다.
- 에이전트가 무엇을 해야 하는지 (역할)
- 어떻게 접근해야 하는지 (동작 방식)
- 어떤 형식으로 결과를 제공해야 하는지 (출력 형식)

**프롬프트 작성 시 유의할 점 3가지**
1. 너무 긴 프롬프트는 에이전트의 주의를 분산시킵니다.
2. 상충하는 지시는 예측할 수 없는 동작을 유발합니다.
3. 암묵적인 가정(=말하지 않고 당연히 알겠지 하는 것)은 오해를 불러일으킵니다. → **모든 지시는 명시적이고 구체적으로** 적어야 합니다.

```yaml
---
name: code-reviewer
description: 코드 품질, 보안, 유지보수성 전문가. 코드 변경 직후 사용
tools: Read, Grep, Glob, Bash
model: inherit
---
고급 코드 리뷰어로서 다음 프로세스를 따릅니다.

호출 시:
1. git diff 실행하여 변경 사항 파악
2. 수정된 파일에 집중하여 분석
3. 즉시 리뷰 시작

리뷰 체크리스트:
- 단순성 및 가독성
- 명확한 네이밍
- 중복 코드 부재
- 적절한 에러 처리
- 노출된 시크릿 없음
- 성능 이슈 여부
- 테스트 커버리지

출력 형식:
## 리뷰 결과
### 발견된 이슈
- [심각도] 파일:라인 - 설명
### 수정 권고
- 구체적인 개선 제안
### 전반적 평가
- 변경 사항에 대한 종합 의견
```

**출력 포맷 지정**: 에이전트의 출력 형식을 명확히 지정하면 결과의 일관성과 활용도가 높아집니다. JSON, 마크다운, 구조화된 텍스트 등 상황에 맞는 형식을 지정할 수 있습니다. 이후 다른 프로그램이 이 결과를 자동으로 처리해야 한다면 JSON 형식을, 사람이 읽어야 한다면 마크다운을 지정하는 식입니다.

```yaml
---
name: api-analyzer
description: API 엔드포인트 분석기. API 문서화 필요시 사용
tools: Read, Grep, Glob
model: sonnet
---
API 엔드포인트 분석기입니다.

**출력 형식: JSON**

​```json
{
  "endpoints": [
    {
      "method": "string(GET|POST|PUT|DELETE)",
      "path": "string",
      "parameters": ["string"],
      "response": {},
      "auth_required": false
    }
  ],
  "summary": {
    "total_endpoints": 0,
    "auth_required_count": 0,
    "methods_distribution": {}
  }
}
​```

**출력 예시:**

​```json
{
  "endpoints": [
    {
      "method": "POST",
      "path": "/api/auth/login",
      "parameters": ["email", "password"],
      "response": {"token": "string", "user_id": "integer"},
      "auth_required": false
    }
  ],
  "summary": {
    "total_endpoints": 1,
    "auth_required_count": 0,
    "methods_distribution": {"POST": 1}
  }
}
​```
```

**경계 조건 명시**: 에이전트가 "이건 처리하지 않는다"는 상황을 명확하게 정해두면, 에이전트가 자기 역할 범위를 넘어서는 행동을 하는 것을 막을 수 있습니다. 이때 "무엇을 해야 하는지(수행하는 작업)"뿐 아니라 "무엇을 하지 말아야 하는지(수행하지 않는 작업)"도 함께 정하고, 범위 밖 요청이 왔을 때 어떻게 응답할지도 정해두면 사용자 경험이 좋아집니다.

```yaml
---
name: code-reviewer
description: 코드 리뷰 전문가
tools: Read, Grep, Glob
---
코드 리뷰어입니다.

수행하는 작업:
- 코드 품질 분석
- 보안 취약점 식별
- 코딩 표준 준수 확인
- 개선 제안 제공

수행하지 않는 작업:
- 코드 직접 수정(제안만 함)
- 새로운 기능 구현
- 테스트 코드 작성
- 문서 작성

범위 외 요청 처리:
"이 작업은 코드 리뷰 범위를 벗어납니다. [적절한 에이전트]를 사용하세요."
```

### 에이전트 테스트와 최적화

커스텀 서브에이전트를 만든 뒤에는 반드시 테스트와 최적화 과정을 거쳐야 합니다. 다양한 시나리오에서 실행해보면서 동작을 검증하고 결과 품질을 평가합니다.

**테스트할 시나리오 3가지**
- **정상 케이스**: 에이전트가 설계된 목적대로 잘 동작하는지 확인
- **경계 케이스**: 입력이 비어있거나, 아주 크거나, 예상치 못한 형식일 때 어떻게 반응하는지 확인
- **실패 케이스**: 에이전트가 처리할 수 없는 상황에서 적절하게 대응하는지 확인

```
[정상 케이스]
> @agent-code-reviewer로 src/auth/login.ts 파일을 리뷰해 줘.
기대 결과: 코드 품질 이슈와 개선 제안 제공

[경계 케이스]
> @agent-code-reviewer로 리뷰해 줘. (파일 경로 없음)
기대 결과: 파일 경로 요청

> @agent-code-reviewer로 src/huge-file.ts 파일을 리뷰해 줘. (10,000줄)
기대 결과: 파일을 섹션별로 분석하거나 경고

[실패 케이스]
> @agent-code-reviewer로 이 버그를 수정해 줘.
기대 결과: 범위 외 작업 안내
```

**최적화 방법**: 프롬프트 다듬기, 도구 권한 조정, 스킬 조합 변경 등을 통해 이루어집니다. 에이전트의 응답 시간, 결과 품질, 컨텍스트 사용량을 측정하며 개선점을 찾습니다. 이 과정을 반복적으로 테스트하고 조정하면서 최적의 에이전트 설정을 찾아가는 것입니다.

**모델 변경도 최적화 수단 중 하나입니다.** 동일한 에이전트를 sonnet, opus, haiku 세 모델로 각각 테스트해보면 최적의 모델을 고를 수 있습니다. opus는 가장 강력하지만 비용이 높고, haiku는 빠르고 저렴하지만 복잡한 작업에서는 품질이 떨어질 수 있습니다.

**모델 선택 기준은 작업의 복잡도와 품질 요구 수준입니다.**
- 단순한 패턴 매칭이나 포맷 변환 → haiku로 충분
- 복잡한 코드 분석이나 심층적인 추론이 필요한 작업 → sonnet이나 opus 적합
- 비용과 속도가 중요한 대량 처리 작업 → sonnet이나 haiku
- 품질이 중요한 일회성 분석 → opus

```yaml
# 테스트용 모델 설정 변경
model: sonnet  # 기본 테스트
model: opus    # 복잡한 작업 테스트
model: haiku   # 속도/비용 테스트
```

## 4-4 에이전트 팀

**쉬운 설명**: 여태까지 배운 서브에이전트가 "요청하면 한 번 응답하고 끝나는" 일회성 관계였다면, **에이전트 팀(Agent Teams)**은 여러 에이전트가 하나의 팀을 이뤄 **계속 함께 일하고 서로 대화하는** 협업 모델입니다. 클로드 코드 v2.1.32에서 실험적으로 도입된 기능이며, 여러 클로드 코드 인스턴스가 함께 작업하도록 조율합니다.

### 에이전트 팀의 핵심 개념

에이전트 팀의 아키텍처는 실제 소프트웨어 개발팀의 구조를 본떠 만들어졌습니다.

- **팀 리더**: 전체 작업을 조망하고 업무를 분배하는 역할
- **팀원**: 각자 자신의 전문 영역에서 독립적으로 작업하며 필요할 때 다른 팀원과 직접 소통

이 구조에서 가장 중요한 특징은 팀원이 **영속적(persistent)**이라는 점입니다. 서브에이전트는 작업이 끝나면 컨텍스트와 함께 소멸하지만, 에이전트 팀의 팀원은 팀이 해산될 때까지 자신의 컨텍스트를 유지하며 여러 태스크를 연속으로 수행할 수 있습니다.

**에이전트 팀을 활성화하려면 다음 세 가지 조건이 모두 충족되어야 합니다.**
1. 클로드 코드 버전이 v2.1.32 이상이어야 합니다.
2. 환경 변수 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`이 설정되어야 합니다. (settings.json의 `env` 필드에서 구성 가능)
3. 사용자가 팀으로 처리를 요청하거나, 클로드가 팀을 제안해서 사용자로부터 확인을 받은 후 진행해야 합니다. **에이전트 팀 제어 권한은 사용자에게 있습니다.** 클로드는 승인 없이 팀을 만들지 않습니다.

이 세 조건이 모두 충족되면 에이전트 팀 API를 사용할 수 있으며, `message`, `broadcast`, `TeammateIdle`, `TaskCompleted` 등의 팀 전용 도구에 접근할 수 있습니다.

> **참고**: 에이전트 팀 활성화 후, `/config` 메뉴의 **Teammate mode** 설정(`auto` / `in-process` / `tmux`)으로 팀원이 터미널에 표시되는 방식을 선택할 수 있습니다. 기본값은 `auto`이며, tmux나 iTerm2 환경이 감지되면 자동으로 분할 창 모드로 작동합니다.

### 팀 생성과 생애 주기 관리

에이전트 팀은 실험적 기능으로 **기본적으로 비활성화**되어 있습니다. 사용하려면 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경 변수를 1로 설정해야 합니다. 셸 환경 또는 settings.json의 env 섹션에서 설정할 수 있습니다.

에이전트 팀은 여러 클로드 코드 인스턴스가 함께 작업하도록 조율하는 기능입니다. 하나의 세션이 팀 리더 역할을 맡아 작업을 조율하고, 태스크를 할당하고, 결과를 종합합니다. 팀원은 각각 독립된 컨텍스트 윈도우에서 자율적으로 작업하며 서로 직접 소통할 수 있습니다. 서브에이전트와 달리 **사용자가 리더를 거치지 않고 개별 팀원과 직접 상호작용할 수도 있습니다.**

**에이전트 팀이 시작되는 경로는 두 가지입니다.**
1. 사용자가 병렬 작업에 적합한 태스크를 주면서 에이전트 팀 생성을 명시적으로 요청하는 방법
2. 클로드가 사용자의 태스크가 병렬 작업에 유리하다고 판단하면 팀 생성을 제안하는 방법

**두 경우 모두 사용자의 승인 없이는 팀이 생성되지 않습니다.** 팀이 활성화된 상태에서 사용자가 원하는 작업과 팀 구조를 설명하면 클로드는 팀을 만듭니다. 그다음 공유 태스크 목록을 생성하고, 각 관점에 맞는 팀원을 생성하며, 팀원들이 문제를 탐색하도록 하고, 결과를 종합한 뒤 팀을 정리합니다.

**에이전트 팀의 아키텍처는 네 가지 구성 요소로 이루어집니다.**
1. **팀 리더**: 팀을 생성하고 팀원을 생성하며 작업을 조율하는 메인 클로드 코드 세션(메인 에이전트)
2. **팀원**: 할당된 태스크를 수행하는 별도의 클로드 코드 인스턴스
3. **태스크 목록**: 팀원들이 선택하고 완료하는 공유 작업 항목 목록
4. **메일박스**: 에이전트 간 통신을 위한 메시지 시스템

팀과 태스크는 로컬에 저장됩니다. 팀 설정은 `~/.claude/teams/{team-name}/config.json`에, 태스크 목록은 `~/.claude/tasks/{team-name}/`에 위치합니다. 팀 설정 파일에는 각 팀원의 이름, 에이전트 ID, 에이전트 타입이 포함된 `members` 배열이 있으며, 팀원은 이 파일을 읽어 다른 팀원을 식별할 수 있습니다.

- 팀원의 수는 클로드가 태스크에 따라 결정하거나, 사용자가 정확히 지정할 수도 있습니다. 팀원의 모델도 지정할 수 있습니다.
- 팀원의 종료는 리더가 종료 요청(shutdown request)을 보내는 것으로 시작됩니다. 팀원은 이를 승인하여 정상 종료하거나 사유를 붙여 거부할 수 있습니다. 팀원은 현재 진행 중인 요청이 도구 호출을 마친 후에야 종료되므로 종료에 시간이 걸릴 수 있습니다.

### 팀 정리(cleanup)

- 팀 정리는 리더가 요청하면 공유 팀 리소스가 제거되는 것을 말합니다.
- 리더가 정리를 실행할 때 아직 활성 상태인 팀원이 있으면 실패합니다. 그래서 반드시 모든 팀원을 먼저 종료한 후 정리가 수행되어야 합니다.
- 팀 정리는 반드시 리더가 수행해야 합니다. 팀원이 직접 정리를 실행하면 팀 컨텍스트가 올바르게 해석되지 않아 리소스가 불완전한 상태로 남을 수 있습니다.

### 컨텍스트와 권한

- 각 팀원은 독립된 자기만의 컨텍스트 윈도우를 가집니다.
- 팀원 생성 시 일반 세션과 동일한 프로젝트 컨텍스트(CLAUDE.md, MCP 서버, 스킬 포함)를 로드합니다.
- 또한 리더가 보낸 생성 프롬프트(spawn prompt)를 받습니다. 단, **리더의 대화 이력은 팀원에게 전달되지 않습니다.**
- 권한 설정은 팀원이 리더의 것을 그대로 상속받습니다. 예를 들어 리더가 `--dangerously-skip-permissions`로 실행 중이면 모든 팀원도 동일하게 적용됩니다. 생성 이후 개별 팀원의 권한 모드는 바꿀 수 있지만, 생성 시점에 팀원 스스로 다른 권한을 설정할 수는 없습니다.

### 메시지 전달 시스템

- 팀원이 메시지를 보내면 수신자에게 자동으로 배달됩니다. 리더가 수동으로 업데이트를 확인할 필요가 없습니다.
- 팀원이 작업을 마치고 멈추면(idle) 리더에게 자동으로 알림이 전송됩니다.
- 모든 에이전트는 태스크 상태를 확인하고 가용한 작업을 선택할 수 있습니다.

**팀원 간 메시지 두 가지 방식**
- `message`: 특정 팀원 한 명에게만 메시지를 보내는 방식
- `broadcast`: 모든 팀원에게 동시에 메시지를 보내는 방식 (비용이 팀 규모에 비례해 증가하므로 신중하게 사용해야 함)

**복잡하거나 위험한 태스크의 경우**, 팀원이 구현하기 전에 먼저 계획을 세우도록 요구할 수 있습니다.
1. 팀원은 리더가 접근 방식을 승인할 때까지 읽기 전용 플랜 모드에서 작업합니다.
2. 팀원이 계획 수립을 마치면 리더에게 **계획 승인 요청**을 보냅니다.
3. 리더는 계획을 검토하고 승인하거나 피드백과 함께 거부합니다.
4. 거부되면 팀원은 플랜 모드에 머물며 피드백을 반영하여 수정한 후 다시 제출합니다. 승인되면 팀원은 플랜 모드를 빠져나와 구현을 시작합니다.
5. 리더의 판단에 영향을 주려면 프롬프트에 기준을 미리 제시하면 됩니다. 예를 들어 "테스트 커버리지를 포함한 계획만 승인하라"거나 "데이터베이스 스키마를 수정하는 계획은 거부하라"고 지시할 수 있습니다.

**사용자가 팀원과 직접 소통하는 방법**
- In-process 모드에서는 `Shift+Down`으로 팀원을 순환하며 메시지를 보낼 수 있고, `Enter`로 팀원의 세션을 확인하고, `Escape`로 현재 턴을 중단할 수 있습니다.
- Split-pane 모드에서는 팀원의 화면 패널을 클릭하여 해당 세션과 직접 상호작용할 수 있습니다.

### 태스크 조정 시스템

- 공유 태스크 목록이 팀 전체의 작업을 조율합니다. 리더가 태스크를 생성하고 팀원들이 이를 처리합니다.
- 태스크에는 `pending`(대기 중), `in progress`(진행 중), `completed`(완료)의 세 가지 상태가 있습니다.
- 태스크는 다른 태스크에 의존할 수 있는데, 미해결 의존성이 있는 `pending` 태스크는 해당 의존성이 완료될 때까지 선택할 수 없습니다.

**태스크 할당 두 가지 방법**
1. **리더가 명시적으로 할당하는 방식** — 리더가 어떤 태스크를 어떤 팀원에게 줄 것인지 직접 지시
2. **자기 선택(self-claim) 방식** — 팀원이 자신의 태스크를 완료한 후 할당되지 않은 태스크를 스스로 가져감

**태스크 선택 시에는 파일 잠금이 적용**되어, 여러 팀원이 동시에 같은 태스크를 선택하려 할 때 발생하는 경쟁 조건(race condition)을 방지합니다. 시스템은 태스크 의존성을 자동으로 관리하며, 팀원이 다른 태스크가 의존하는 태스크를 완료하면 차단된 태스크는 수동 개입 없이 자동으로 해제됩니다.

**태스크 크기도 중요합니다.**
- 태스크가 너무 작으면 조율 비용이 이점을 초과합니다.
- 태스크가 너무 크면 팀원이 오랫동안 점검 없이 작업하게 되어 낭비 위험이 커집니다.
- 적절한 크기는 테스트 파일, 리뷰 하나와 같이 명확한 산출물을 생성하는 자기 완결적 단위입니다.
- 리더가 충분한 수의 태스크를 생성하지 않으면 작업을 더 작은 단위로 나누라고 요청할 수 있습니다.
- 팀 하나당 5~6개의 태스크를 유지하면 생산성이 유지되고, 누군가 막혔을 때 리더가 작업을 재할당할 수 있습니다.

### 파일 소유권 전략

에이전트 팀에서 중요한 안전장치 중 하나는 **파일 소유권 전략**입니다. 여러 팀원이 동시에 같은 파일을 수정하면 충돌이 발생할 수 있으므로, 각 팀원에게 배타적인 파일 소유권을 할당하여 이를 방지합니다.

- 보통 웹 프로젝트에서는 백엔드 개발자는 `src/api/`, `src/models/`, `src/services/` 디렉터리를, 프런트엔드 개발자는 `src/ui/`, `src/components/`, `src/pages/` 디렉터리를 소유합니다. 테스터는 `tests/`, `__tests__/`, 그리고 `*.test.*`, `*.spec.*` 패턴의 파일을 전담합니다.
- 모노레포 구조에서는 각 도메인 담당 팀원이 `packages/<domain>/` 또는 `apps/<domain>/` 경로를 배타적으로 소유합니다.
- **파일 소유권의 핵심 규칙은 두 팀원이 동일한 파일을 소유하지 않는다는 것입니다.**
- QA 팀원은 특수한 경우로 파일 소유권이 없습니다. 이 팀원은 읽기 전용(`permissionMode: plan`) 모드로 작동하여 코드를 직접 수정하지 않고 품질 기준 준수 여부만 검증합니다.

### 서브에이전트와 에이전트 팀

서브에이전트는 요청-응답에, 에이전트 팀은 협업에 목적이 있습니다.

**표 4-5. 서브에이전트와 에이전트 팀의 차이**

| 특성 | 서브에이전트 | 에이전트 팀 |
|---|---|---|
| 실행 모델 | 요청-응답(일회성) | 협업(영속적) |
| 통신 방식 | 오케스트레이터 경유만 가능 | 팀원 간 직접 통신 가능 |
| 컨텍스트 수명 | 단일 작업 완료 시 종료 | 팀 해산 시까지 유지 |
| 태스크 관리 | 오케스트레이터가 명시적으로 위임 | 공유 목록에서 자율 선택 |
| 파일 충돌 방지 | 순차 실행으로 회피 | 소유권 전략으로 관리 |
| 오류 복구 | 작업 재시작 | 대체 팀원 생성, 태스크 재배정 |
| 안정성 | 정식 기능 | 실험적 기능(v2.1.32+) |
| 적합한 규모 | 1~3개 독립 작업 | 3개 이상 도메인, 10개 이상 파일 |

**작업의 특성에 따라 둘 중 하나를 선택하면 되는데, 다음 세 가지를 기준으로 삼습니다.**

1. **도메인 교차 정도**: 작업이 백엔드, 프런트엔드, 데이터베이스 등 3개 이상의 도메인에 걸쳐 있다면 에이전트 팀이 효과적입니다.
2. **영향 파일 수**: 수정 대상 파일이 10개 이상이면 파일 소유권 관리가 필요하므로 에이전트 팀을 고려합니다.
3. **복잡도 점수**: 자동 판단 시스템에서 복잡도 점수가 7점 이상이면 에이전트 팀을 권장합니다.

클로드 코드는 에이전트 팀을 사용할 수 없거나 팀 생성 과정에서 오류가 발생하면 자동으로 서브에이전트 모드로 폴백하도록 설계되어 있습니다. 이 폴백 메커니즘은 마지막으로 완료된 태스크부터 작업을 이어가므로 데이터 손실이 없습니다.

현재는 에이전트 팀이 실험 기능이므로 향후 API 변경이나 동작 방식의 수정이 있을 수 있습니다. 그러나 여러 에이전트가 하나의 프로젝트에서 실시간으로 협업하는 패러다임은 에이전틱 코딩의 자연스러운 진화 방향입니다.

## 4-5 CLI 기반 동적 에이전트 활용

앞서 다룬 커스텀 서브에이전트나 에이전트 팀은 모두 사전에 에이전트를 정의하고 반복적으로 활용하는 방식이었습니다. 이번에는 **일회성 작업**을 위해 에이전트를 즉석에서 구성하는 방식을 살펴봅니다.

클로드 코드의 CLI에는 두 가지 플래그가 있습니다.
- `--agent` 플래그: 기존에 정의된 에이전트를 CLI에서 직접 지정하여 실행하는 기능
- `--agents` 플래그: JSON 형식으로 에이전트를 즉석에서 정의하여 생성하는 기능

### --agent 플래그: 기존 에이전트 직접 실행

**쉬운 설명**: 평소에는 클로드 코드가 사용자의 자연어 요청을 분석해서 적절한 에이전트를 자동으로 선택하지만, `--agent` 플래그를 쓰면 이 자동 선택 과정을 건너뛰고 지정된 에이전트를 즉시 활성화합니다.

```bash
$ claude -p "최근 변경 사항을 리뷰해 줘." --agent code-reviewer
```

이 방식은 **자동화 시나리오에 특히 적합**합니다. `--agent` 플래그를 `-p` 플래그(비대화형 실행)와 결합하면 특정 에이전트를 비대화형으로 실행할 수 있어 CI/CD 파이프라인 통합에 좋습니다.

```bash
# CI 파이프라인에서 보안 감사 에이전트 실행
$ claude -p "이 PR의 보안 취약점을 분석해 줘." \
  --agent security-auditor --output-format json

# 배포 전 성능 검증 에이전트 실행
$ claude -p "프로덕션 배포 전 성능 영향을 평가해 줘." \
  --agent perf-checker --max-turns 10
```

**서브에이전트와의 중요한 차이점**: `--agent` 플래그로 실행되는 에이전트는 **'메인 대화(세션)'로 작동**한다는 점에서 서브에이전트와 구분됩니다.
- 서브에이전트는 오케스트레이터의 컨텍스트 내에서 `Agent` 도구를 통해 호출되며, 다른 서브에이전트를 생성할 수 없습니다.
- 반면 `--agent` 플래그로 실행되는 에이전트는 독립적인 최상위 에이전트로 동작합니다. 자신의 `tools` 필드에 `Agent`(worker, researcher)를 포함시켜 지정된 서브에이전트를 생성할 수도 있습니다.

### --agents 플래그: 인라인 에이전트 정의

**쉬운 설명**: `--agents` 플래그는 JSON 객체를 받아 에이전트를 즉석에서 정의합니다. 파일 시스템에 별도로 마크다운 파일을 만들지 않고도 에이전트를 구성할 수 있어, 스크립트에서 일회성으로 쓰거나 실험적 테스트에 적합합니다.

```bash
$ claude --agents '{
  "quick-reviewer": {
    "description": "간단한 코드 리뷰 전문가",
    "prompt": "변경된 코드의 품질과 보안을 간략히 검토합니다. 핵심 이슈만 보고합니다.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  }
}'
```

JSON 객체의 각 필드는 마크다운 파일의 프론트매터 필드에 대응합니다. 단, 마크다운 파일 기반 정의와 달리 **`skills` 필드는 인라인 정의에서 사용할 수 없습니다.** 따라서 필요한 지식은 `prompt` 필드에 직접 포함해야 합니다.

**인라인 정의의 우선순위 한계도 알아두어야 합니다.** `--agents` 플래그로 정의된 에이전트는 우선순위가 가장 낮습니다. 동일한 이름의 에이전트가 프로젝트나 사용자 수준에 이미 있으면, CLI 인라인 정의는 무시됩니다.
- 우선순위 순서: **프로젝트 수준(최고) > 사용자 수준(중간) > CLI 인라인(최저)**

### 에이전트 해석 과정

클로드 코드가 에이전트를 탐색하고 선택하는 과정은 **계층적 검색**을 따릅니다. 사용자가 에이전트 이름을 지정하거나 자동 위임이 발생하면, 시스템은 다음 순서로 에이전트를 검색합니다.

1. 먼저 현재 프로젝트의 `.claude/agents/` 디렉터리에서 일치하는 에이전트를 찾습니다.
2. 프로젝트 수준에서 발견되지 않으면 사용자의 홈 디렉터리(`~/.claude/agents/`)를 검색합니다.
3. 여기서도 발견되지 않으면 `--agents` 플래그로 인라인 정의된 에이전트를 검색합니다.
4. 마지막으로 내장 에이전트(Plan, Explore 등)와 대조합니다.

### CLI 에이전트와 에이전트 팀의 관계

- `--agent` 플래그로 실행되는 에이전트는 **에이전트 팀의 팀 리더 역할**을 수행할 수 있습니다.
- 예를 들어 프로젝트에 `team-coordinator`라는 에이전트를 정의해두었다면, 이 에이전트를 `claude --agent team-coordinator`로 실행하면 이 에이전트는 메인 스레드로서 `TeamCreate`를 호출해 팀원을 생성하며 전체 워크플로를 조율할 수 있습니다.

**CLI 기반 동적 에이전트 활용은 마크다운 파일 기반의 정적 에이전트 정의와 상호 보완적인 관계에 있습니다.**
- 반복적으로 사용하는 에이전트는 마크다운 파일로 정의하여 버전 관리
- 일회성 또는 실험적 에이전트는 CLI 플래그로 즉석에서 구성
- 자동화 파이프라인에서는 `--agent` 플래그로 미리 정의된 에이전트를 직접 지정하여 실행의 예측 가능성을 확보

## 4장 마무리

4장에서는 다음 내용을 다뤘습니다.

1. **오케스트레이터-워커 패턴**을 기반으로 복잡한 작업을 전문화된 인스턴스에 위임하는 서브에이전트의 호출 문법과 컨텍스트 격리의 이점을 파악했습니다.
2. 커스텀 서브에이전트를 직접 정의하여 스킬을 주입하고 도구 권한을 제어하는 방법을 배웠습니다.
3. 특히 여러 에이전트가 파일 소유권을 나누어 병렬로 협업하고 실시간으로 메시지를 주고받는 **에이전트 팀**의 혁신적 협업 패러다임과 활용 기준을 깊이 있게 다뤘습니다.

5장에서는 클로드 코드가 응답하는 방식을 결정하는 **출력 인터페이스**를 살펴봅니다. 출력 스타일 커스터마이징을 통해 응답의 톤과 형식을 제어하고, 상태 표시줄을 설정하여 세션 정보를 시각화하며, 대화형 모드의 고급 단축키를 활용해 작업 생산성을 높이는 방법을 배웁니다.
