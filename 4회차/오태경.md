# 서브 에이전트
## 1. 서브 에이전트의 개념과 아키텍쳐

### 서브 에이전트란?
스킬이 "무엇을 알아야하는지"를 정의했다면 <br>
서브 에이전트는 "그 지식을 어떻게 활용하여 실제 작업을 수행하는지"를 정의한다. <br>

클로드 코드에는 내장 도구에 `Agent` 라는 도구가 있고 이 도구는 **서브 에이전트를 생성**한다. <br>
각 서브 에이전트는 **자체 컨텍스트 윈도우, 커스텀 시스템 프롬프트, 할당된 도구, 독립적인 권한**을 가지고 있다. <br>

서브 에이전트는 .md 파일로 정의하며 프론트매터, 마크다운 본문으로 구성한다. 이때 마크다운 본문이 시스템 프롬프트가 된다.

<br>

### 서브 에이전트가 필요한 이유
**<u>1.</u>** 서브 에이전트의 핵심은 `컨텍스트 격리`이다. <br>
메인 대화에서 특정 작업을 별도의 서브 에이전트에게 위임하면 독립된 컨텍스트 윈도우에서 작업을 처리하고 결과만 돌려주는 방식 <br>
핵심 결과만 메인 대화에 반환되어 메인 컨텍스트는 정제된 정보만 가진다.


**<u>2.</u>** 컨텍스트 보존 말고도 서브 에이전트마다 쓸 수 있는 도구를 제한할 수 있는 이점이 있다. <br>
읽기 전용 탐색 에이전트에게는 파일 수정 권한을 안 주는 식으로 안전하게 운용할 수 있다.

**<u>3.</u>** 서브 에이전트별로 다른 모델을 지정이 가능하다. 이는 파일 탐색은 빠르고 저렴한 Haiku에게 맡기고, 복잡한 코드 리뷰는 Opus로 설정한다.

<br>

### 에이전트 위임 패턴 : 오케트스레이터-워커
서브에이전트는 오케스트레이터-워커 패턴을 따른다. <br>
- 메인 에이전트 : 오케스트레이터 역할, 서브 에이전트의 작업을 관리하고 결과를 통합한다. 스타로 치면 캐리어
- 서브 에이전트 : 실제 작업을 수행하는 워커, 스타로 치면 인터셉터

<br>

### SRP, 단일책임 원칙
서브에이전트는 하나의 명확한 책임만을 가지도록 한다.

<br>

### 컨텍스트 엔지니어링 원칙
컨텍스트 엔지니어링은 컨텍스트 윈도우를 딱 맞는 정보로 채우는 섬세한 기술이자 과학" — Andrej Karpathy, 2025 <br>

즉, **의도/정보의 전달 체계를 만드는 것**이 중요!!

에이전트에게 방향을 줄 만큼 구체적이되, 판단을 발휘할 수 있는 여지를 남겨두는 것이 핵심 <br>

서브 에이전트에서 컨텍스트 엔지니어링은 두가지 관점에서 접근할 수 있다.
1. 초기 컨텍스트 설계
   - 코드 리뷰 에이전트라고 했을 때, 전체 코드가 아닌 변경된 파일과 관련 의존성만 전달하는 것이 효과적이다.
2. 작업중 컨텍스트의 관리
   - 컨텍스트는 무한 저장소가 아니라 쓸수록 썩는 유한 자원
   - 컨텍스트가 누적되므로 필요한 정보만 선택적으로 수집하도록 지침이 필요

<br>

### 컨텍스트 격리의 이점
각 서브 에이전트는 독립적인 20만 토큰 컨텍스트를 가진다. <br>
이를 활용하여 격리된 컨텍스트에서 작업 후 핵심 결과만 반환한다면 메인 대화의 컨텍스트는 정제된 정보만 가지게 된다. <br>
반대로 서브에이전트에는 메인 세션의 대화 이력, 메인 세션의 Auto Memory 등은 전달되지 않는다.

<br>

### 에이전트 체인과 병렬 실행
독립적인 작업을 동시에 처리해야하는 경우 병렬 실행도 가능하다. <br>

여러 서브 에이전트를 순서대로 연결하는 체인을 구성할 수 있다. <br>
각 서브 에이전트의 입출력을 명확하게 정의해야한다.
- 에이전트의 결과가 다음 에이전트의 입력이 되는 파이프라인 구조이다.
- 코드 분석 에이전트가 분석한 결과를 optimizer 에이전트가 코드를 구현하고 테스트 에이전트가 구현된 코드를 테스트하는 식으로 연결 가능

<br>

### 다중 클로드 워크플로
여러 클로드 인스턴스를 병렬로 서로 다른 터미널에서 실행하여 워크플로를 구성할 수 있다. <br>
이 방식은 서브 에이전트를 깃 작업 트리로 물리적으로 분리하고 동일한 코드베이스를 복사하여 각 에이전트가 병렬로 처리하는 구조

<br>

### 언제 서브에이전트를 사용하는가?
1. 복잡한 분석이나 장시간 실행되는 작업은 서브 에이전트에 위임하는것이 좋다. 단순하고 짧은건 메인 대화에서 처리해도 된다.
2. 컨텍스트 소모량이 많아질 것 같은 케이스
3. 특정 도메인이나 상세한 지침이 필요한 작업

## 2. 서브 에이전트 사용하기
클로드 코드에는 상황에 따라 자동으로 쓰이는 내장 서브 에이전트가 있고 직접 정의하는 커스텀 서브에이전트를 지원한다. <br>
위에서 설명한 내장 도구에 `Agent` 라는 도구를 통해 서브 에이전트를 호출한다. <br>

활성화된 모든 서브 에이전트 목록은 `/agents` 으로 확인 가능 <br>
/agents 명령어 결과 목록 마지막에 Built-in Agents는 기본 내장 에이전트 목록이다. <br>
```
Explore · haiku           # 코드베이스 탐색 및 검색
general-purpose · inherit # 탐색과 코드 수정을 모두 할 수 있는범용
Plan · inherit            # 구현 계획 및 설계, 메인 대화의 모델을 그대로 사용
```

<br>

### 서브 에이전트 호출하기
**<u>1.</u>** **자동 위임** <br>
클로드 코드가 프론트매터의 description을 읽고 적절한 서브 에이전트를 자동 선택 <br>
description에 `use PROACTIVELY` or `MUST BE USED` 키워드를 포함시킨다. <br>

**<u>2.</u>** **자연어 명령으로 호출** <br>
프롬프트에 서브에이전트 이름을 언급하여 명시적으로 호출 <br>
**"code-reviewer 서브 에이전트로 내 최근 변경 사항들을 리뷰해줘"** <br>
명시적으로 에이전트 호출시 추가 컨텍스트를 같이 전달할 수 있다. <br>
> @<에이전트명>(agent) <br>
> @code-review(agent) 서브 에이전트를 사용하여 @src/auth/Login.java 파일을 리뷰해줘

/code-reviewer처럼 슬래시 커맨드로 서브 에이전트 호출은 불가능하다.

**<u>3.</u>** **세션 전체 적용** <br>
> claude --agent <이름> <br>
> claude --agent code-reviewer

으로 클로드를 시작하면 메인 세션 자체가 해당 서브에이전트의 시스템 프롬프트, 도구 제한, 모델을 그대로 사용\
<br>

### 서브 에이전트 우선순위
우선순위 : **프로젝트 서브 에이전트 > 사용자 에이전트** 즉 이름이 같다면 프로젝트 우선

### 서브 에이전트 결과 통합
클로드는 여러 에이전트의 결과를 통합하여 종합적인 결론을 만드는데, 여러 에이전트의 결과가 상충하는 경우는 우리가 우선순위와 상황에 따라 판단해야한다.

<br>

### 서브 에이전트 재개
서브 에이전트는 기본적으로 매번 새 컨텍스트에서 시작한다. <br>
이전 작업을 이어서 하고 싶을 때 `재개(resume) 기능`을 명시적으로 사용한다. <br>
각 서브에이전트는 고유한 agentId가 할당되며, 이 ID를 사용하여 재개가 가능 <br>
재개시 이전 대화의 전체 컨텍스트가 로드되므로 중단된 작업을 이어서 가능 <br>

초기 호출시
```
code-reviewer 서브 에이전트로 인증 모듈을 리뷰해줘. 완료되면 agentId를 알려줘.
```

작업이 끝나고 재개해야하는 경우
```
에이전트 asdf1 을 재개하고 권한 부여 로직도 함께 분석해줘.
```

<br>

서브 에이전트 대화 기록은 ~/.claude/projects/{프로젝트명}/{세션ID}/subagents/agent-{agentId}.jsonl 파일로 저장된다. <br>
메인 대화가 압축되더라도 서브 에이전트 기록은 영향을 받지 않는다.

<br>

### 서브에이전트 성능 측정
서브에이전트의 효과를 측정하기 위해서는 **작업 완료 시간, 컨텍스트 사용량** 등을 지표로 삼는다. <br>

작업 완료 시간 : 에이전트가 시작하고 결과를 반환하기까지의 시간 <br>
컨텍스트 사용량 : 에이전트가 사용한 컨텍스트의 양을 나타내는 지표 <br>

## 3. 커스텀 서브 에이전트
### 프론트매터
- 서브 에이전트 프론트매터에 `name` 과 `description` 필드는 필수이다.
- `tools` 필드는 에이전트가 사용할 수 있는 도구를 명시한다.
- `model` 필드는 에이전트가 사용할 모델을 명시하며, 지정하지 않으면 현재 세션에서 사용 중인 모델을 상속(inherit)한다.
- `disallowedTools` 필드는 `tools` 필드와 반대로 차단할 도구 목록. `tools` 필드와 같이 같이 사용할 수 없다.
- `permissionMode` 필드는 에이전트가 토구를 실행할 때 권한을 어떻게 처리할지 정의한다.
    - default, acceptEdits, dontAsk, bypassPermissions, plan
- `hooks` 필드는 서브 에이전트에 적용할 훅을 정의한다. 전역 훅 과는 독립적으로 작동
  - `PreToolUse` : 도구(Bash, Edit, Write 등)를 호출하기 직전에 실행, 전처리 작업이라고 생각하면 된다. 도구 호출을 차단하거나 허용 가능
  - `PostToolUse` : 도구 호출이 완료된 후에 실행
  - `Stop` : 에이전트 실행이 종료될 때 호출
- `memory` : 에이전트가 학습한 내용을 유지, MEMORY.md 파일에 컨텍스트가 로드되고 메모리 파일에 접근하기 위해 Read, Write, Edit 도구가 자동 활성화된다.
- `skills` : 시작 시 컨텍스트에 주입할 스킬 목록
  - 스킬 나열시 쉼표로 구분하며 나열한 순서가 우선 순위이다.
  - 스킬 조합시 컨텍스트가 증가하므로 일반 규칙은 먼저, 특수한 규칙(도메인)은 나중에 배치한다.

```markdown
---
name: code-reviewer
description: 코드 리뷰를 수행하는 서브 에이전트입니다. 이 에이전트는 코드 변경 사항을 분석하고, 코드 품질, 보안, 성능 및 유지보수성 측면에서 개선점을 제안합니다. 또한, 코드 스타일 가이드라인 준수 여부를 확인하고, 잠재적인 버그나 문제점을 식별합니다.
disallowedTools: Write, Edit
permissionMode: plan
memory: project
hooks:
  Stop:
    - command: "echo 'Code reviewer agent has completed its task.'"
  PostToolUse:
    - command: "echo 'Tool usage completed. Reviewing results...'"
---
```

<br>

### 마크다운 본문
- 본문은 프롬프트가 되며
    - 너무 긴 프롬프트는 지침을 놓치게된다.
    - 상충되는 지시는 넣지 않는다.
    - 암묵적인 가정은 오해를 발생시킨다.
    - 모든 지시는 구체적이고 명시적이어야한다.
- 출력 형식을 명시할 수 있는데, 후속 처리가 필요할 때는 JSON 형식이 좋다.
- 에이전트가 처리하지 않아야할 경계 조건을 명시하는 것도 중요하다. 경계 조건은 긍정적인 지시와 부정적인 지시 모두 포함한다.
  - 무엇을 해야하는지, 무엇을 하지 말아야하는지

## 4. 에이전트 팀
- settings.json env 필드에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 로 설정하면 클로드가 역할별 에이전트를 자동으로 생성하고 병렬로 실행시킬 수 있다.

에이전트 팀의 시작은 사용자가 직접 팀 생성을 명시적으로 요청하거나 클로드가 팀 생성을 제한할 수도 있다.

> 3명의 팀원을 만들어서 아키텍쳐 설계, 코드 구현, 코드 리뷰를 각각 담당해

에이전트 팀 아키텍쳐는 네가지 구성 요소로 되어있다.
- 팀 리더 : 메인 에이전트, 세션
- 팀원 : 할당된 태스크를 수행
- 태스크 목록 : 팀원들이 공유하는 작업 항목 목록
- 메일 박스 : 에이전트 통신을 위한 메시징 시스템

### 서브에이전트와 에이전트 팀의 차이는
서브에이전트끼리 직접 주고받을 순 없지만 에이전트 팀의 각 팀원은 서로 직접 메시지를 주고받을 수 있다.
서브에이전트는 작업이 끝나면 컨텍스트와 함께 소멸되지만, 에이전트의 팀원은 팀이 해산될 때 까지 컨텍스트를 유지한다.

<br>

### CLI 기반 일회성 에이전트
보통은 사전에 에이전트를 정의하고 반복 활용하여 사용하지만 CLI로 즉석에서 일회성으로 만들수도 있다.

> $ claude -p "최근 변경 사항을 리뷰해줘" --agent code-reviewer

        서브 에이전트는 새로 시작할 때 컨텍스트를 다시 수집해야하는 오버헤드가 발생 <br>
        반복적인 피드백이 필요하거나 하나의 흐름으로 이어지는 작업은 메인 대화에서 직접 하는게 이득
    </div>
    </details>
3. context:fork 와 서브 에이전트의 차이
    <details>
    <summary>토글</summary>
    <div markdown="1">
        context:fork는 현재 컨텍스트를 복제하여 새로운 컨텍스트를 생성하는 기능이고, <br>
        서브 에이전트는 실행시 독립적인 컨텍스트를 가진 별도의 작업 단위이다. <br>
        서브 에이전트는 자체 시스템 프롬프트와 도구를 가질 수 있으며, 작업을 위임하고 결과만 반환하는 구조로 설계되어 있다.
