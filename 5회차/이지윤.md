# 클로드 코드 출력 스타일 & 상태 표시줄, 쉽게 이해하기

## 1. 그림 5-1: 스타일을 고르면 벌어지는 일

사용자가 요청을 하면 → 어떤 출력 스타일(Output Style)을 쓸지 고르고 → Default/Explanatory/Learning/Custom 중 하나에 맞는 지침서(시스템 프롬프트: AI가 지켜야 할 기본 규칙 문서)로 바뀌고 → 그 지침서대로 답변을 만들고 → 최종 출력이 나옵니다. 연극배우가 대본(시스템 프롬프트)을 바꿔 들면 그 대본에 맞는 역할을 연기하는 것과 같습니다.

이렇게 대본을 통째로 바꿀 수 있다는 건, 클로드 코드를 코딩 말고 다른 용도로도 쓸 수 있다는 뜻입니다. 예를 들면 기술 문서 작성이나 교육 콘텐츠 만들기처럼요. 파일을 읽고 쓰는 손발(로컬 스크립트 실행, 파일 작업)은 그대로 두고, 말투와 성격만 완전히 바꿀 수 있습니다.

## 2. 내장된 3가지 성격 모드

터미널에 `/output-style`이라고 치면 목록이 뜨는데, 세 가지가 기본 내장돼 있습니다.

**Default(기본형)** — 소프트웨어 엔지니어링에 최적화된 원래 성격입니다. 코드 분석, 생성, 디버깅을 효율적이고 간결하게 해줍니다.

**Explanatory(설명형)** — 코드를 짜면서 "왜 이렇게 짰는지" 교육적인 통찰(인사이트)을 곁들여줍니다. 선생님이 문제를 풀면서 풀이 과정을 같이 설명해주는 것과 같습니다. 코드 리뷰 학습이나 새 기술 스택 익힐 때 좋습니다.

**Learning(학습형)** — AI가 힌트와 핵심 조각만 주고, 실제 코드 작성은 사람이 직접 하게 시킵니다. 코드 안에 `TODO(human)`이라는 표시를 남겨서 "이 부분은 네가 채워라"라고 알려줍니다. 헬스 트레이너가 운동을 대신 안 해주고 옆에서 자세만 코치해주는 것과 비슷합니다. AI에 너무 의존하지 않고 직접 배우고 싶은 사람에게 맞습니다.

## 3. 스타일 바꾸는 방법

가장 간단한 방법은 `/output-style` 명령입니다. 그냥 치면 목록이 뜨고 고를 수 있고, `/output-style explanatory`처럼 이름을 바로 붙이면 즉시 전환됩니다.

이렇게 바꾼 설정은 `.claude/settings.local.json`(내 컴퓨터에만 저장되는 개인 설정, 깃(Git, 버전관리 도구)에 안 올라감)에 저장됩니다. 그래서 팀원마다 자기 취향대로 다르게 쓸 수 있습니다. 반대로 팀 전체가 같은 스타일을 쓰게 강제하려면 `.claude/settings.json`(팀 전체 공용 설정)에 다음처럼 적으면 됩니다.

```json
{
  "outputStyle": "explanatory"
}
```

이렇게 해두면 그 프로젝트를 열 때마다 자동으로 그 스타일이 적용됩니다.

## 4. 나만의 스타일 직접 만들기

내장된 3개로 부족하면 직접 만들 수 있습니다. 마크다운(글쓰기용 텍스트 형식) 파일로 쓰는데, 내 계정 전체에서 쓰고 싶으면 `~/.claude/output-styles/`(모든 프로젝트 공용 서랍)에, 이 프로젝트에서만 쓰고 싶으면 `.claude/output-styles/`(이 프로젝트 전용 서랍)에 넣습니다.

파일 구조는 이렇습니다. 맨 위에 `---`로 감싼 부분(프론트매터라고 부름)에 이름(`name`)과 설명(`description`)을 적고, 그 아래 본문에 자유롭게 규칙을 적습니다.

```markdown
---
name: korean-architecture-focused
description: 한국어로 응답하며 아키텍처 결정에 집중하는 스타일
---

# 커스텀 지침

## 응답 언어
항상 한국어로 응답합니다.
기술 용어는 필요에 따라 영어 원문을 사용하되
한국어 설명을 함께 제공합니다.

## 집중 영역
- 아키텍처 결정과 트레이드오프 설명
- 상세한 코드 주석 제공
- 디자인 패턴과 적용 방법 논의

## 코드 예제
코드 예제를 제공할 때:
- 한국어로 된 포괄적인 인라인 주석 포함
- 구현 선택의 이유 설명
- 잠재적인 대안과 트레이드오프 강조
```

주의할 점 하나: 커스텀 스타일을 만들면 원래 있던 코딩 관련 기본 규칙이 사라지고 내가 쓴 규칙으로 완전히 교체됩니다. 만약 기존 코딩 규칙은 유지하면서 내 규칙만 추가하고 싶으면 `keep-coding-instructions: true`라는 옵션을 넣으면 됩니다.

## 5. 비슷한 기능들과의 차이 (그림 5-2)

여기서 헷갈리는 4가지를 비교합니다.

| 기능 | 하는 일 | 비유 |
|---|---|---|
| Output Style (출력 스타일) | 시스템 프롬프트를 완전히 대체(통째로 교체) | 대본을 새 대본으로 바꿔치기 |
| CLAUDE.md | 시스템 프롬프트에 추가(원래 규칙 유지 + 메모 덧붙이기) | 대본에 포스트잇 붙이기 |
| 서브에이전트 (Sub-agent) | 독립된 컨텍스트(대화 맥락)에서 작업을 위임받아 처리 | 다른 방에 보내서 따로 일 시키기 |
| 슬래시 명령어 (Slash command) | 저장해둔 프롬프트를 실행 | 즐겨찾기 버튼 누르기 |

이 네 가지가 합쳐져서 "응답 방식"과 "작업 실행" 두 갈래로 나뉘어 최종 출력을 만듭니다.

정리하면 CLAUDE.md는 원래 규칙 + 추가 메모, 출력 스타일은 규칙 전체 교체입니다. 서브에이전트는 독립된 방에서 일하기 때문에 메인 대화의 출력 스타일에 영향을 안 받습니다.

그리고 스킬(Skill)이라는 것도 있는데, 출력 스타일이 "어떻게 말할지"를 정한다면 스킬은 "특정 작업을 할 때 자동으로 불러오는 전문 지식"입니다. 이 둘을 조합하면 특정 스타일로 특정 작업을 일관되게 처리하는 흐름을 만들 수 있습니다.

## 6. 커스텀 스타일 저장 후 사용

저장한 커스텀 스타일은 `/output-style:new` 명령으로 전환할 수 있고, 목록에도 이름이 뜨고, 이름을 바로 지정해서 전환할 수도 있습니다.

## 7. 팀 표준 스타일 만들기

팀 전체가 같은 스타일을 쓰려면 프로젝트의 `.claude/output-styles/`에 표준 스타일을 정의하고 깃(버전관리)에 커밋해서 올려두면 모든 팀원이 공유하게 됩니다.

표준을 설계할 때 고려할 3가지가 있습니다.

1. **팀의 코딩 관습과 문서화 표준을 반영** — 주석 언어, 변수명 규칙, 설명 방식, 아키텍처 결정 문서화 수준 등을 명시
2. **응답 상세 수준 정하기** — 숙련팀은 간결하게, 주니어가 많으면 상세하게
3. **보안 관련 지침 포함** — 민감 정보 처리 방식, 피해야 할 패턴 등

책 예시입니다.

```markdown
---
name: team-standard
description: 팀 표준 출력 스타일 - 코드 리뷰 가이드라인 준수
---
# 팀 표준 스타일
## 언어와 톤
설명은 한국어로, 코드 주석은 영어로 작성
전문적이지만 접근하기 쉬운 톤을 유지

## 코드 품질 집중 사항
- 새로운 코드에 대한 테스트 케이스를 항상 제안
- 잠재적인 보안 문제를 강조
- 적용 가능한 경우 디자인 패턴을 추천

## 문서화 표준
- JSDoc/docstring 형식을 일관되게 사용
- 알고리즘에 대한 복잡도 분석을 포함
- 의존성과 버전 요구 사항을 명시

## 리뷰 가이드라인
- 코드 스멜 패턴을 지적
- 리팩터링 기회를 제안
- 관련된 경우 팀 코딩 표준을 참조
```

이 팀 표준을 기본값으로 만들려면 `.claude/settings.json`에 `outputStyle` 필드에 그 스타일 이름을 넣으면 됩니다.

## 8. 스타일 검증과 다국어 출력

새로 만든 스타일이 잘 작동하는지 확인하는 3단계입니다.

1. 간단한 질문으로 언어, 톤, 형식이 맞는지 확인
2. 복잡한 코드 분석·리팩터링 작업을 시켜보며 실전에서도 일관되는지 확인
3. 특정 패턴 회피 같은 세부 지침이 실제로 지켜지는지 확인

지침이 너무 모호하면 일관성이 떨어지고, 너무 상세하면 유연성이 줄어드니 균형을 맞춰야 합니다.

여러 나라 사람이 같이 쓰는 팀이면 언어 전환 규칙도 정할 수 있습니다. 예를 들면 기술 문서는 영어, 사용자 설명은 한국어, 코드 주석은 영어인데 대화만 한국어 같은 식으로요. 책 예시입니다.

```markdown
## 언어 규칙
- 대화 응답: 한국어
- 코드 주석: 영어
- 기술 용어 첫 언급: 영어(한국어 설명)
- 에러 메시지: 검색 가능성을 위해 영어
- 문서화: 내부용은 한국어, 공개 API는 영어
```

규칙이 복잡할수록 예시를 곁들여 설명해야 일관성이 유지됩니다.

## 9. 환각(Hallucination) 방지 지침

환각(Hallucination)이란 AI가 모르는 걸 아는 척 지어내는 현상입니다. 존재하지 않는 인터넷 주소(URL)를 마치 진짜처럼 알려주거나, 확인 안 된 API 정보를 단정적으로 말하거나, 실제로 실행 안 한 검증을 실행한 것처럼 보고하는 식으로 나타납니다. 출력 스타일 안에 이걸 막는 규칙을 심어둘 수 있습니다.

### URL·출처 검증 지침

실제 검색 도구(WebFetch, WebSearch)가 돌려준 URL만 인용하고, 패턴으로 추측한 URL은 절대 만들지 말라는 규칙입니다.

```markdown
## 출처 검증 규칙

### URL 인용 정책
- 도구 응답에 실제로 나타난 URL만 인용한다.
- 패턴 기반으로 URL을 추측하여 생성하지 않는다.
- URL이 필요하지만 찾을 수 없는 경우: "검색 결과에서 URL을 찾을 수 없습니다."라고 명시한다.
- 모든 출처에 표시: "WebSearch를 통해 검색됨" 또는 "WebFetch로 확인됨"

### 불확실성 표현
정보를 검증할 수 없는 경우:
- "사용 가능한 문서를 기반으로..." 같은 표현을 사용한다.
- "공식 출처를 통한 검증이 필요합니다."라고 명시한다.
- 검증되지 않은 정보를 확정된 사실로 제시하지 않는다.
```

### 불확실성 표현 허용

AI가 "모르겠다"라고 말하기 어려워하는 것 자체가 환각의 주요 원인입니다. 그래서 "이 정보는 검증이 필요합니다", "확인해 주세요", "확실하지 않습니다" 같은 표현을 대놓고 허용해줍니다.

```markdown
## 불확실성 가이드라인

### 허용되는 표현
- "이 정보는 검증이 필요합니다."
- "제 분석을 기반으로 하지만, 확인해 주세요..."
- "공식 문서를 참조해야 합니다..."
- "이 세부 사항에 대해 확실하지 않습니다."

### 응답 신뢰도 수준
- [검증됨]: 도구 결과를 통해 확인된 정보
- [가능성 높음]: 강력한 패턴 기반이지만 직접 검증되지 않음
- [불확실]: 사용 전 추가 검증 필요
```

### 도구 결과 기반 응답 강제

기술 사양은 반드시 Context7(라이브러리 문서 검색 도구)나 WebFetch로 검색한 걸 근거로 삼고, 패키지 버전은 실제 프로젝트 파일을 확인해서 말하고, API 문서는 AI의 일반 지식이 아니라 도구가 실제로 돌려준 응답을 인용하게 합니다.

```markdown
## 도구 기반 응답 요구 사항

### 정보 출처
- 기술 사양: Context7 또는 WebFetch를 통해 검색해야 한다.
- 패키지 의존성: 실제 프로젝트 파일을 통해 검증해야 한다.
- API 문서: 일반적인 지식이 아닌 특정 도구 응답을 인용해야 한다.

### 인용 형식
모든 기술적 주장에 대해:
- 출처: [사용한 도구 이름과 쿼리]
- 검색 날짜: [가능한 경우]
- 신뢰도: [검증됨/추론됨/불확실]
```

### 팀 환경에서의 환각 방지 표준

이걸 팀 전체에 적용하려면 `.claude/output-styles/`에 검증 요구사항이 포함된 팀 표준 스타일을 만듭니다.

```markdown
---
name: team-verified-style
description: 환각 방지 지침이 포함된 팀 표준 스타일
---
# 팀 검증 스타일

## 핵심 원칙
모든 기술 정보는 검증 가능한 출처로 추적할 수 있어야 한다.

## 검증 요구 사항
- 도구 검증 없이 URL 제공 금지
- 파일 시스템 확인 없이 버전 번호 제공 금지
- 문서 검색 없이 API 동작 주장 금지

## 응답 형식
모든 기술적 주장에 출처 표시 포함:
- 파일 기반: (출처: 경로/파일명:줄번호)
- 웹 기반: (출처: WebFetch/WebSearch를 통한 URL)
- 문서: (출처: Context7를 통한 라이브러리/버전)
```

이렇게 시스템 수준에 심어두면 요청할 때마다 매번 "검증해줘"라고 말할 필요 없이 항상 정확한 답변을 유지할 수 있습니다. 기술 문서 생성, 코드 리뷰, API 통합 작업처럼 정확성이 중요한 작업에서 특히 차이가 납니다.

## 10. 5-2 상태 표시줄 (Status line)

상태 표시줄(Statusline)은 터미널 화면 맨 아래에 항상 떠 있는 한 줄 정보창입니다. 게임 화면 구석의 HP바나 미니맵처럼, 지금 작업 상황을 한눈에 보여줍니다. 기본값도 쓸만하지만 내 스타일대로 커스터마이징하면 더 편해집니다.

책 저자가 쓰는 예시 화면에는 이런 정보들이 나열돼 있었습니다.

| 표시 항목 | 예시 값 | 의미 |
|---|---|---|
| 모델 정보 | Opus 4.6 | 지금 쓰는 AI 종류 |
| 버전 정보 | v2.1.37 | 클로드 코드 버전 |
| 비용 정보 | 125/200K | 지금까지 쓴 토큰/비용 |
| 출력 스타일 | R2-D2 | 별명 붙인 커스텀 스타일 |
| 깃 상태 | +0 M171 ?63 | 코드 추가/수정/추적안됨 파일 개수 |
| 브랜치 | main | 지금 작업 중인 갈래 |

클로드 코드는 이 정보들을 JSON(데이터를 상자처럼 정리해서 담는 형식)으로 스크립트에 전달합니다. 여기 담기는 정보는 모델 이름, 작업 폴더, 깃 브랜치, 세션 비용, 실행 시간, 추가/삭제된 코드 줄 수 등 다양합니다. 사용자는 이 중 원하는 걸 골라 표시줄에 조합할 수 있습니다.

## 11. 상태 표시줄 설정 방법

`/statusline` 명령을 쓰면 클로드 코드가 안내하면서 지금 쓰는 터미널 프롬프트 느낌으로 표시줄을 만들어줍니다. "모델 이름을 주황색으로 표시"처럼 추가 요청도 같이 전달할 수 있습니다.

```
> /statusline 모델 이름을 주황색으로 표시
```

이렇게 만들어진 스크립트는 `~/.claude/statusline.sh`(내 컴퓨터의 스크립트 파일)에 저장되고, 나중에 직접 열어서 고칠 수도 있습니다. 없애고 싶으면 `/statusline clear` 또는 `/statusline delete`를 실행합니다.

파일을 직접 편집하고 싶으면 `.claude/settings.json` 또는 `~/.claude/settings.json`에 `statusLine` 키를 추가합니다.

```json
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh",
    "padding": 0
  }
}
```

`type`은 "command"로 고정(외부 스크립트를 실행하겠다는 뜻), `command`는 그 스크립트 파일 경로, `padding`은 표시줄 주변 여백을 조절합니다.

## 12. 상태 표시줄 스크립트 작동 원리

스크립트는 클로드 코드로부터 JSON 데이터를 stdin(표준 입력, 프로그램이 데이터를 받는 입구)으로 받아서 가공한 다음, 한 줄짜리 텍스트를 stdout(표준 출력, 프로그램이 결과를 내보내는 출구)으로 뱉어냅니다. 배달원이 재료 상자를 주방 입구에 놓고 가면(stdin), 요리사가 요리해서(가공) 완성된 접시를 손님 테이블에 내놓는(stdout) 것과 같습니다.

전달되는 JSON 데이터 예시입니다.

```json
{
  "cwd": "/current/working/directory",
  "session_id": "abc123...",
  "transcript_path": "/path/to/transcript.jsonl",
  "model": {
    "id": "claude-opus-4-6",
    "display_name": "Opus"
  },
  "workspace": {
    "current_dir": "/current/working/directory",
    "project_dir": "/original/project/directory"
  },
  "version": "1.0.80",
  "output_style": {
    "name": "default"
  },
  "cost": {
    "total_cost_usd": 0.01234,
    "total_duration_ms": 45000,
    "total_api_duration_ms": 2300,
    "total_lines_added": 156,
    "total_lines_removed": 23
  },
  "context_window": {
    "total_input_tokens": 15234,
    "total_output_tokens": 4521,
    "context_window_size": 200000,
    "used_percentage": 8,
    "remaining_percentage": 92,
    "current_usage": {
      "input_tokens": 8500,
      "output_tokens": 1200,
      "cache_creation_input_tokens": 5000,
      "cache_read_input_tokens": 2000
    }
  },
  "exceeds_200k_tokens": false,
  "vim": {
    "mode": "NORMAL"
  },
  "agent": {
    "name": "security-reviewer"
  }
}
```

각 필드가 뭔지 정리하면 이렇습니다.

- `cwd` — 현재 작업 폴더 경로
- `session_id` — 이 대화 세션의 고유 번호
- `transcript_path` — 대화 기록이 저장된 파일 경로
- `model` — 현재 쓰는 AI 모델의 id와 표시 이름
- `workspace` — 현재 작업 폴더와 원래 프로젝트 루트 폴더 정보
- `version` — 클로드 코드 버전
- `output_style` — 지금 적용된 출력 스타일 이름
- `cost` — 세션 비용, 총 실행 시간, API 호출 시간, 추가/삭제된 코드 줄 수
- `context_window` — 입출력 토큰 수, 전체 컨텍스트 창 크기, 사용률/남은 비율, 지금 요청의 토큰 사용량(입력/출력/캐시생성/캐시읽기)
- `exceeds_200k_tokens` — 20만 토큰을 넘었는지 여부
- `vim` — vim 편집 모드를 쓴다면 지금 모드
- `agent` — 지금 실행 중인 서브에이전트 이름

책은 특히 `model` 객체에서 모델 ID와 표시 이름을 가져올 수 있고, `workspace` 객체는 현재/프로젝트 루트 폴더를 알려주고, `cost` 객체는 비용·실행시간·코드 변경량 통계를 담고 있다고 강조합니다.

## 13. 업데이트 주기

상태 표시줄은 대화 메시지가 업데이트될 때마다 갱신되고, 최대 300밀리초 간격으로 실행됩니다. 스크립트 실행이 이보다 오래 걸리면 화면에는 이전 값이 그대로 남아 있습니다. 그래서 스크립트는 최대한 빠르게 실행되도록 최적화해야 합니다.

## 14. 그림 5-3: 데이터가 흘러가는 순서

```
클로드 코드
   ↓ (JSON 데이터 생성)
stdin (입구)
   ↓

   ↓
데이터 파싱 (뜯어보기) → 포매팅 (보기 좋게 다듬기)
   ↓
stdout (출구)
   ↓
터미널 하단 표시
```

## 15. Bash로 직접 만들어보는 상태 표시줄 스크립트

이제부터는 실제로 상태 표시줄 스크립트(status line script, 화면 아래 한 줄을 만들어내는 작은 프로그램)를 코드로 짜보는 부분입니다. 요리 레시피처럼 "재료 받기 → 필요한 것만 골라내기 → 예쁘게 담아 내놓기" 순서로 진행됩니다.

가장 간단한 버전은 모델 이름과 현재 폴더만 보여주는 것입니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# 모델 표시 이름과 현재 디렉터리 추출
MODEL_DISPLAY=$(echo "$input" | jq -r '.model.display_name')
CURRENT_DIR=$(echo "$input" | jq -r '.workspace.current_dir')

# 포매팅 된 상태 표시줄 출력
echo "[$MODEL_DISPLAY] ${CURRENT_DIR##*/}"
```

이 스크립트는 먼저 stdin(입력 상자)에서 JSON 데이터를 통째로 받아 `input`이라는 변수에 담습니다. 그다음 `jq`(JSON을 쉽게 잘라 쓸 수 있게 도와주는 도구, 마치 상자 안에서 원하는 물건만 집게로 집어내는 것과 같음)를 써서 필요한 값만 뽑아내고, `echo`(화면에 글자를 출력하는 명령)로 보기 좋게 찍어냅니다. `${CURRENT_DIR##*/}`는 Bash의 파라미터 확장(문자열에서 원하는 부분만 잘라내는 문법)인데, 긴 폴더 경로에서 맨 마지막 폴더 이름만 뚝 떼어내는 역할을 합니다.

스크립트 파일을 만든 뒤에는 실행 권한(이 파일을 프로그램으로 실행해도 된다는 허락 도장)을 줘야 합니다.

```
$ chmod +x ~/.claude/statusline.sh
```

여기에 깃 브랜치(Git branch, 코드 작업의 갈래) 정보까지 추가한 버전입니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# 모델 표시 이름과 현재 디렉터리 추출
MODEL_DISPLAY=$(echo "$input" | jq -r '.model.display_name')
CURRENT_DIR=$(echo "$input" | jq -r '.workspace.current_dir')

# 깃 브랜치 정보 확인
GIT_BRANCH=""
if git rev-parse --git-dir > /dev/null 2>&1; then
    BRANCH=$(git branch --show-current 2>/dev/null)
    if [ -n "$BRANCH" ]; then
        GIT_BRANCH=" | $BRANCH"
    fi
fi

# 상태 표시줄 출력(브랜치 정보 포함)
echo "[$MODEL_DISPLAY] ${CURRENT_DIR##*/}$GIT_BRANCH"
```

`git rev-parse --git-dir`은 "여기가 깃 저장소 맞아?"라고 물어보는 명령이고, `git branch --show-current`는 "지금 어느 갈래에 있어?"라고 물어서 브랜치 이름을 가져옵니다. 깃 저장소가 아닌 폴더라면 브랜치 표시 없이 기본 정보만 보여줍니다.

장시간 작업하거나 비용이 중요한 프로젝트라면 세션 비용(session cost, 지금까지 쓴 돈)과 코드 변경량 통계도 보고 싶을 수 있습니다. `cost` 객체(JSON 안의 비용 관련 정보 묶음)에서 뽑아내면 됩니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# 필요한 정보 추출
MODEL_DISPLAY=$(echo "$input" | jq -r '.model.display_name')
CURRENT_DIR=$(echo "$input" | jq -r '.workspace.current_dir')
COST=$(echo "$input" | jq -r '.cost.total_cost_usd')
LINES_ADDED=$(echo "$input" | jq -r '.cost.total_lines_added')
LINES_REMOVED=$(echo "$input" | jq -r '.cost.total_lines_removed')

# 비용을 소수점 4자리까지 표시
COST_FORMATTED=$(printf "%.4f" "$COST")

# 상태 표시줄 출력(비용 및 코드 변경량 포함)
echo "[$MODEL_DISPLAY] ${CURRENT_DIR##*/} | \$$COST_FORMATTED | +$LINES_ADDED -$LINES_REMOVED"
```

이 스크립트는 모델 이름, 폴더 이름과 함께 세션 비용(달러 단위)과 추가·삭제된 코드 줄 수를 같이 보여줍니다. 비용은 소수점 4자리까지 보여줘서 아주 세밀하게 추적할 수 있습니다.

## 16. 파이썬과 Node.js로도 만들 수 있어요

Bash가 낯설거나 복잡한 로직이 필요하면 파이썬(Python)으로도 똑같은 걸 만들 수 있습니다. 파이썬은 JSON 다루기가 특히 편합니다.

```python
#!/usr/bin/env python3
import json
import sys
import os

# stdin에서 JSON 데이터 로드
data=json.load(sys.stdin)

# 필요한 정보 추출
model=data['model']['display_name']
current_dir=os.path.basename(data['workspace']['current_dir'])

# 깃 브랜치 정보 확인
git_branch=""
if os.path.exists('.git'):
    try:
        with open('.git/HEAD', 'r') as f:
            ref=f.read().strip()
            if ref.startswith('ref: refs/heads/'):
                git_branch=f" | {ref.replace('ref: refs/heads/', '')}"
    except Exception:
        pass  # 깃 저장소가 아니거나 HEAD 파일을 읽을 수 없음

# 상태 표시줄 출력
print(f"[{model}] {current_dir}{git_branch}")
```

파이썬 버전은 `json.load`(JSON 문자열을 파이썬이 다룰 수 있는 데이터로 바꿔주는 함수)로 stdin을 바로 읽어들입니다. 그리고 깃 브랜치 정보를 `git` 명령을 실행하지 않고 `.git/HEAD`라는 파일을 직접 열어서 읽는데, 이 방법이 `git` 명령을 실제로 실행하는 것보다 더 빠릅니다.

프론트엔드 개발자에게 익숙할 Node.js 버전도 있습니다.

```javascript
#!/usr/bin/env node

const fs=require('fs');
const path=require('path');

// stdin에서 JSON 데이터 수신
let input='';
process.stdin.on('data', chunk => input += chunk);
process.stdin.on('end',() => {
    // JSON 파싱
    const data=JSON.parse(input);

    // 필요한 정보 추출
    const model=data.model.display_name;
    const currentDir=path.basename(data.workspace.current_dir);

    // 깃 브랜치 정보 확인
    let gitBranch='';
    try {
        const headContent=fs.readFileSync('.git/HEAD', 'utf8').trim();
        if(headContent.startsWith('ref: refs/heads/')) {
            gitBranch=` | ${headContent.replace('ref: refs/heads/', '')}`;
        }
    } catch(e) {
        // 깃 저장소가 아니거나 HEAD 파일을 읽을 수 없음
    }

    // 상태 표시줄 출력
    console.log(`[${model}] ${currentDir}${gitBranch}`);
});
```

Node.js 스크립트는 비동기 이벤트(데이터가 들어올 때마다 알림을 받는 방식) 기반으로 stdin을 처리합니다. 데이터가 다 들어오면(`end` 이벤트) 그때 JSON을 해석하고 상태 표시줄 문자열을 만듭니다. `process.stdin.on('data', ...)`은 데이터가 조각조각 들어올 때마다 이어붙이는 부분이고, `end`가 오면 완성된 데이터를 파싱합니다.

## 17. 코드 정리하기(헬퍼 함수)와 색깔 입히기(ANSI 코드)

스크립트가 점점 복잡해지면 헬퍼 함수(helper function, 반복되는 작은 작업을 따로 떼어내 이름 붙여둔 미니 함수)로 정리하는 게 좋습니다. 서랍마다 라벨을 붙여두면 나중에 찾기 쉬운 것처럼, 각 정보를 꺼내오는 로직을 함수로 감싸두면 유지보수가 훨씬 편해집니다. 새 정보를 추가하거나 형식을 바꿀 때 그 함수만 고치면 됩니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# 헬퍼 함수 정의: 각 필드를 추출하는 함수
get_model_name() { echo "$input" | jq -r '.model.display_name'; }
get_current_dir() { echo "$input" | jq -r '.workspace.current_dir'; }
get_project_dir() { echo "$input" | jq -r '.workspace.project_dir'; }
get_version() { echo "$input" | jq -r '.version'; }
get_cost() { echo "$input" | jq -r '.cost.total_cost_usd'; }
get_duration() { echo "$input" | jq -r '.cost.total_duration_ms'; }
get_lines_added() { echo "$input" | jq -r '.cost.total_lines_added'; }
get_lines_removed() { echo "$input" | jq -r '.cost.total_lines_removed'; }

# 상태 표시줄 구성: 필요한 정보 조합
MODEL=$(get_model_name)
DIR=$(get_current_dir)
COST=$(get_cost)

# 상태 표시줄 출력
echo "[$MODEL] ${DIR##*/} | \$$(printf '%.3f' $COST)"
```

그리고 표시줄에 색깔을 입히면 중요한 정보를 더 빨리 알아볼 수 있습니다. 클로드 코드는 ANSI 색상 코드(터미널 화면에 글자 색을 입히는 특수 문자 코드)를 지원합니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# ANSI 색상 코드 정의
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[0;33m'
BLUE='\033[0;34m'
MAGENTA='\033[0;35m'
CYAN='\033[0;36m'
NC='\033[0m' # 색상 없음(No Color)

# 필요한 정보 추출
MODEL_DISPLAY=$(echo "$input" | jq -r '.model.display_name')
CURRENT_DIR=$(echo "$input" | jq -r '.workspace.current_dir')
LINES_ADDED=$(echo "$input" | jq -r '.cost.total_lines_added')
LINES_REMOVED=$(echo "$input" | jq -r '.cost.total_lines_removed')

# 깃 브랜치 정보 확인
GIT_BRANCH=""
if git rev-parse --git-dir > /dev/null 2>&1; then
    BRANCH=$(git branch --show-current 2>/dev/null)
    if [ -n "$BRANCH" ]; then
        GIT_BRANCH=" ${CYAN}$BRANCH${NC}"
    fi
fi

# 색상이 적용된 상태 표시줄 출력
echo -e "${YELLOW}[$MODEL_DISPLAY]${NC} ${BLUE}${CURRENT_DIR##*/}${NC}$GIT_BRANCH | ${GREEN}+$LINES_ADDED${NC} ${RED}-$LINES_REMOVED${NC}"
```

이 스크립트에서는 모델 이름은 노란색, 폴더는 파란색, 깃 브랜치는 청록색, 추가된 줄 수는 초록색, 삭제된 줄 수는 빨간색으로 표시됩니다. 꼭 `echo -e` 옵션(이스케이프 문자를 색깔로 해석해달라는 지시)을 써야 색상 코드가 진짜 색깔로 나타납니다.

## 18. 빨리 실행되게 만들기(성능 최적화, 캐싱)

상태 표시줄 스크립트는 대화할 때마다 자주 실행되기 때문에 속도가 중요합니다. 특히 깃 상태 확인처럼 시간이 좀 걸리는 작업은 캐싱(caching, 한 번 구한 결과를 잠깐 저장해두고 재사용하는 것 — 매번 새로 요리하지 않고 방금 만든 음식을 잠깐 보관해뒀다 다시 내놓는 것과 비슷함)하는 게 좋습니다.

```bash
#!/bin/bash
# stdin에서 JSON 데이터 읽기
input=$(cat)

# 캐시 파일 설정
CACHE_FILE="/tmp/claude-statusline-git-cache"
CACHE_TTL=5  # 5초 캐시 유효 시간

# 캐시 확인: 캐시가 존재하고 유효 시간 내에 있는지 체크
if [ -f "$CACHE_FILE" ]; then
    CACHE_AGE=$(($(date +%s) - $(stat -f%m "$CACHE_FILE" 2>/dev/null || stat -c%Y "$CACHE_FILE" 2>/dev/null)))
    if [ "$CACHE_AGE" -lt "$CACHE_TTL" ]; then
        GIT_BRANCH=$(cat "$CACHE_FILE")
    fi
fi

# 캐시가 없거나 만료되었으면 새로 가져오기
if [ -z "$GIT_BRANCH" ]; then
    if git rev-parse --git-dir > /dev/null 2>&1; then
        GIT_BRANCH=$(git branch --show-current 2>/dev/null)
        echo "$GIT_BRANCH" > "$CACHE_FILE"
    fi
fi

# 기본 정보 추출
MODEL_DISPLAY=$(echo "$input" | jq -r '.model.display_name')
CURRENT_DIR=$(echo "$input" | jq -r '.workspace.current_dir')

# 상태 표시줄 출력(브랜치 정보가 있으면 포함)
if [ -n "$GIT_BRANCH" ]; then
    echo "[$MODEL_DISPLAY] ${CURRENT_DIR##*/} | $GIT_BRANCH"
else
    echo "[$MODEL_DISPLAY] ${CURRENT_DIR##*/}"
fi
```

이 스크립트는 깃 브랜치 정보를 5초 동안 캐시(TTL, Time To Live: 캐시가 살아있는 유효 시간)해둡니다. 브랜치는 자주 안 바뀌니까 5초 정도면 충분하다는 뜻입니다.

## 19. 안 될 때 확인할 것들(문제 해결)

상태 표시줄이 안 뜬다면 세 가지를 확인하라고 책은 알려줍니다.

첫째, 스크립트에 실행 권한이 있는지 확인합니다.

```
$ chmod +x ~/.claude/statusline.sh
```

둘째, 스크립트가 stdout(정상 출력 통로)으로 결과를 내보내는지 확인합니다. stderr(에러 전용 통로)로 내보내면 화면에 안 나타납니다. JSON 데이터를 직접 흉내 내서 스크립트를 테스트해볼 수 있습니다.

```
$ echo '{"model":{"display_name":"Test"},"workspace":{"current_dir": "/test"}}' | ./statusline.sh
```

셋째, `jq`(JSON을 잘라 쓰는 도구)가 설치돼 있는지 확인합니다. 없으면 Bash 스크립트의 JSON 파싱이 실패합니다.

```
# macOS
$ brew install jq

# 우분투/데비안
$ sudo apt-get install jq
```

## 20. 5-3 대화형 모드 활용 — 키보드 단축키 기본기

이 절은 클로드 코드를 터미널에서 타자로 조작할 때 쓰는 단축키(키보드 조합)를 다룹니다. 기본 4가지입니다.

`Ctrl+C`는 지금 하고 있는 입력이나 클로드의 답변 생성을 멈추는 표준 중단 신호입니다. 클로드가 긴 답변을 쓰는 중에 방향을 바꾸고 싶으면 이걸로 끊고 새로 지시하면 됩니다.

`Ctrl+D`는 클로드 코드 세션 자체를 끝내는 신호(EOF, End of File: "더 이상 보낼 데이터가 없다"는 뜻의 신호)입니다. 작업이 진행 중이면 정말 끝낼 건지 확인을 물어볼 수 있습니다.

`Ctrl+L`은 화면만 깨끗이 지우고 대화 기록은 그대로 남겨둡니다. 화면이 지저분해졌을 때 정리하는 용도입니다.

`Ctrl+O`는 상세 출력 모드(클로드가 어떤 도구를 썼고 결과가 어땠는지 자세히 보여주는 모드)를 켜고 끕니다. 평소엔 간결하게 보다가 디버깅할 때만 상세하게 켜면 됩니다.

## 21. 고급 제어 기능

화살표 키(위/아래)로 예전에 쳤던 입력을 다시 불러올 수 있습니다. 반복 명령을 다시 쓰거나 조금 고쳐서 재사용할 때 편합니다. 이 기록은 작업 폴더별로 따로 저장돼서, 프로젝트마다 다른 기록을 갖습니다.

`Esc`를 두 번 연속으로 누르면 코드와 대화를 이전 상태로 되감을 수 있습니다(체크포인팅 시스템: 게임의 세이브 포인트처럼 특정 시점으로 되돌아가는 구조와 연결돼 있음). 클로드가 실수했거나 원하는 방향이 아니면 이 단축키로 빠르게 되돌릴 수 있습니다.

`Tab` 키는 확장 사고(extended thinking, AI가 답을 내기 전에 더 깊게, 더 체계적으로 생각하는 모드 — 문제를 풀기 전에 낙서장에 한참 끄적이며 고민하는 것과 비슷함)를 켜고 끕니다. 자원을 더 많이 쓰기 때문에 복잡한 아키텍처 결정이나 까다로운 디버깅에서만 선택적으로 쓰는 게 좋습니다.

`Shift+Tab` 또는 `Alt+M`은 권한 모드(클로드가 도구를 쓸 때 얼마나 허락을 받아야 하는지 정하는 모드)를 순환 전환합니다. 표준 모드(매번 확인받음), 자동 수락 모드(허용된 도구는 알아서 실행), 계획 모드(계획만 세우고 실행은 대기) 세 가지를 오갑니다.

## 22. 여러 줄 한 번에 입력하기

복잡한 지시나 코드 블록을 넣을 때는 줄바꿈이 있는 여러 줄 입력이 필요합니다. 방법이 여러 가지입니다.

가장 간단한 방법은 줄 끝에 백슬래시(`\`)를 쓰고 `Enter`를 누르는 것입니다. 모든 터미널에서 작동합니다. macOS 터미널이라면 `Option+Enter`도 됩니다. `Ctrl+J`는 줄바꿈 문자를 직접 찍어 넣는 제어 시퀀스입니다.

`/terminal-setup` 명령을 실행하면 `Shift+Enter`로 줄바꿈을 넣을 수 있게 터미널 설정을 조정해줍니다(iTerm2, VS Code 터미널 대상). 한 번 설정해두면 계속 `Shift+Enter`로 여러 줄을 입력할 수 있습니다.

코드 블록이나 로그처럼 긴 여러 줄 텍스트는 그냥 붙여넣기(paste)로도 됩니다. macOS는 `Cmd+V`, 윈도우·리눅스는 `Ctrl+V`로 클립보드 내용을 붙이면 여러 줄이 한 번에 들어갑니다.

## 23. 첫 글자만 봐도 다르게 동작하는 빠른 명령 접두사

클로드 코드는 입력이 특정 문자로 시작하면 특별하게 처리합니다. 네 가지 접두사(prefix, 문장 맨 앞에 붙는 특수 기호 — 우편번호처럼 어디로 보낼지 미리 알려주는 표시)가 있습니다.

`#`으로 시작하면 메모리 단축키가 됩니다. 이렇게 입력한 내용은 프로젝트의 `CLAUDE.md` 파일(AI가 계속 기억해야 할 프로젝트 규칙을 적어두는 메모장)에 자동으로 추가됩니다. 예를 들어 "#이 프로젝트는 타입스크립트 5.0을 사용한다"라고 치면 그 정보가 저장되어 다음 대화에서도 계속 참조됩니다.

`/`로 시작하면 슬래시 명령(`/help`, `/clear`, `/config` 같은 내장 명령이나 커스텀 명령, 스킬 등을 부르는 것)이 실행됩니다.

`!`로 시작하면 Bash 모드가 켜집니다. `!`로 시작한 입력은 클로드가 해석하지 않고 컴퓨터의 셸 명령(운영체제에 직접 내리는 명령)으로 바로 실행됩니다.

```
> !npm test
> !git status
> !ls -la
```

이 방식은 클로드의 승인 절차 없이 바로 실행되기 때문에 결과를 빨리 확인하거나 클로드에게 현재 상황을 알려줄 때 유용합니다. 예를 들어 `!git status`로 상태를 확인한 뒤 "이 상태를 보고 커밋 메시지를 작성해 줘"라고 요청할 수 있습니다.

`@`로 시작하면 파일 경로 멘션(특정 파일을 콕 집어서 언급하는 것)이 트리거됩니다. `@`를 입력하면 경로 자동완성이 시작되어 파일을 빠르게 지정할 수 있습니다. "@src/components/Button.tsx 이 컴포넌트를 분석해 줘"라고 입력하면 그 파일이 자동으로 클로드가 보는 맥락(컨텍스트)에 포함됩니다.

팁: 터미널 편집(vim 방식)에 익숙하다면 `/vim`(한 번만 적용)이나 `/config`(계속 적용)로 Vim 편집기 모드를 쓸 수 있습니다.

## 24. 예전에 쳤던 명령 다시 찾기

`Ctrl+R`은 역방향 명령 기록 검색(옛날에 입력했던 내용을 키워드로 거꾸로 찾아가는 기능)을 켭니다. 비슷한 작업을 반복할 때 유용합니다.

검색을 시작하면 입력창이 검색 모드로 바뀌고, 검색어를 치면 일치하는 예전 입력이 강조되어 보입니다. `Ctrl+R`을 계속 누르면 그다음으로 일치하는 항목으로 넘어갑니다. `Tab`이나 `Esc`를 누르면 그 항목을 편집할 수 있게 가져오고, `Enter`를 누르면 바로 실행됩니다. `Ctrl+C`를 누르거나 빈 검색에서 백스페이스를 누르면 검색이 취소됩니다.

명령 기록은 작업 폴더별로 저장되고, `/clear` 명령을 실행하면 지워집니다. `!`로 시작하는 기록 확장 기능은 기본적으로 꺼져 있어서, 실수로 옛날 명령이 확장돼버리는 문제를 막아줍니다.

## 25. 명령을 뒤로 보내고 계속 작업하기(백그라운드 Bash)

명령이 끝나기 전에도 클로드 코드를 계속 쓰고 싶다면 백그라운드 실행(background, 화면에 안 보이는 뒤쪽에서 계속 일이 진행되는 것 — 세탁기를 돌려놓고 그 사이에 다른 집안일을 하는 것과 비슷함) 기능을 씁니다.

방법은 두 가지입니다. 클로드에게 직접 "백그라운드에서 실행해줘"라고 요청하거나, Bash 명령이 실행되는 도중에 `Ctrl+B`를 누르면 그 명령이 뒤로 넘어갑니다. Tmux(터미널을 여러 창으로 나눠 쓰는 도구)를 쓰는 사람은 `Ctrl+B`가 Tmux 자체 단축키와 겹치기 때문에 두 번 눌러야 합니다.

백그라운드로 넘어간 명령은 고유한 작업 번호(작업 ID)를 갖고 비동기적으로(다른 일과 동시에) 실행됩니다. 그동안 클로드는 새 질문에 답할 수 있습니다. 명령의 출력 결과는 저장돼 있다가 나중에 `BashOutput`이라는 도구로 확인할 수 있고, 세션을 끝내면 백그라운드 작업도 자동으로 정리됩니다.

이 기능은 빌드 도구(webpack, vite, make), 패키지 관리자(npm, yarn, pnpm), 테스트 도구(jest, pytest), 개발 서버, 도커나 테라폼처럼 오래 걸리는 작업에 씁니다. 빌드가 도는 동안 다른 파일을 분석하거나, 테스트가 도는 동안 코드 리뷰를 요청할 수 있습니다.

## 26. 이미지도 붙여넣을 수 있어요

`Ctrl+V`(macOS/리눅스) 또는 `Alt+V`(윈도우)로 이미지를 클로드 코드에 바로 붙여넣을 수 있습니다. 스크린샷이나 다이어그램을 보여주고 분석을 요청하거나, UI 디자인 이미지를 보고 코드를 만들어달라고 할 수 있습니다.

특히 디버깅할 때 유용합니다. 에러 메시지 화면을 캡처해서 보여주며 해결책을 물어보거나, 화면이 이상하게 나오는 UI 버그를 사진으로 보여줄 수 있습니다. 말로 설명하기 어려운 상황을 이미지 한 장으로 명확히 전달할 수 있습니다.

## 27. 대화를 잘 이끄는 4가지 요령

효과적인 대화 흐름을 짜면 클로드 코드와의 협업 품질이 좋아집니다.

1. **맥락(컨텍스트)을 먼저 준다** — 파일 참조(`@`)나 Bash 명령(`!`)으로 지금 상황을 먼저 알려준 뒤 작업을 요청하면, 클로드가 더 정확하게 답합니다.
2. **복잡한 작업은 잘게 쪼갠다** — "전체 애플리케이션을 리팩터링해 줘"처럼 크게 시키기보다, "먼저 src/utils 폴더의 구조를 분석해 줘"처럼 작게 시작해서 점점 넓혀갑니다. 중간중간 답변을 확인하며 방향을 조정할 수 있습니다.
3. **피드백 루프(되먹임 고리)를 쓴다** — 답이 기대와 다르면 바로 "아니, 그게 아니라..."로 수정을 요청합니다. `Esc+Esc`로 되감은 뒤 다시 시도하는 것도 방법입니다.
4. **확장 사고를 적절히 쓴다** — 복잡한 아키텍처 결정이나 여러 선택지의 장단점(트레이드오프)을 따질 때만 `Tab`으로 확장 사고를 켭니다. 단순한 작업에는 기본 모드가 더 빠르고 효율적입니다.

**그림 5-5: 효과적인 대화 흐름 패턴**

작업 시작 → 컨텍스트 제공 → `@`파일 참조 / `!`Bash 명령 → 작업 요청 → 응답 평가 →

- 불만족이면: 수정 방법 선택 → (수정 요청 → 피드백 제공 → 다시 작업 요청) 또는 (되감기 → `Esc+Esc` 복구 → 다시 작업 요청)
- 만족이면: 다음 단계 → 복잡한 작업인가? → 아니면 기본 모드 유지 → 작업 완료 / 맞으면 `Tab`으로 확장 사고 켜기 → 작업 완료

마치 요리하다가 맛을 보고(응답 평가), 싱거우면 소금을 더 넣거나(수정 요청) 처음부터 다시 하고(되감기), 맛있으면 다음 요리로 넘어가는 과정과 비슷합니다.

## 28. 단축키 한눈에 정리표

| 단축키 | 설명 | 컨텍스트 |

| `Shift+Tab` / `Alt+M` | 권한 모드 토글 | 자동 수락 모드, 계획 모드 및 일반 모드 사이 전환 |
| `Ctrl+B` | 현재 Bash 명령을 백그라운드로 전환 | 백그라운드 작업 실행 |
| `Ctrl+T` | 태스크 목록 토글 | 에이전트 팀 모드에서 공유 태스크 목록 표시 |
| `Shift+Down` | 팀메이트 순환 | 에이전트 팀 모드에서 활성 팀메이트 간 전환 |

## 29. 5장 마무리 & 6장 예고

5장에서는 클로드 코드의 기본 지침서(시스템 프롬프트)를 통째로 바꿔치기하는 출력 스타일 커스터마이징 방법과, AI가 없는 사실을 지어내는 환각(hallucination)을 막는 시스템 수준 지침 작성법을 배웠습니다. 또한 터미널 하단에 작업 폴더, 비용, 깃 상태 등을 실시간으로 보여주는 상태 표시줄 스크립트를 Bash, 파이썬, Node.js로 만드는 법을 익혔고, 여러 줄 입력이나 백그라운드 Bash 실행 같은 대화형 모드의 고급 단축키로 작업 효율을 끌어올리는 방법도 살펴봤습니다.

6장에서는 에이전틱 코딩(agentic coding, AI가 스스로 계획하고 도구를 써가며 작업을 진행하는 방식)의 핵심인 "맥락 유지"를 위한 메모리와 대화 세션 관리 방법을 다룹니다. `CLAUDE.md`를 활용한 계층적 메모리 구성, 모듈형 룰 시스템, 그리고 체크포인팅(`/rewind`, 되감기 기능)과 깃을 결합한 복구 전략으로 장기 프로젝트를 안정적으로 이끄는 방법을 배우게 됩니다.
