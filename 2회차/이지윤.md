# 2-1 기본 작업 방식
- *탐색-계획-구현-커밋 워크플로* 단계별 품질 높이는 방법

### 탐색-계획-구현-커밋 워크플로
- 탐색 단계
  - 코드베이스의 구조와 맥락 파악
  - 광범위한 질문에서 좁혀가는 방식 효과적
- 계획 단계
  - 구현 전략 수립
  - 복잡한 아키텍처 결정이나 다단계 구현이 필요할 때 확장 사고 활용
  - 변경 대상 파일이 3개를 넘거나 구현 방향이 불확실할 때 플랜 모드 사용
- 구현 단계
  - 실제 코드 작성
  - 대규모 수정을 시도하지 않고, 작은 단위로 나누어 검증하면서 오류 최소화
- 커밋 단계
 
### 탐색과 분석
- 특정 기능과 관려된 코드를 찾을 때 세 단계로 접근. 파일 찾기-> 파일 간 상호작용 파악 -> 실행흐름 추적
- 플랜 모드
  - 읽기 전용 모드
  - 적합한 상황
    1. 다단계 기능 구현: 여러 파일에 덜쳐 기능을 구현할 때
    2. 코드 조사 단계: 변경 전 코드베이스를 철저히 탐색할 때
    3. 대화형 협의: 반복 피드백으로 방향 조정 시에 유용
   
### 점진적 개발 전략
- 복잡한 기능을 작은 단위로 분해해 순차적으로 구현하는 접근 방식
- 점진적 개발의 구성  
  - 작업 범위의 명확한 정의  
      전체 작업을 독립적으로 테스트 가능한 단위로 분해
  - 피드백 루프 활용
      - 테스트 결과, 린터 출력, 빌드 로그처럼 구조화된 피드백을 받을 때 오류를 정확히 교정
      - 피드백 루프가 없으면 사용자가 유일한 검증 수단
  - 컨텍스트 관리
    1. `/compact`, `/clear`
    2. `/mcp` 명령을 사용하지 않는 서버 비활성화
    3. 자동 압축 설정 점검
   
#### TDD 기본 패턴
- TDD의 **레드-그린-리팩터 사이클**은 '실패하는 테스트 작성 → 테스트를 통과하는 최소 코드 구현 → 코드 구조 개선'의 세 단계로 구성
- 레드 단계
  - 구현할 기능의 예상 동작을 정의하는 테스트를 먼저 작성
  - 아직 구현 코드가 없으므로 테스트는 실패
- 그린 단계
  - 테스트를 통과하기 위한 최소한의 코드 작성
  - 클로드 코드가 Bash 도구로 테스트를 실행하고 결과를 확인하면서 통과할 때까지 코드를 수정
- 리팩터 단계
  - 통과된 코드의 기능을 유지하며서 구조를 개선하거나 중복을 제거
- TDD에서는 테스트 범위의 누락 부분을 파악하는 것도 중요

### 깃 통합 워크플로
- 여러 기능을 동시에 개발할 때는 git worktree가 유용
- git worktree는 브랜치 전환 없이 여러 브랜치를 동시에 자겅ㅂ 가능

### 비대화형 모드에서의 세션 재개
`claude --continue -p "내 작업을 계속 진행해 줘"`와 같이 `-p` 플래그와 함게 비대화형 모드로 사용하면 스크립트나 자동화 환경에서도 이전 컨텍스트를 유지하면서 작업을 진행할 수 있다.    

### 파이프와 클로드
```
{
  "script": {
    "lint": "eslint src/",
    "lint:claude": "claude -p '너는 코드 리뷰어야. main 브런치 대비 변경된 코드를 검토하고 오타, 네이밍 오루, 잠재적 버그를 찾아줘.'"
  }
}
```
이 방식은 ESLint가 잡지 못하는 맥락적 오류를 AI가 검토할 수 있다는 장점.

# 2-2 설정 관리
settings.json 파일은 개인 설정부터 팀 전체 표준까지 다양한 수준의 설정을 관리할 수 있는 계층적 구조를 제공

### sessions.json 설정과 우선순위
- 권한 규칙, 환경 변수, 도구 설정 등을 JSON 형식으로 정의
- 배치된 위치에 따라 적용 범위가 달라짐
- `/config` 으로 현재 적용 중인 설정을 확인하고 수정
- 설정 우선 순위
    1. 관리자 설정
       - 사용자가 재정의할 수 없으며, 조직 전체의 보안 정책을 강제하는데 사용
    2. 엔터프라이즈 정책
       - 인프라 팀이 시스템 디렉터리에 배포. 사용자 재정의 안됨
    3. 프로젝트 설정
       - `.claude/settinsg.json`에 정의
       - 깃 저장소에 공유해 팀 공유
       - 코드 스타일 가이드라인, 테스트 절차, 프로젝트별 도구 권한
    4. 프로젝트 로컬 설정
       - `.claude/settings.local.json`에 정의
       - 프로젝트 설정보다 우선순위가 높음
       - 개인별 조정
       - 깃 무시
    5. 사용자 설정(우선순위 최하)
       - `~/.claude/settings.json`에 정의
       - 사용자의 모든 프로젝트에 적용
      
### 기본 설정 옵션
- `apiKeyHelper`
  - 개인 사용자는 사용할 일이 없음. 일반 사용자는 `claude login`으로 인증하면 끝.
  - 주로 기업/팀 환경의 보안 정책 때문에 사용
  - API 키가 1시간 마다 만료되는 환경
- `cleanupPeriodDays`
  - 로컬 채팅 기록의 유지 기간 설정 값
- `env`
  - 모든 세션에 적용될 환경 변수를 정의
  - `settings.json`의 `env`는 Claude Code 도구 자체의 환경변수이고, 프로젝트 `.env`는 앱의 환경변수. 서로에게 영향을 주지않음
- `includeCoAuthoredBy`
  - 깃 커밋에 `co-authored-by Claude` 표시를 포함할지 결정
  - 기본값은 `true`
  - 커밋 히스토리에 AI 사용 흔적을 남기고 싶지 않을 때 끄기
- `claude --model sonnet`
  - 프로젝트 틀성에 따라 적합한 모델 선택
- `outputStyle`
  - 출력 형식과 스타일 변경
  - 미리 정의된 프리셋으로 충분하면 `settings.json`만 쓰면 되고, 더 세밀한 제어가 필요하면 `.claude/outputSytle/` 폴더를 만들어서 사용
- `hooks`
  - 도구 실행 전후에 실행되는 커스텀 명령을 정의
  - `{"PreToolUse": {"Bash": "echo 'Running...'"}}` 처럼 사용
 
설정옵션
| 키 | 설명 |
| :---: | :---: |
| `permissions` | 권한 구조 |
| `disableAllHook` | 모든 훅 비활성화 |

권한 설정 옵션
| 키 | 설명 |
| :---: | :---: |
| `allow` | 도구 사용을 허용할 권한 규칙 배열. Bash 규칙은 접두사 일치 사용 |
| `ask` | 도구 사용 시 요청할 권한 규칙 배열 |
| `deny` | 도구 사용을 거부할 권한 규칙 배열. 민감한 파일 제외에 활용 |
| `additionalDirectories` | 클로드 코드가 접근할 수 있는 추가 작업 디렉토리 |
| `defaultMode` | 클로드 코드 시작 시 기본 권한 모드 |
| `disableBypassPermissionsMode` | "diable"로 설정하여 `bypassPermissions` 모드 비활성화 |

### 프로젝트별 설정 전략
- 권한 규칙은 `permissions` 객체 아래 `allow`, `deny`, `ask` 세 배열로 구셩
- 평가 순서는 `deny` -> `ask` -> `allow`
- `deny`에 해당하면 다른 규칙에 관계없이 차단

### 민감한 파일 보호

### 도구별 설정
- 승인 없이 사용 가능한 도구: Read, Glob, Grep, Agent는 읽기 전용 도구
- 승인이 필요한 도구: Bash, Edit, Write, WebFetch, WebSearch 시스템 상태를 변경하거나 외부에 접근할 수 있음

### 클로드 코드의 다섯 가지 확장 도구 비교

  - 클로드가 사용자 요청 맥락을 분석하여 활성화 여부 결정
