# 2-1 기본 작업 방식
- *탐색-계획-구현-커밋 워크플로* 단계별 품질 높이는 방법

### 탐색-계획-구현-커밋 워크플로
- 탐색 단계
  - 코드베이스의 구조와 맥락 파악
  - 광범위한 질문에서 좁혀가는 방식 효과적
- 계획 단계
  - 구현 전략 수립
  - 복잡한 아키텍처 결정이나 다단계 구현이 필요할 때 확장 사고 활용
  - `/config`에서 활성화하거나 `MAX_THINK_TOKENS` 환경 변수로 토큰 예산 조절
- 구현 단계
  - 실제 코드 작성
  - 대규모 수정을 시도하지 않고, 작은 단위로 나누어 검증하면서 오류 최소화
 
### 탐색과 분석
- 프로젝트 구조를 모른다면
  1. `Glob`과 `Grep`로 검색된 경로만 컨텍스트에 적재
  2. 식별된 파일만 Read
- 프로젝트 구조를 안다면
  - `@` 참조로 범위 좁히기
- 플랜 모드
  - 읽기 전용 모드
  - 적합한 상황
    1. 다단계 기능 구현
    2. 코드 조사 단계
    3. 대화형 협의
   
### 점진적 개발 전략
- 복잡한 기능을 작은 단위로 분해해 순차적으로 구현하는 접근 방식
- 점진적 개발의 구성  
  - 작업 범위의 명확한 정의  
      전체 작업을 독립적으로 테스트 가능한 단위로 분해
  - 피드백 루프 활용
      - 테스트 결과, 린터 출력, 빌드 로그처럼 구조화된 피드백을 받을 때 오류를 정확히 교정
      - 피드백 루프가 없으면 사용자가 유일한 검증 수단
  - 컨텍스트 관리
    1. `/compact`, `/clear`
    2. `/mcp` 명령을 사용하지 않는 서버 비활성화
    3. 자동 압축 설정 점검
   
#### TDD 기본 패턴
- TDD의 **레드-그린-리팩터 사이클**은 '실패하는 테스트 작성 → 테스트를 통과하는 최소 코드 구현 → 코드 구조 개선'의 세 단계로 구성
- "먼저 테스트를 작성하고 테스트가 실패하는지 확인한 뒤 테스트를 통과하도록 코드를 구현해 줘"라고 지시하면, 클로드 코드가 Bash 도구로 테스트를 실행하고 결과를 확인하면서 통과할 때까지 코드를 수정한다.    
  🤔🤔🤔 테스트를 claude가 작성하는거같으면 자기 입맛대로 만들건데 정확한 검증이 될까? 사람이 검증해야하는거 아닐까? 무조건 성공하게끔 테스트코드를 짤 수 있는거 아닌가?

### 깃 통합 워크플로
- 여러 기능을 동시에 개발할 때는 깃 작업 트리가 유용

### 비대화형 모드에서의 세션 재개
`claude --continue -p "내 작업을 계속 진행해 줘"`와 같이 `-p` 플래그와 함게 비대화형 모드로 사용하면 스크립트나 자동화 환경에서도 이전 컨텍스트를 유지하면서 작업을 진행할 수 있다.    

### 파이프와 클로드
```
{
  "script": {
    "lint": "eslint src/",
    "lint:claude": "claude -p '너는 코드 리뷰어야. main 브런치 대비 변경된 코드를 검토하고 오타, 네이밍 오루, 잠재적 버그를 찾아줘.'"
  }
}
```
이 방식은 ESLint가 잡지 못하는 맥락적 오류를 AI가 검토할 수 있다는 장점.

# 2-2 설정 관리
settings.json 파일은 개인 설정부터 팀 전체 표준까지 다양한 수준의 설정을 관리할 수 있는 계층적 구조를 제공

### sessions.json 설정과 우선순위
- 권한 규칙, 환경 변수, 도구 설정 등을 JSON 형식으로 정의
- 배치된 위치에 따라 적용 범위가 달라짐
- `/config` 으로 현재 적용 중인 설정을 확인하고 수정
- 설정 우선 순위
    1. 관리자 설정
       - 내가 쓸 일 없음
    2. 엔터프라이즈 정책
       - 내가 쓸 일 없음
    3. 프로젝트 설정
       - `.claude/settinsg.json`에 정의
       - 깃 저장소에 공유해 팀 공유
       - 코드 스타일 가이드라인, 테스트 절차, 프로젝트별 도구 권한
    4. 프로젝트 로컬 설정
       - `.claude/settings.local.json`에 정의
       - 프로젝트 설정보다 우선순위가 높음
       - 개인별 조정
       - 깃 무시
    5. 사용자 설정(우선순위 취하)
       - `~/.claude/settings.json`에 정의
       - 사용자의 모든 프로젝트에 적용
      
### 기본 설정 옵션
- `apiKeyHelper`
  - 인증값을 생성하는 커스텀 스크립트 경로
- `cleanupPeriodDays`
  - 로컬 채팅 기록의 유지 기간 설정 값
- `env`
  - 모든 세션에 적용될 환경 변수를 정의
- `includeCoAuthoredBy`
  - 깃 커밋에 `co-authored-by Claude` 표시를 포함할지 결정
  - 기본값은 `true`
- `claude --model sonnet`
- `outputStyle`
  - 출력 형식과 스타일 변경
- `hooks`
  - 도구 실행 전후에 실행되는 커스텀 명령을 정의
 
설정옵션
| 키 | 설명 |
| :---: | :---: |
| `permissions` | 권한 구조 |
| `disableAllHook` | 모든 훅 비활성화 |
| `statusLine` | 컨텍스트를 표시할 커스텀 상태 표시줄 |
| `outputStyle` | 출력 스타일 구성 |
| `forceLoginMethod` | 로그인 방식 제한 | 
| `enableAllProjectMcpServers` | 프로젝트.mcp.json의 모든 MCP 서버 자동 승인. |
| `enableMcpjsonServers` | 승인할 특정 MCP 서버 목록 | 
| `disableMcpjsonServers` | 거부할 특정 MCP 서버 목록 |

권한 설정 옵션
| 키 | 설명 |
| :---: | :---: |
| `allow` | 도구 사용을 허용할 권한 규칙 배열. Bash 규칙은 접두사 일치 사용 |
| `ask` | 도구 사용 시 요청할 권한 규칙 배열 |
| `deny` | 도구 사용을 거부할 권한 규칙 배열. 민감한 파일 제외에 활용 |
| `additionalDirectories` | 클로드 코드가 접근할 수 있는 추가 작업 디렉토리 |
| `defaultMode` | 클로드 코드 시작 시 기본 권한 모드 |
| `disableBypassPermissionsMode` | "diable"로 설정하여 `bypassPermissions` 모드 비활성화 |

### 프로젝트별 설정 전략
- 마이크로서비스 환경에서는 서비스별로 `.claude/settings.json`을 독립 구성하되, 공통 규칙은 상요자 설정(~./claude/settings.json)에 정의

### 환경 변수 관리
1. 셸에서 직접 export(우선)
2. settings.json의 env 필드
   - 세션을 시작할 때마다 자동 적용

### 팀 표준 설정 구축
- CLAUDE.md
  - 세션 시작 시 클로드 코드의 컨텍스트에 로드되는 메모리 파일
  - 프로젝트의 빌드 명령, 코드 스타일 가이드라인, 테스트 절차, 아키텍처 결정 사항등 기록
  - 약 200줄로 유지
  - 참조용 자료는 스킬로 분리
- 스킬
  - 반복적인 워크플로를 표준화려면 스킬 정의
  - SKILL.md 파일에 YAML프로트메터와 마크다운 지침을 작성

### 민감한 파일 보호
- `permissions.deny` 배열에 차단 규칙 생성
- 스킬: 어떤 워크플로를 반복 실행할지

### 도구별 설정
- 승인 없이 사용 가능한 도구: Read, Glob, Grep, Agent는 읽기 전용 도구
- 승인이 필요한 도구: Bash, Edit, Write, WebFetch, WebSearch 시스템 상태르 ㄹ변경하거나 외부에 접근할 수 있음

### 클로드 코드의 다섯 가지 확장 도구 비교
- 스킬
  - 가장 가볍고 선언적인 확장 방식
  - SKILL.md 파일과 스크립트, 템플릿, 참조 문서 드으이 지원 파일로 구성
  - 클로드가 사용자 요청 맥락을 분석하여 활성화 여부 결정
- 서브에이전트
  - 스킬보다 한 단계 높은 수준의 확장
  - 독릭적인 컨텍스트 윈도우, 커스텀 시스템 프롬프트 그리고 명시적으로 할당된 도구 세트를 갖춘, AI 어시스턴트
- MCP 서버
  - 클로드 코드를 외부 세계와 연결하는 통로
  - `claude mcp add` 명령으로추가
- 훅
  - LLM의 판단을 위회하여 결정론적으로 실행되는 셸 명령
- 플로그인
  - 앞의 확장 방식 네 가지를 하나의 배포 단위로 묶는 컨테이너
  - 마켓플레이스 시스템을 통해 배포와 설치가 이루어짐.
