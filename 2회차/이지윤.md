# 2-1 기본 작업 방식
- *탐색-계획-구현-커밋 워크플로* 단계별 품질 높이는 방법

### 탐색-계획-구현-커밋 워크플로
- 탐색 단계
  - 코드베이스의 구조와 맥락 파악
  - 광범위한 질문에서 좁혀가는 방식 효과적
- 계획 단계
  - 구현 전략 수립
  - 복잡한 아키텍처 결정이나 다단계 구현이 필요할 때 확장 사고 활용
  - 변경 대상 파일이 3개를 넘거나 구현 방향이 불확실할 때 플랜 모드 사용
- 구현 단계
  - 실제 코드 작성
  - 대규모 수정을 시도하지 않고, 작은 단위로 나누어 검증하면서 오류 최소화
- 커밋 단계
 
### 탐색과 분석
- 특정 기능과 관려된 코드를 찾을 때 세 단계로 접근. 파일 찾기-> 파일 간 상호작용 파악 -> 실행흐름 추적
- 플랜 모드
  - 읽기 전용 모드
  - 적합한 상황
    1. 다단계 기능 구현: 여러 파일에 덜쳐 기능을 구현할 때
    2. 코드 조사 단계: 변경 전 코드베이스를 철저히 탐색할 때
    3. 대화형 협의: 반복 피드백으로 방향 조정 시에 유용
   
### 점진적 개발 전략
- 복잡한 기능을 작은 단위로 분해해 순차적으로 구현하는 접근 방식
- 점진적 개발의 구성  
  - 작업 범위의 명확한 정의  
      전체 작업을 독립적으로 테스트 가능한 단위로 분해
  - 피드백 루프 활용
      - 테스트 결과, 린터 출력, 빌드 로그처럼 구조화된 피드백을 받을 때 오류를 정확히 교정
      - 피드백 루프가 없으면 사용자가 유일한 검증 수단
  - 컨텍스트 관리
    1. `/compact`, `/clear`
    2. `/mcp` 명령을 사용하지 않는 서버 비활성화
    3. 자동 압축 설정 점검
   
#### TDD 기본 패턴
- TDD의 **레드-그린-리팩터 사이클**은 '실패하는 테스트 작성 → 테스트를 통과하는 최소 코드 구현 → 코드 구조 개선'의 세 단계로 구성
- 레드 단계
  - 구현할 기능의 예상 동작을 정의하는 테스트를 먼저 작성
  - 아직 구현 코드가 없으므로 테스트는 실패
- 그린 단계
  - 테스트를 통과하기 위한 최소한의 코드 작성
  - 클로드 코드가 Bash 도구로 테스트를 실행하고 결과를 확인하면서 통과할 때까지 코드를 수정
- 리팩터 단계
  - 통과된 코드의 기능을 유지하며서 구조를 개선하거나 중복을 제거
- TDD에서는 테스트 범위의 누락 부분을 파악하는 것도 중요

### 깃 통합 워크플로
- 여러 기능을 동시에 개발할 때는 git worktree가 유용
- git worktree는 브랜치 전환 없이 여러 브랜치를 동시에 자겅ㅂ 가능

### 비대화형 모드에서의 세션 재개
`claude --continue -p "내 작업을 계속 진행해 줘"`와 같이 `-p` 플래그와 함게 비대화형 모드로 사용하면 스크립트나 자동화 환경에서도 이전 컨텍스트를 유지하면서 작업을 진행할 수 있다.    

### 파이프와 클로드
```
{
  "script": {
    "lint": "eslint src/",
    "lint:claude": "claude -p '너는 코드 리뷰어야. main 브런치 대비 변경된 코드를 검토하고 오타, 네이밍 오루, 잠재적 버그를 찾아줘.'"
  }
}
```
이 방식은 ESLint가 잡지 못하는 맥락적 오류를 AI가 검토할 수 있다는 장점.

# 2-2 설정 관리
settings.json 파일은 개인 설정부터 팀 전체 표준까지 다양한 수준의 설정을 관리할 수 있는 계층적 구조를 제공

### sessions.json 설정과 우선순위
- 권한 규칙, 환경 변수, 도구 설정 등을 JSON 형식으로 정의
- 배치된 위치에 따라 적용 범위가 달라짐
- `/config` 으로 현재 적용 중인 설정을 확인하고 수정
- 설정 우선 순위
    1. 관리자 설정
       - 사용자가 재정의할 수 없으며, 조직 전체의 보안 정책을 강제하는데 사용
    2. 엔터프라이즈 정책
       - 인프라 팀이 시스템 디렉터리에 배포. 사용자 재정의 안됨
    3. 프로젝트 설정
       - `.claude/settinsg.json`에 정의
       - 깃 저장소에 공유해 팀 공유
       - 코드 스타일 가이드라인, 테스트 절차, 프로젝트별 도구 권한
    4. 프로젝트 로컬 설정
       - `.claude/settings.local.json`에 정의
       - 프로젝트 설정보다 우선순위가 높음
       - 개인별 조정
       - 깃 무시
    5. 사용자 설정(우선순위 최하)
       - `~/.claude/settings.json`에 정의
       - 사용자의 모든 프로젝트에 적용
      
### 기본 설정 옵션
- `apiKeyHelper`
  - 개인 사용자는 사용할 일이 없음. 일반 사용자는 `claude login`으로 인증하면 끝.
  - 주로 기업/팀 환경의 보안 정책 때문에 사용
  - API 키가 1시간 마다 만료되는 환경
- `cleanupPeriodDays`
  - 로컬 채팅 기록의 유지 기간 설정 값
- `env`
  - 모든 세션에 적용될 환경 변수를 정의
  - `settings.json`의 `env`는 Claude Code 도구 자체의 환경변수이고, 프로젝트 `.env`는 앱의 환경변수. 서로에게 영향을 주지않음
- `includeCoAuthoredBy`
  - 깃 커밋에 `co-authored-by Claude` 표시를 포함할지 결정
  - 기본값은 `true`
  - 커밋 히스토리에 AI 사용 흔적을 남기고 싶지 않을 때 끄기
- `claude --model sonnet`
  - 프로젝트 틀성에 따라 적합한 모델 선택
- `outputStyle`
  - 출력 형식과 스타일 변경
  - 미리 정의된 프리셋으로 충분하면 `settings.json`만 쓰면 되고, 더 세밀한 제어가 필요하면 `.claude/outputSytle/` 폴더를 만들어서 사용
- `hooks`
  - 도구 실행 전후에 실행되는 커스텀 명령을 정의
  - `{"PreToolUse": {"Bash": "echo 'Running...'"}}` 처럼 사용
 
설정옵션
| 키 | 설명 |
| :---: | :---: |
| `permissions` | 권한 구조 |
| `disableAllHook` | 모든 훅 비활성화 |
| `statusLine` | 컨텍스트를 표시할 커스텀 상태 표시줄 |
| `outputStyle` | 출력 스타일 구성 |
| `forceLoginMethod` | 로그인 방식 제한 |

| `disableMcpjsonServers` | 거부할 특정 MCP 서버 목록 |

권한 설정 옵션
| 키 | 설명 |
| :---: | :---: |
| `allow` | 도구 사용을 허용할 권한 규칙 배열. Bash 규칙은 접두사 일치 사용 |
| `ask` | 도구 사용 시 요청할 권한 규칙 배열 |
| `deny` | 도구 사용을 거부할 권한 규칙 배열. 민감한 파일 제외에 활용 |
| `additionalDirectories` | 클로드 코드가 접근할 수 있는 추가 작업 디렉토리 |
| `defaultMode` | 클로드 코드 시작 시 기본 권한 모드 |
| `disableBypassPermissionsMode` | "diable"로 설정하여 `bypassPermissions` 모드 비활성화 |

### 프로젝트별 설정 전략
- 권한 규칙은 `permissions` 객체 아래 `allow`, `deny`, `ask` 세 배열로 구셩
- 평가 순서는 `deny` -> `ask` -> `allow`
- `deny`에 해당하면 다른 규칙에 관계없이 차단

### 환경 변수 관리

2. settings.json의 env 필드

# 2-3 IAM 권한 관리
### IAM 개념
- IAM: '누가 무엇을 할 수 있는가'를 정의하는 보안 프레임워크
- 개인 사용자
  - Claude 실행 -> 브라우저 로그인 -> 끝
- 팀/회사 사용자
  - Teams 플랜: 관리자가 초대 -> 초대받은 이메일로 로그인
  - Enterprise 플랜: SSO(회사 계정으로 통합 로그인), 역할별 권한 설정 가능
- API로 쓰는 경우
  - 우리 서비스에 AI 기능을 넣고싶다.
  - `fetch`로 Antropic 서버에 요청을 보내 Claude 응답을 받아오는 것
  - 클로드 API, 아마존 베드락, 구글 버텍스 AI 중 어디를 통해서 Claude를 쓰느냐
  - 결국, Claude 모델은 똑같은데, 결제/관리를 어디서 하느냐 차이

### 파일 및 명령 권한 설정
- 도구 유형(읽기, Bash 명령, 쓰기)에 따라 권한 요구 사항을 달라짐
- 한번 승인된 명령은 프로젝트 및 명령 당 영구적으로 저장
- 승인 기록은 세션 종료 시 초기화

### 도구별 권한 체계
- Bash 구칙의 두가지 패턴
  - 정확한 명령일치: `Bash(npm run build)`
  - 접두사 패턴: `Bash(npm run test *)`
- `WebFetch` 규칙은 도메인 단위로 네트워크 접근 제어
- MCP 도구 권한
  - 서버 수준: `mcp__puppeteer`
  - 도구 수준: `mcp__puppeteer_pupputeer_navigate`
  - 세분화된 권한 체계를 통해 MCP 서버의 기능을 필요한 만큼만 활성화

### 안전한 실행 환경 구축
| 모드 | 설명 |
| -- | -- |
| default | - 표준 동작 - 각 도구의 첫 사용시 권한 요청 </br> - 새로운 프로젝트나 익숙하지 않은 코드베이스에서 작업할 때 적합 |
| acceptEdits | - 세션에 대해 파일 편집 권한을 자동으로 수락<br> - 빈번한 코드 수정이 필요한 리펙터링 작업에 적합 |
| plan | - 파일 분석만 가능하고 수정이나 명령 실행은 불가능<br> - 코드베이스를 탐색하고 구현 계획을 수립하는 단계에서 유용 |
| bypassPermissions | - 모든 권한 프롬프트를 건너뜀<br> - 격리된 환경에서 사용 |
- 샌드박스 설정
  - 샌드박스 = 격리된 안전한 실험 공간
  - settings.json의 `sandbox` 필드에서 샌드박스 활성화
- 작업 디렉토리 확장
  - 시작 시 `claude --add-dir` CLI 인수 사용
  - 세션 중 `/add-dir` 슬래시 명령 실행
  - 설정 파일의 `additionalDirectories`에 영구 추가
 
### 권한 문제 해결
- 일반적인 권한 문제와 그 해결 방법을 이해하면 개발 흐름의 중단을 최소화
- 권한 거부 오류
  - settings.json 파일을 확인하여 어떤 파일에 해당 규칙을 정의했는지 파알
- 설정 파일 간 충돌
  - 예상과 다른 동작이 발생하면 상위 우선순위 설정에서 해당 규칙이 정의되었는지 확인
 
### 보안 유의 사항
- 권한 설정
  - 필요한 최소한의 권한 부여
  - 새로운 요구 사항 발생시 점진적 확장
- 자격 증명 관리
- 민감한 데이터 보호
  - permissions.deny 배열에 Read(./.env), Read(./.env.*), Read(./secret/**)와 같은 규칙을 추가하여 민감한 파일에 대한 접근 원천 차단
 
### 권한 설정 실무 예시
- 권한 설정은 ./claude/settings.json에 정의하여 팀과 공유하고, 개인적인 조정이 필요한 경우에만 ./claude/settings.local.json에 재정의

### 데이터 사용 정책
- 일반 사용자(Free, Pro, Max 플랜)
  - 데이터를 사용하여 향후 클로드 모델을 훈련할 수 있도록 옵트인 가능
  - 개인정보 설정에서 데이터 사용 정책을 변경할 수 있다.

  = 생성 모델 훈련에 상요하지 않는다
  - 기업 고객의 지식 재산권과 기밀 정보를 보호
 
### 데이터 보존 기간

- 비즈니스 사용자(Team, Enterprise, API): 표준 보존기간은 30일. API 키 설정을 통해 보존 없음을 구성 가능

### 원격 분석과 비활성화 옵션
클로드 코드는 두가지를 몰래 기록
- Statsing -> "몇 명이 얼마나 썼는지" 같은 사용 통계
- Sentry -> "오류가 났을 때 어디서 났는지" 기록
- 원격 분석과 오류 보고는 환경 변수를 통해 비활성화 가능
- 베드락과 버텍스는 기업 환경이라 기본적으로 통계/오류 기록 비활성화

# 2-4 모델 구성과 API 최적화
- default
  - Free, Pro, Max 등 사용하는 계정 유형에 따라 권장 모델 자동 선택
  - Max 사용자는 Opus 사용량 임계값 도달시 자동으로 Sonnet으로 풀백
- sonnet
  - 속도와 품질의 균형이 잘 압혀 있어 대부분의 개발 작업에 적합
- opus
  - 가장 강력한 추론 능력을 갖추고 있어 아키텍처 설계, 복잡한 알고리즘 구현, 까다로운 버그 부넉 등 깊은 사고가 필요한 작업에 적합
- haiku
  - 가장 빠른 응답 시간과 가장 낮은 비용을 제공하지만, 복잡한 추론 능력은 제한적
- openplan
  - 하이브리드 접근 방식
  - 계획단계에서는 Opus 모델을 사용, 실제 구현 단계에서는 Sonnet 모델로 자동 전환
 
### 모델 설정 방법의 우선 순위
- 1순위: 세션중: /model <별칭|이름>
- 2순위: 시작 시: claude --model <별칭|이름>
- 3순위: 환경 변수: ANTRHROPIC_MODEL=<별칭|이름>
- 4순위: 설정 파일: settings.json의 model 필드

### 환경변수를 통한 모델 별칭 커스터마이징
- 별칭이 가리키는 실제 모델을 환경 변수로 재정의
- 특정 모델 버전을 고정하거나 조직 내부의 모델 정책을 구현하는데 유용
- 환경 변수를 셸 설정 파일에 추가하면 영구 적용

### 프롬프트 캐싱 이해와 최적화
- 이전 요청에서 사용된 프롬프트 일부를 캐시, 동일한 프롬프트가 다시 전송될 때 캐시된 결과를 재사용하는 매커니즘
- 디버깅 과정에서 캐시가 예상치 못한 동작을 일으키거나 클라우드 제공자별(Anthropic, 베드락, 버텍스)로 캐싱 구현이 달라 호환성 문제가 발생할 때 비활성화

### 모델 선택 전략
- Haiku: 단순한 코드 수정이나 형식 변환
- Sonnet: 일상적인 코딩 작업, 유지보수
- Opus: 복잡한 아키텍처 설계나 알고리즘 구현, 신규 프로젝트 개발 초기
- Opusplan: 설계와 구현이 모두 필요한 종합적인 작업, 신규 프로젝트 개발 초기
