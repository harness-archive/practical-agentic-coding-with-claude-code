## 1. 탐색-계획-구현-커밋 워크플로
클로드 코드는 큰 목표가 주어지면 코드베이스 분석부터 변경 사항 커밋까지 자율적으로 수행합니다 [1].

- **탐색 (Explore)**: 프로젝트 루트에서 아키텍처를 파악합니다 [2]. 파일 전체를 한 번에 읽으면 컨텍스트 토큰이 낭비될 수 있으므로, Glob과 Grep 도구로 관련 파일을 먼저 식별하고 필요한 파일만 제한적으로 읽는(Read) 패턴이 효율적입니다 [3, 4]. 필요한 경우 UI 분석을 위해 이미지 파일을 첨부할 수도 있습니다 [5].
- **계획 (Plan)**: 다단계 기능 구현 시 `Shift + Tab`을 눌러 '플랜 모드'로 진입하는 것이 안전합니다 [2, 5, 6]. 읽기 전용 상태로 코드를 건드리지 않고 파일 간의 관계나 수정 범위를 파악할 수 있으며, 이 상태에서 계획을 확정하면 자동으로 실행 모드로 전환됩니다 [2].
- **구현 (Implement)**: 한 번에 크게 수정하기보다는 오류 원인을 파악하기 쉽게 작은 단위로 검증하며 코드를 작성합니다 [6, 7].
- **커밋 (Commit)**: Bash 환경을 직접 제어하므로 작업 단위별로 깃(Git) 커밋을 자동 수행하거나 변경 이력을 추적할 수 있습니다 [7, 8].

## 2. 점진적 개발과 CLI 파이프라인 연동
- **TDD (테스트 주도 개발) 패턴**: 에이전틱 코딩의 피드백 루프를 극대화하는 방식입니다 [9]. 클로드 코드에게 '먼저 실패하는 테스트를 작성하고, 이를 통과하는 코드를 구현해달라'고 지시하면 Bash로 테스트 명령을 반복 실행하며 코드를 완성합니다 [9].
- **자동화 스크립트 결합**: 비대화형 플래그(`-p`)와 파이프(`|`)를 사용해 기존 터미널 작업과 유연하게 연동합니다 [10]. 예를 들어 `cat error.txt | claude -p "오류 원인 설명해 줘"`와 같이 빌드 에러를 직접 넘길 수 있으며, `package.json`에 클로드 기반의 코드 리뷰 스크립트(`npm run lint:claude`)를 등록하여 컨텍스트 기반 오류를 자동으로 찾아낼 수도 있습니다 [10-12].

## 3. 설정 관리 및 5대 확장 도구
프로젝트별 규칙은 `.claude/settings.json`을 통해 관리하며, 팀 공통 규칙을 Git에 커밋하여 개발 환경을 통일할 수 있습니다 [13, 14]. 

- **설정 우선순위**: `관리자 설정` ➔ `엔터프라이즈 정책` ➔ `프로젝트 설정` ➔ `로컬 설정` ➔ `사용자 전역 설정` 순으로 적용됩니다 [13, 14].
- **팀 표준화 (CLAUDE.md)**: 빌드/테스트 명령어, 코드 컨벤션, 아키텍처 결정 사항 등을 200줄 이내로 기록해두면, 클로드 코드가 작업 전 항상 이 맥락을 우선 참조합니다 [15, 16].
- **5가지 확장 시스템**:
  1. **스킬(Skills)**: 팀 공통의 코딩 가이드나 워크플로를 특정 키워드에 반응해 자동 로드하는 확장 파일(`SKILL.md`)입니다 [17, 18].
  2. **서브에이전트**: 별도의 컨텍스트 윈도우를 가진 독립된 AI 어시스턴트로, 복잡한 디버깅 등을 위임하여 메인 대화의 토큰을 아낄 수 있습니다 [18, 19].
  3. **MCP 서버**: GitHub, Sentry, DB 등 외부 시스템과 연결하는 통로입니다 [18, 20].
  4. **훅(Hooks)**: 도구 실행 전후에 자동 실행되는 셸 명령(예: 저장 직후 Prettier 린팅 적용)입니다 [18, 21].
  5. **플러그인**: 위 4가지 요소를 하나로 패키징해 배포하는 통합 형태입니다 [18, 21].

## 4. IAM 권한 관리 및 보안 제어
파일 및 시스템에 직접 접근하는 만큼 최소 권한 원칙 기반의 권한 제어가 필수적입니다 [22].

- **권한 계층 (Allow/Ask/Deny)**:
  - 읽기/검색(LS, Grep, Read) 등은 승인 없이 작동하지만, Bash 실행과 파일 수정(Write/Edit)은 사용자 승인이 필요합니다 [23, 24].
  - 권한 충돌 시 **Deny 규칙이 항상 최우선**으로 적용됩니다 [25].
- **보안 설정 실무**: `settings.json`에서 npm 테스트처럼 안전한 작업은 `allow`에 등록하고, DB 마이그레이션처럼 민감한 명령은 `ask`로 매번 승인받게 구성합니다 [26, 27]. 특히 `.env`나 `secrets/**` 등의 시크릿 파일은 반드시 `deny` 규칙에 등록해 유출을 원천 차단해야 합니다 [22, 26, 28].
- **데이터 프라이버시**: Free/Pro 사용자는 데이터 학습 옵트아웃이 필요하지만, 팀 및 엔터프라이즈 플랜은 기본적으로 데이터를 훈련에 사용하지 않습니다 [29, 30]. 원격 측정 및 에러 보고를 막고 싶다면 `DISABLE_TELEMETRY=1` 환경 변수를 사용합니다 [30, 31].

## 5. 모델 구성 및 API 최적화
작업의 복잡도와 비용을 고려하여 적절한 모델 별칭을 선택합니다 [32, 33].

- **모델 별칭**:
  - `sonnet`: 일반적인 코드 작성과 디버깅 (비용/속도 최적) [32].
  - `opus`: 아키텍처 설계 등 깊은 논리와 추론이 필요한 경우 [32].
  - `haiku`: 단순 형식 변환이나 간단한 수정 [34].
  - `opusplan` (하이브리드): **계획 단계에서는 Opus를, 구현 단계에서는 Sonnet으로 모델을 자동 전환**하여 비용과 품질의 균형을 맞춥니다 [34].
- **프롬프트 캐싱**: 중복된 컨텍스트 요청 시 캐시된 결과를 재사용하여 시간과 토큰 비용을 아껴줍니다. 오류가 발생할 경우 `DISABLE_PROMPT_CACHING=1`로 전역 비활성화할 수 있습니다 [35].
