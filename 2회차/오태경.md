## 1~2 탐색과 계획
탐색시 실전 패턴은 3단계로 접근하면 효과적이다.

1. 관련된 파일을 찾고
2. 파일간 상호작용을 파악한 다음
3. 실행 흐름을 추적한다.

ex) "사용자 인증을 처리하는 파일을 찾아" -> "인증 파일이 어떻게 동작해?" -> "로그인 프로세스를 확인해"

why? 클로드 코드 사용방식과 관련이 있는데,
1. Glob, Grep 으로 파일을 식별하도록 하면 파일 경로가 컨텍스트에 적재된다.
2. 두번째 패턴에서는 식별된 파일만 읽으므로 불필요한 토큰을 줄일 수 있다. <br> 만약 프로젝트 구조를 파악하고 있다면 @ 참조를 사용하여 디렉토리 단위로 범위를 좁혀 탐색할 수 있다.


### 파일 및 디렉토리 탐색 기법
- 클로드 코드는 분석 워크플로에서 Glob, Grep, Ls, Read 등의 도구를 사용해서 파일 및 디렉터리를 탐색하는데
- 탐색 범위가 넓을 때는 단계적으로 좁혀가는것이 컨텍스트를 절약한다.
- 텍스트만으로 파악하기 어렵다면 이미지를 대화에 추가하는것도 도움이 된다.

### 플랜 모드를 사용하여 안전한 코드 분석
- 탐색 과정에서 의도지 않은 변경을 차단하고 계획을 세우게 하려면 읽기 전용 모드인 `플랜 모드`가 필수이다.
  - 세션 중 플랜 모드로 : shift + tab
  - 새 세션을 플랜 모드로 시작 : $ claude --permission-mode plan

## 설정 관리
- `settings.json` 파일은 클로드 코드의 동작을 구성하는 설정 파일이다.
- `settings.json` 파일은 개인 설정부터 팀 표준까지 다양한 수준의 설정을 관리할 수 있는 계층적 구조를 제공
  - 권한 규칙, 환경 변수, 도구 설정 등을 JSON 형식으로 정의함
  - 배치된 위치에 따라 적용 범위가 다름
  - /config 명령으로 현재 적용 중인 설정을 확인하고 수정할 수 있다.
- 클로드 코드의 설정은 다섯 단계 범위로 적용된다.
  - 우선 순위가 높은 설정이 낮은 설정을 덮어쓴다.
  - 권한 규칙 (allow, deny, ask) 배열은 값이 합산되지만 같은 키가 존재하면 우선순위가 높은 쪽이 적용된다.

[설정 우선 순위](https://code.claude.com/docs/ko/settings#settings-precedence) 
1. 관리자 설정
   - 조직 전체의 보안 정책을 강제하는데 사용
   - managed-settings.json 파일을 시스템 디렉토리나 서버 관리 설정에 배포
   - 사용자가 재정의 불가능
2. 엔터프라이즈 정책
   - 인프라팀이나 시스템 디렉토리에 배포하는 정책 파일
   - 사용자가 재정의 불가능
3. 프로젝트 로컬 설정
   - `.claude/settings.local.json`에 정의
   - 팀 설정을 따르면서 개인화된 설정을 적용할 때 유용
   - 개인 개발 환경에서만 적용되고 깃 저장소에 커밋하지 않음
4. 프로젝트 설정
    - `.claude/settings.json`에 정의
    - 깃 저장소에 공유하여 팀 전체에 적용
    - 코드 스타일, 가이드라인, 테스트 절차, 프로젝트별 도구 권한 등을 정의
5. User 설정 (최하)
   - ~/.claude/settings.json에 정의
   - 개인 전역 설정

<br>

### 기본 설정 옵션
모델 선택, 권한 관리, 환경 변수등을 설정할 수 있다.

- **apiKeyHelper** 
  - 사용자가 API 키를 안전하게 관리하도록 돕는 도구
  - 인증값을 생성하는 커스텀 스크립트 경로를 지정 (ex. /bin/generate_api_key.sh)
  - 해당 스크립트의 반환 값은 자동으로 Authorization 헤더로 전송된다.
- **env**
  - 모든 세션에 적용할 환경 변수를 설정하여 프로젝트별로 필요한 환경 변수를 미리 정의
- **hooks**
  - 도구 실행 전후에 실행되는 커스텀 명령을 정의
  - 스킬·에이전트·도구 라이프사이클 중 특정 이벤트 발생 시 자동 트리거
- 이 외에도 다양한 설정 옵션이 존재하며, 자세한 내용은 [공식 문서](https://code.claude.com/docs/ko/settings)를 참고하면 된다.

### 프로젝트별 설정 전략
- .claude/settings.json의 permissions 객체 아래 allow, deny, ask 배열로 구성된다.
- 평가 순서는 deny -> ask -> allow
  - deny에 해당하면 뒤 규칙은 차단된다.
```json
{
  "permissions": {
    "deny": [
      "Read(./.env)"
    ],
    "ask": [
      "Bash(npm run migrate:*)",
      "Bash(npm run deploy:*)"
    ]
  }
}
```

### 환경 변수 관리
클로드 코드에서 환경 변수 관리방법은 2가지이다.
1. 셀에서 직접 export
2. settings.json 의 env 필드에 정의

우선 순위는 셀 > settings.json

env 필드는 세 새션 시작할때마다 자동 적용되어 매번 export할 필요 없다.

### 팀 표준 설정 구축
팀 협업시 CLAUDE.md 에는 어떤 맥락과 규칙을 따를지를 명시하고 스킬은 어떤 워크플로를 반복 실행할지 정의한다.

### CLAUDE.md
- 세션 시작시 컨텍스트에 로드되는 메모리 파일이다.
- settings.json 처럼 배치 위치에 따라 적용 범위가 달라진다.
- CLAUDE.md
  - 프로젝트 루트
  - 팀 전체가 공유
- CLAUDE.local.md
  - 프로젝트 루트
  - 개인화된 설정, 팀 표준을 따르되 개인별 조정이 필요시 사용
- ~/.claude/CLAUDE.md
  - 사용자 개인 설정, 모든 프로젝트에 적용
- 아무래도 메모리에 로드되다보니 200줄 이내로 유지하는 것을 권장한다.

### 스킬 (/명령)
- 반복적인 워크플로를 SKILL.md 파일에 정의한다.
- 스킬은 디렉토리 단위로 관리도 할 수 있고 상황에 따라 클로드가 자동 로드하게 할 수 있다.
- 스킬 파일 위치에 따라 공유 범위가 달라진다.

### 도구별 설정
- 승인 없이 사용 가능한 도구: Read, Glob, Grep, Agent (읽기 도구들)
- 승인이 필요한 도구 : Bash, Edit, Write, WebFetch, WebSearch (시스템 상태를 변경하거나 외부에 접근할 수 있는 도구들)
  - 한번 승인된 도구는 세션 종료시 초기화 된다.
- 각 도구에 대한 권한도 settings.json에 permssions 에 명시하면 된다.

## 클로드 코드의 다섯가지 확장 도구 비교
클로드 코드는 플러그인 개념을 채택하여 도구들은 독립적인 확장이 가능하다.

### 스킬 (무엇을 알고있는가)
- 가볍고 선언적인 확장 방식
- SKILL.md에 기술된 name과 description 필드 기반으로 요청과 의미적 유사성을 평가해 관련 스킬을 컨텍스트에 로드
- 실행 주체는 클로드(AI 추론)이기에 메인 추론 과정에 지식을 주입하는 방식이기에 메인 대화의 토큰을 소비한다.
- 지식

### 서브 에이전트 (무엇을 할 수 있는가)
- 스킬보다 한단계 위 높은 수준의 확장 방식
- 서브 에이전트는 독립적인 컨텍스트, 명시적으로 할당된 도구, 커스텀 프롬프트 등 특화된 AI 어시스턴트로써 동작한다.
- 서브 에이전트는 별도의 컨텍스트에서 작업을 진행하므로 메인 컨텍스트를 오염시키지 않는다. (컨텍스트 격리)
  - 실행 결과만 요약되어 반환한다.
- 독립 컨텍스트로 실행되므로 메인 대화의 토큰을 소비하지 않는다.
- 작업 격리

### MCP 서버
- MCP 서버는 클로드 코드가 외부 도구나 데이터에 접근할 수 있는 통로를 제공
- claude mcp add 명령으로 추가
- 외부 연결

### 훅
- 스킬·에이전트·도구 라이프사이클 중 특정 이벤트 발생 시 자동 트리거
- 정책 적용

### 플러그인
- 위 네가지를 하나의 배포 단위로 묶는 컨테이너
- 하나의 플러그인 안에 스킬, 서브 에이전트, MCP 서버, 훅을 포함할 수 있다.
- 배포

## IAM 권한 관리
- IAM은 '누가 무엇을 할 수 있는가'를 정의하는 보안 프레임워크
- AI 에이전트가 파일 시스템, 셸 명령, 외부 서비스에 접근하는 방식을 제어
- 인증과 권한을 분리하여 관리

### Bash 패턴
Bash 규칙은 아래 두가지 패턴을 지원
1. 정확한 명령 일치
   - ex. Bash(npm run build)
2. 패턴 일치
   - ex. Bash(npm run build:*)

Bash 패턴의 한계도 존재한다. Bash(curl http://example.com/ *)로 example URL만 허용하고싶지만 막지 못한다.

WebFetch 규칙은 도메인 단위로 네트워크 접근을 제어

### 안전한 실행 환경 구축
클로드 코드는 allow/deny/ask 규칙 외에 defaultMode 옵션으로 추가적인 보안 레이어를 제공
1. default
   - 표준 동작
   - 각 도구 첫 사용시 권한 요청
2. acceptEdits
   - 세션에 대해 파일 편집 권한을 자동 승인
   - 빈번한 코드 수정하는 경우 적합
3. plan
   - 읽기 전용 모드   
   - 탐색 분석 단계에서 안전하게 코드 분석 가능
4. bypassPermissions
   - 모든 권한 프롬프트를 건너뛴다.
   
```json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

### 샌드박스
https://code.claude.com/docs/ko/sandboxing
- Claude가 대부분의 셸 명령을 권한을 요청하지 않고 실행할 수 있는 옵션
- 각 명령을 승인하는 대신 명령이 접근할 수 있는 파일과 네트워크 도메인을 정의하면 `운영 체제`가 모든 Bash 명령과 그 자식 프로세스에 대해 해당 경계를 적용
- 샌드박스 실행은 Claude Code 세션을 시작하고 /sandbox 명령을 실행하거나 ~/.claude/settings.json에서 sandbox.enabled를 true로 설정

## 모델 구성과 API 최적화
- default 모델 : 계정 유형에 따라 권장 모델이 자동으로 선택된다. MAX 사용자는 Opus 사용량이 임계값에 도달하면 자동으로 Sonnet으로 폴백된다.
- sonnet : 속도와 품질의 균형이 잡혀있고, 개발 작업에 적합.
- opus : 복잡한 추론 작업용, 아키텍처 설계, 복잡한 알고리즘 구현, 버그 분석 등 깊은 사고에 필요한 작업
  - 다만 응답속도가 sonnet 보다 느리고 토큰비용도 높다.
- haiku : 가장 빠른 응답시간을 제공, 낮은 토큰 비용을 제공하지만 복잡한 추론 능력엔 제한
- opusplan : 하이브리드 접근 방식, 계획 단계에서는 Opus, 구현 단계는 sonnet으로 전환된다.

### 모델 설정 방법의 우선순위
1. 세션 중 : /model <별칭|이름>
    ```shell
    > /model opusplan
    ```
2. 시작 시 : claude --model <별칭|이름>
    ```shell
    >  claude --model opusplan
    ```
3. 환경 변수 : ANTHROPIC_MODEL=<별칭|이름>
    ```shell
    >  export ANTHROPIC_MODEL=opusplan
    ```
4. 설정 파일 : settings.json 의 model 필드
    ```json
   {
      "model": "opusplan"
   }
    ```

## 프롬프트 캐싱 이해와 최적화
- 프롬프트 캐싱은 이전 요청에서 사용한 프롬프트 일부를 캐시에 저장한다.
- 동일한 프롬프트가 다시 전송되면 캐시된 결과를 재사용한다.
- 프롬프트 캐싱은 전역 또는 모델별로 비활성화가 가능하다.

```shell
# 모든 캐싱 비활성화
$ export DISABLE_PROMPT_CACHING=1

# 모델별로 캐싱 제어를 하고 싶다면 환경 변수 사용
## haiku만 캐싱 비활성화
$ export DISABLE_PROMPT_CACHING_HAIKU=1
## opus만 캐싱 비활성화
$ export DISABLE_PROMPT_CACHING_OPUS=1
## sonnet만 캐싱 비활성화
$ export DISABLE_PROMPT_CACHING_SONNET=1
```

# 질문과 답변

Q. bypassPermissions 모드를 사용한다면 allow/deny/ask 규칙은 무시하는 걸까?

A. bypassPermissions 모드에서도 규칙 평가는 아래 순서로 진행된다. [공식문서](https://code.claude.com/docs/en/agent-sdk/permissions#)
1. deny 체크 : 매칭되면 bypassPermissions 모드라도 즉시 차단
2. ask 체크 : 매칭되면 bypassPermissions 모드라도 프롬프트가 표시되어 사용자 승인을 기다린다.
3. allow 규칙 체크
4. 1~3번 순서를 통과한 도구만 bypassPermissions가 자동 승인
