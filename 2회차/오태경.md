## 탐색 - 계획 - 구현 - 커밋 워크플로우
- 사용자가 높은 수준을 요구하면 클로드 코드가 수행하는 과정
- 각 단계는 독립적이며, 순환 반복

1. **탐색 단계**
   - 코드 베이스 구조와 맥락 파악 단계
2. **계획 단계**
   - 플랜 모드 도구를 사용하여 구현 전략을 수립하는 단계
   - 복잡한 아키텍쳐 결정, 다단계 구현시 확장 사고를 활용
3. **구현 단계**
   - 위에서 수립된 계획에 따라 코드 작성
   - 점진적 접근을 하며 대규모 수정이 아니라 작은 단위로 나누어 진행하는 것이 좋다.
4. **커밋 단계**
   - 구현된 변경 사항을 저장

## 1~2 탐색과 계획
탐색시 실전 패턴은 3단계로 접근하면 효과적이다.

1. 관련된 파일을 찾고
2. 파일간 상호작용을 파악한 다음
3. 실행 흐름을 추적한다.

ex) "사용자 인증을 처리하는 파일을 찾아" -> "인증 파일이 어떻게 동작해?" -> "로그인 프로세스를 확인해"

why? 클로드 코드 사용방식과 관련이 있는데,
1. Glob, Grep 으로 파일을 식별하도록 하면 파일 경로가 컨텍스트에 적재된다.
2. 두번째 패턴에서는 식별된 파일만 읽으므로 불필요한 토큰을 줄일 수 있다. <br> 만약 프로젝트 구조를 파악하고 있다면 @ 참조를 사용하여 디렉토리 단위로 범위를 좁혀 탐색할 수 있다.


### 파일 및 디렉토리 탐색 기법
- 클로드 코드는 분석 워크플로에서 Glob, Grep, Ls, Read 등의 도구를 사용해서 파일 및 디렉터리를 탐색하는데
- 탐색 범위가 넓을 때는 단계적으로 좁혀가는것이 컨텍스트를 절약한다.
- 텍스트만으로 파악하기 어렵다면 이미지를 대화에 추가하는것도 도움이 된다.

### 플랜 모드를 사용하여 안전한 코드 분석
- 탐색 과정에서 의도지 않은 변경을 차단하고 계획을 세우게 하려면 읽기 전용 모드인 `플랜 모드`가 필수이다.
  - 세션 중 플랜 모드로 : shift + tab
  - 새 세션을 플랜 모드로 시작 : $ claude --permission-mode plan

## 3.점진적 개발 전략
작업 범위의 명확한 정의, 피드백 루프 활용, 컨텍스트 관리가 세축으로 구성된다.
- 작업 범의의 명확한 정의   
  - 작업 범위를 명확히 정의하여 클로드 코드가 집중할 수 있도록 한다.
  - 예시) "사용자 인증 로직을 개선하는데, 로그인 프로세스에서 비밀번호 검증 부분에 집중해"
- 피드백 루프 활용
  - 구조화된 피드백을 활용하여 오류를 정확히 교정한다.
  - **"검증을 견고하게 만들어라" 공식 문서의 권장 사항**
- 컨텍스트 관리
  - `/compact`, `/clear` 명령으로 컨텍스트를 관리한다.
  - 자동 압축 설정이 활성화되어 있는지 점검한다.

### TDD 기본 패턴
- 에이전틱 코딩 환경에서 가장 효과적인 방법론이 TDD이다.
- 레드 - 그린 - 리팩토링 사이클
- 실패 테스트 작성 → 테스트를 통과하는 최소 코드 구현 → 코드 구조 개선
- 레드 단계에서는 기능의 예상 동작을 정의하면서 테스트가 실패하는지 확인한다.
- 그린 단계에서는 최소한의 코드를 작성하면서 테스트 통과하도록 구현한다.
- 리팩토링 단계에서는 통과된 코드의 기능을 유지하면서 구조 개선 및 중복 제거

### 비대화형 모드에서 세션 재개
- 세선 재개 기능인 --continue, --resume 옵션은 비대화형 모드에서 활용 가능
  - `claude --continue -p "내 작업을 계속 진행해 줘"`와 같이 -p 플래그와 함께 사용하면 이전 컨텍스트를 유지하면서 작업을 진행할 수 있다.

<br>

### 파이프와 클로드
- 클로드 코드는 파이프(|)와 표준 입출력(ex. cat)을 활용하여 다른 도구와 연동이 유연
- ex) cat file.txt | claude -p "이 텍스트에서 중요한 내용을 요약해줘" > output.txt
  - file.txt의 내용을 클로드 코드로 전달하여 요약한 뒤 output.txt에 저장하는 예시
  - --output-format 플래그로 출력 형식 제어 가능. txt 뿐만 아니라 json도 가능

## 설정 관리
- `settings.json` 파일은 클로드 코드의 동작을 구성하는 설정 파일이다.
- `settings.json` 파일은 개인 설정부터 팀 표준까지 다양한 수준의 설정을 관리할 수 있는 계층적 구조를 제공
  - 권한 규칙, 환경 변수, 도구 설정 등을 JSON 형식으로 정의함
  - 배치된 위치에 따라 적용 범위가 다름
  - /config 명령으로 현재 적용 중인 설정을 확인하고 수정할 수 있다.
- 클로드 코드의 설정은 다섯 단계 범위로 적용된다.
  - 우선 순위가 높은 설정이 낮은 설정을 덮어쓴다.
  - 권한 규칙 (allow, deny, ask) 배열은 값이 합산되지만 같은 키가 존재하면 우선순위가 높은 쪽이 적용된다.

[설정 우선 순위](https://code.claude.com/docs/ko/settings#settings-precedence) 
1. 관리자 설정
   - 조직 전체의 보안 정책을 강제하는데 사용
   - managed-settings.json 파일을 시스템 디렉토리나 서버 관리 설정에 배포
   - 사용자가 재정의 불가능
2. 엔터프라이즈 정책
   - 인프라팀이나 시스템 디렉토리에 배포하는 정책 파일
   - 사용자가 재정의 불가능
3. 프로젝트 로컬 설정
   - `.claude/settings.local.json`에 정의
   - 팀 설정을 따르면서 개인화된 설정을 적용할 때 유용
   - 개인 개발 환경에서만 적용되고 깃 저장소에 커밋하지 않음
4. 프로젝트 설정
    - `.claude/settings.json`에 정의
    - 깃 저장소에 공유하여 팀 전체에 적용
    - 코드 스타일, 가이드라인, 테스트 절차, 프로젝트별 도구 권한 등을 정의
5. User 설정 (최하)
   - ~/.claude/settings.json에 정의
   - 개인 전역 설정

### 기본 설정 옵션
모델 선택, 권한 관리, 환경 변수등을 설정할 수 있다.

- **apiKeyHelper** 
  - 사용자가 API 키를 안전하게 관리하도록 돕는 도구
  - 인증값을 생성하는 커스텀 스크립트 경로를 지정 (ex. /bin/generate_api_key.sh)
  - 해당 스크립트의 반환 값은 자동으로 Authorization 헤더로 전송된다.
- **env**
  - 모든 세션에 적용할 환경 변수를 설정하여 프로젝트별로 필요한 환경 변수를 미리 정의
- **hooks**
  - 도구 실행 전후에 실행되는 커스텀 명령을 정의
  - 스킬·에이전트·도구 라이프사이클 중 특정 이벤트 발생 시 자동 트리거
- 이 외에도 다양한 설정 옵션이 존재하며, 자세한 내용은 [공식 문서](https://code.claude.com/docs/ko/settings)를 참고하면 된다.
