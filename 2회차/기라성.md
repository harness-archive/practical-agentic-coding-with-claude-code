# 워크플로와 설정

## 2-1 기본 작업방식
### 1) 탐색-계획-구현-커밋 워크플로
- 사용자가 높은 수준의 목표를 제시하면 클로드코드가 코드베이스 탐색, 계획 수립, 코드 구현, 깃 커밋까지 수행하는 과정을 체계화 한것

**탐색 단계**
- 코드 베이스의 구조와 맥락을 파악하는 단계
- 프로젝트 루트 디렉토리에서 시작하여 전체 아키텍처 분석, 구성요소간의 상호작용 파악

**계획 단계**
- 구현 전략을 수립하는 단계 
- Shift + Tab으로 전환하며 클로드 코드가 읽기 전용 상태로 작동한다. 수정 도구가 비활성화 되므로 코드를 건드리지 않고 파일을 탐색하며 구현 계획을 수립할 수 있다.

**구현 단계**
- 수립된 계획에 따라 실제 코드를 작성한다. 
- 이 단계에서는 점진적 접근이 중요. 
- 대규모 수정을 시도하지 않고 작은 단위로 나누어 검증하면서 진행하면 오류를 최소화 할 수 있다.

**커밋 단계**
- 구현된 변경 사항을 깃 저장소에 저장

### 2) 탐색과 분석
- 특정 기능과 관련되 코드를 찾을때는 세단계로 접근
- 먼저 관련 파일을 찾고, 파일간 상호 작용을 파악한 다음, 실행 흐름을 추적
예시) 사용자 인증을 처리하는 파일을 찾아줘 -> 이 인증 파일이 어떻게 함께 동작해? -> 프런트엔드에서 DB까지 로그인 프로세스를 추적해줘 
  - 첫번째 질문에서 파일을 식별하면 해당 파일 경로가 컨텍스트에 적재된다.
  - 두번째 질문에서 이미 식별된 파일만 읽으므로 불필요한 토큰 소비를 줄인다.
  - 프로젝트 구조를 이미 파악하고 있다면 @참조로 범위를 좁혀 탐색가능
- 코드를 직접 수정하지 않고 안전하게 개발 계획을 세우려면 읽기 전용 모드인 **플랜모드**가 필수
- 플랜모드가 적합한 상황
  - 다단계 기능 구현 시 : 여러 파일에 걸쳐 기능을 구현할 때 플랜모드로 전체 구조를 설계
  - 코드 조사 단계 : 변경 전 코드베이스를 철저히 탐색할 때
  - 대화형 협의 : 반복 피드백으로 방향 조정시에 유용

### 3) 점진적 개발 전략
- 복잡한 기능을 작은 단위로 분해해 순차적으로 구현하는 접근 방식
- 클로드 코드가 한번에 넓은 범위를 수정하면 오류원인을 특저하기 어렵지만 단위를 좁히면 문제가 발생한 지점을 즉시 파악할 수 있다.

### 4) 깃 통합 워크플로
- Bash 도구로 깃 명령을 직접 실행

## 2-2 설정 관리
- settings.json 파일로 개인설정부터 팀 전체 표준까지 다양한 수준의 설정을 관리할수 있는 계층적 구조를 제공
### 1) sttings.json 설정과 우선순위
- 권한 규칙, 환경 변수, 도구 설정등을 JSON 형식으로 정의하며 배치된 위치에 따라 적용 범위가 달라진다. 
- /config 명령으로 현재 적용중인 설정으 확인하고 수정할 수 있다. 
- 우선순위(가장 위에가 높음)
  - 관리자 설정 : managed-settings.json 파일을 시스템 디렉터리에 배포하거나 서버 관리 설정을 통해 웹 인터페이스에서 원격 배포
  - 엔터프라이즈 정책 : 인프라 팀이 시스템 디렉터리에 배포하는 정책파일
  - 프로젝트 설정 : .claude/settings.json에 정의, 팀과 공유하는 설정
  - 프로젝트 로컬 설정 : .claude/settings.local.json에 정의, 팀 표준을 따르되 개인별로 조정이 필요할 때 사용
  - 사용자 설정 : ~/.claude/settings.json에 정의, 사용자의 모든 프로젝트에 적용

### 2) 기본 설정 옵션
- 설정 옵션으로 모델선택, 권환관리, 환경변수 등을 설정할 수 있다. 
  - apiKeyHelper : 인증값을 생성하는 커스텀 스크립트의 경로를 지정하는 설정
  - cleanupPeriodDays : 로컬 채팅 기록의 유지기간을 설정
  - env : 모든 세션에 적용될 환경 변수를 정의하는 설정
  - includeCoAuthoredBy : 깃 커밋에 co-authored-by Claude 표시를 포함할지를 결정하는 설정
  - claude --model sonnet 처럼 model로 기본 모델을 재정의
  - outputStyle : 시스템 프롬프트를 조정하여 출력 형식과 스타일을 변경하는 설정
  - hook : 도구 실행 전후에 실행되는 커스텀 명령을 정의하는 설정
  - 추가 옵션은 p.42, 43 참조

### 3) 팀 표준 설정 구축
- 팀 협업 환경에서 일관된 개발 경험을 제공하려면 settings.json의 권환 규칙과 함께 CLAUDE.md, 스킬을 조합해야한다. 
- CLAUDE.md
  - 세션 시작시 클로드 코드의 컨텍스트에 로드되는 메모리 파일이다. 
  - settings.json 처럼 배치 위치에 따라 적용 범위가 달라진다. 
  - CLAUDE.md (프로젝트 루트) : 팀 전체가 공유하는 지침, 깃 저장소에 컴밋한다.
  - CLAUDE.local.md (프로젝트 루트) : 개인용 지침, 깃 무시 대상으로 자동 설정한다. 팀표준을 따르되 개인별 조정이 필요할때 사용
  - ~/.claude/CLAUDE.md : 사용자 전역 지침. 모든 프로젝트에 적용
  - CLAUDE.md를 약 200줄 이내로 유지하고 참조용 자료는 스킬로 분리할것을 권장
- 스킬(슬래시 명령)
  - 반복적인 워크플로를 표준화 하려면 스킬을 정의
  - SKILL.md 파일에 YAML와 MD 지침을 작성하여 만든다.
  - 스킬 파일은 위치에 따라 공유 범위가 결정된다.
    - .claude/skills/ : 프로젝트 스킬. 깃 저장소에 커밋해 팀과 공유
    - ~/.claude/skills/ : 사용자 스킬. 개인용이며 모든 프로젝트에서 사용

### 4) 클로드 코드의 다섯 가지 확장 도구 비교
- 스킬
  - 마크다운 형식의 SKILL.md 파일과 스크립트, 템플릿, 참조 문서등의 지원파일로 구성
- 서브에이전트
  - 스킬보다 한단계 높은 수준의 확장방식으로 각 서브에이전트는 독립적인 컨텍스트 윈도우, 커스텀 시스템 프롬프트 그리고 명시적으로 할당된 도구 세트를 갖춘 특화된 AI 어시스턴트로 동작
- MCP 서버
  - 외부 세계와 연결하는 통로
  - 외부 도구, DB, API에 대한 접근을 제공
- 훅 
  - LLM의 판단을 우회하여 결정론적으로 실행되는 셸 명령
- 플러그인 
  - 앞의 확장 방식 네가지를 하나의 배포 단위로 묶는 컨테이너
  - .claude-plugin/plugin.json 매니페스트를 통해 메타데이터를 정의

## 2-4 모델 구성과 API의 최적화
- sonnet : 속도와 품질의 균형이 잘 잡혀 있어 대부분의 개발 작업에 적합
- opus : 가장 강력한 추론능력을 갖추고 있어 아키텍처 설계, 복잡한 알고리즘 구현, 까다로운 버그 분석 등 깊은 사고가 필요한 작업에 적합
- haiku : 가장 빠른 응답 시관과 가장 낮은 비용을 제공하지만 복잡한 추론능력은 제한적
- opusplan : 계획단계에서는 Opus 모델을 사용하고 실제 구현단계에서는 Sonnet 모델로 자동전환
